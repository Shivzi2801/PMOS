# PMOS — Slice S2.6: Observability & Monitoring — Architecture

## 1. Why observability exists

PMOS is an enterprise AI Knowledge Operating System assembled from many
independent slices: connectors, processing, extraction, resolution, indexing,
retrieval, context assembly, generation and grounding (Wave 1), plus the API
layer, workflow orchestration, identity/multi-tenancy, feedback and
administration/governance (Wave 2). Each slice makes decisions that affect
latency, cost, answer quality and reliability, but none of them can see the
whole picture on its own.

The Observability & Monitoring subsystem is the cross-cutting layer that gives
operators, tenants and the platform itself a single, coherent view of *what is
happening and how well it is happening*. It answers questions such as: How many
requests are in flight? What is p95 retrieval latency for tenant `t1`? How many
tokens and dollars did generation consume this month? Is the grounding score
trending down? Is any dependency unhealthy? Are we violating an SLA right now?

Three design commitments shape everything in this slice:

1. **Observability must never harm the host.** Recording a metric, emitting an
   event, evaluating a health check or firing an alert can never raise into the
   calling slice. Every public boundary isolates exceptions. A broken exporter,
   an unknown model price or a misconfigured alert rule degrades observability,
   not the product.
2. **No external services.** Everything runs in-process: an in-memory metric
   collector, in-memory telemetry sinks, a contextvar-based tracer, in-memory
   token/cost ledgers. Exporters to external systems are an *extension seam*
   (`TelemetrySink`), not a dependency.
3. **Framework-agnostic and dependency-injection friendly.** Clocks, sinks,
   registries, pricing tables and processors are all injectable. The W3C
   trace-context wire format is reproduced without importing OpenTelemetry, so
   PMOS interoperates without taking on a heavy runtime dependency.

## 2. Every file and its responsibility

### Metrics core

- **`metrics_registry.py`** — The vocabulary. `MetricDefinition` (name, kind,
  unit, label keys, histogram buckets) is the schema for a single metric;
  `MetricsRegistry` is the thread-safe catalog. `build_default_registry()`
  registers the canonical PMOS metric set (`pmos.api.*`, `pmos.workflow.*`,
  `pmos.retrieval.*`, `pmos.generation.*`, `pmos.grounding.*`,
  `pmos.ingestion.*`, `pmos.tokens_total`, `pmos.cost_usd_total`). Defining
  metrics centrally means every collector validates kind and labels against one
  source of truth.
- **`metrics_collector.py`** — `MetricsCollector` is the write path. It offers
  `increment`, `set_gauge`, `adjust_gauge` and `observe`, each validated
  against the registry. Internally it uses lock-striped accumulators
  (`_CounterAcc`, `_GaugeAcc`, `_HistogramAcc`) keyed by normalized label sets,
  so concurrent writers do not contend on a single lock. `snapshot()` produces
  an immutable read-model.
- **`metrics_snapshot.py`** — The read path. `MetricsSnapshot` is a frozen
  point-in-time view containing `MetricSample`s; `HistogramState` carries bucket
  counts plus `mean` and `quantile(q)` (linear interpolation within the
  breaching bucket). Snapshots are what alerting, SLA evaluation and dashboards
  consume, decoupling readers from the live collector.

### Telemetry

- **`telemetry_event.py`** — `TelemetryEvent` is the canonical structured
  event: name, `EventCategory`, `EventSeverity`, timestamp, UUID, an
  `EventContext` (tenant/user/trace/span/workflow/request correlation ids) and
  free-form attributes. Events are immutable; `with_context` / `with_attributes`
  return enriched copies (event enrichment & correlation).
- **`telemetry_sink.py`** — The publish path. `TelemetrySink` is the ABC;
  `InMemoryTelemetrySink` (ring buffer + query), `LoggingTelemetrySink` (stdlib
  bridge), `FilteringTelemetrySink` (severity floor / category allow-list /
  predicate) and `CompositeTelemetrySink` (fan-out with per-child exception
  isolation) are the built-ins. New exporters subclass the ABC.

### Tracing

- **`trace_context.py`** — `TraceContext` is the immutable identity of a
  position in a trace (trace id, span id, sampled flag, baggage). It serializes
  to/from W3C `traceparent`/`tracestate` headers, enabling propagation across
  service and process boundaries without an external library.
- **`trace_span.py`** — `TraceSpan` is a single timed operation (mutable while
  active, sealed into an immutable `SpanData` on `end()`). It records
  attributes, events, status and exceptions; as a context manager it captures
  exceptions and never suppresses them. Double-ending raises `SpanError`.
- **`distributed_tracer.py`** — `DistributedTracer` ties spans to ambient
  context via a `ContextVar` (correct under threads and asyncio). It starts
  spans, activates context for the `with` block, auto-ends, and emits a
  `TRACE` telemetry event per span (ERROR severity if the span errored).
  `inject`/`extract`/`continue_from_headers` move context across boundaries.
- **`workflow_tracer.py`** — `WorkflowTracer` layers PMOS workflow semantics on
  the tracer: `workflow_span(...)` and `step_span(...)` create correctly-named
  spans and feed duration/run metrics back through injected recorder callbacks,
  flipping span status to error on exception.

### Metric facades (domain → metric-name mapping)

Each facade turns a typed domain outcome into the right metric writes, so call
sites never hard-code metric-name strings:

- **`retrieval_metrics.py`** — latency, result count, top score; increments
  `pmos.retrieval.empty_total` on zero results.
- **`generation_metrics.py`** — latency, prompt/completion tokens, cost, run
  count, plus platform rollups `pmos.tokens_total` and `pmos.cost_usd_total`.
- **`grounding_metrics.py`** — latency, grounding score, unsupported-claim
  count.
- **`ingestion_metrics.py`** — per-`IngestionStage` latency, document and chunk
  counts.
- **`api_metrics.py`** — request count, latency, errors and an in-flight gauge.
  `track_request()` is a context manager whose `RequestTracker` defaults to 200,
  flips to 500 on exception, and always balances the in-flight gauge.

### LLM cost / tokens / usage

- **`token_tracker.py`** — `TokenTracker` keeps an exact, thread-safe ledger
  keyed by `(tenant, model)`. Metrics are sampled/aggregated; the ledger is the
  exact source of truth for billing-grade token accounting.
- **`cost_tracker.py`** — `ModelPricing` + `PricingTable` (with sensible
  defaults for Claude/GPT/embedding models) convert tokens to USD;
  `CostTracker` records a per-`(tenant, model)` cost ledger. Unknown models with
  no default raise `CostTrackingError` (handled, not fatal).
- **`usage_report.py`** — `UsageReportBuilder` merges the token and cost ledgers
  into a `UsageReport` with per-tenant, per-model summaries (including cost-only
  entries like embeddings), powering usage and cost reports.

### Alerts

- **`alert_rule.py`** — `AlertCondition` is the evaluation ABC. Conditions:
  `CounterRateCondition` (ratio of two counters), `LatencyQuantileCondition`
  (worst per-label quantile), `GaugeThresholdCondition` (max gauge),
  `HealthCondition` (overall or named component at/over a severity). `AlertRule`
  binds a condition to a severity, a debounce (`for_consecutive`) and labels.
- **`alert_engine.py`** — `AlertEngine` runs the rule set against a snapshot and
  health report, driving each rule through the `OK → PENDING → FIRING →
  RESOLVED → OK` state machine, emitting `alert.fired` / `alert.resolved`
  events. A misbehaving rule is isolated and cannot break the cycle.

### Health

- **`health_status.py`** — `HealthState` (HEALTHY/DEGRADED/UNHEALTHY/UNKNOWN
  with a severity ordering and a `worst` combinator), `HealthCheckResult` and
  the aggregate `HealthReport` (with `is_ready` / `is_live`).
- **`service_health.py`** — `HealthCheck` ABC plus `CallableHealthCheck`,
  `DependencyHealthCheck` (ping; slow ⇒ DEGRADED via `degraded_after_ms`) and
  `MetricThresholdHealthCheck` (derives health from a snapshot). Each check
  times itself and falls back to UNKNOWN if its probe raises.
- **`health_aggregator.py`** — `HealthAggregator` runs all checks and combines
  them: any non-healthy *critical* check drives the overall state; non-critical
  problems cap the overall at DEGRADED. Provides readiness and liveness.

### SLA, dashboard, composition

- **`sla_monitor.py`** — `AvailabilitySlo` and `LatencySlo` express targets and
  compile to alert rules; `SlaMonitor.evaluate()` computes attainment and
  remaining error budget as `SloStatus`.
- **`dashboard_projection.py`** — `DashboardProjectionBuilder` folds a snapshot,
  health report, active alerts and SLO statuses into a single immutable
  `DashboardProjection` read-model (latency panels p50/p95/p99/max, throughput
  totals, component states, alert/SLO summaries) ready for any UI.
- **`observability_service.py`** — `ObservabilityService` is the composition
  root and the single public dependency for every other slice. It wires the
  registry, collector, sinks, tracer, ledgers, health aggregator, alert engine,
  SLA monitor and dashboard builder together and exposes high-level operations
  (`record_generation`, `record_retrieval`, `evaluate_health`,
  `run_alert_cycle`, `usage_report`, `dashboard`, `span`, `event`). Every public
  method isolates exceptions. `build_observability_service(...)` is the factory.
- **`errors.py`** — The exception hierarchy rooted at `ObservabilityError`.
- **`__init__.py`** — Curated public surface and `__version__`.
- **`test_observability.py`** — 46 deterministic unit/integration tests using a
  `FakeClock` and in-memory sinks.

## 3. Metrics lifecycle

1. **Define** — At startup `build_default_registry()` registers every canonical
   metric with its kind, unit, label keys and (for histograms) buckets.
2. **Collect** — A facade (or any caller) calls `increment` / `set_gauge` /
   `adjust_gauge` / `observe` on the `MetricsCollector`. The call is validated
   against the registry (unknown metric ⇒ `MetricNotRegisteredError`, wrong kind
   ⇒ `MetricTypeError`, bad labels ⇒ `LabelError`) and written into a
   lock-striped accumulator keyed by the normalized label set.
3. **Snapshot** — On demand, `snapshot()` produces an immutable
   `MetricsSnapshot` of `MetricSample`s (histograms carry full `HistogramState`).
4. **Read** — Alert engine, SLA monitor and dashboard builder consume the
   snapshot. Because the snapshot is immutable and decoupled from the live
   collector, readers never block writers.

## 4. Tracing lifecycle

1. **Start** — `DistributedTracer.span("op")` reads ambient context from the
   `ContextVar`; if absent it mints a new root `TraceContext`, otherwise it
   derives a child. A `TraceSpan` is created and the context is activated for
   the `with` block.
2. **Annotate** — Inside the block, code sets attributes/events and (via the
   `WorkflowTracer`) records workflow/step metrics.
3. **Propagate** — `inject()` serializes the active context to `traceparent`/
   `tracestate` headers for an outbound call; the remote side calls `extract()`
   / `continue_from_headers()` to resume the same trace.
4. **End** — On block exit the span ends (recording any exception and flipping
   status to ERROR), the previous context is restored, and the tracer emits a
   `TRACE` telemetry event carrying the sealed `SpanData`.

## 5. Health-monitoring lifecycle

1. **Register** — Components register `HealthCheck`s with the
   `HealthAggregator`, marking each critical or non-critical.
2. **Probe** — `evaluate()` runs each check; every check times itself and, if
   its probe raises, returns UNKNOWN rather than propagating.
3. **Aggregate** — Any non-healthy critical check drives the overall state;
   non-critical problems cap the overall at DEGRADED. The result is a
   `HealthReport` with readiness (`is_ready`) and liveness (`is_live`).
4. **Publish** — `ObservabilityService.evaluate_health()` emits a `HEALTH`
   telemetry event and caches the last report for the alert cycle.

## 6. Alert lifecycle

1. **Configure** — Rules are registered directly or generated from SLOs
   (`register_sla_alerts()`).
2. **Evaluate** — `run_alert_cycle()` snapshots metrics and the last health
   report, then `AlertEngine.evaluate(...)` runs every rule's condition.
3. **Debounce & transition** — Each rule advances through `OK → PENDING →
   FIRING → RESOLVED → OK`; `for_consecutive` suppresses flapping by requiring N
   consecutive breaches before FIRING.
4. **Notify** — Transitions emit `alert.fired` / `alert.resolved` telemetry
   events and invoke registered listeners. A faulty rule is isolated.

## 7. Reporting lifecycle

1. **Accumulate** — Every `record_generation` updates the exact token and cost
   ledgers in addition to metrics.
2. **Build** — `usage_report()` merges both ledgers per tenant and model
   (including cost-only embedding usage) into a `UsageReport`.
3. **Project** — `dashboard()` folds the latest snapshot, health report, active
   alerts and SLO statuses into a single `DashboardProjection` for operators.

## 8. Integration with previous PMOS slices

The integration contract is intentionally narrow: slices depend only on
`ObservabilityService`, never on internals.

- **S1.1 Connectors / S1.2 Processing / S1.3 Extraction / S1.4 Resolution /
  S1.5 Indexing** — Call `record_ingestion(...)` with the relevant
  `IngestionStage`, producing per-stage latency, document and chunk metrics.
- **S1.6 Retrieval** — Calls `record_retrieval(...)`; empty-result and
  top-score signals feed retrieval-quality dashboards and alerts.
- **S1.7 Context Assembly / S1.8 Generation** — Generation calls
  `record_generation(...)`, which updates latency/token/cost metrics *and* the
  exact ledgers behind usage and cost reports.
- **S1.9 Grounding** — Calls `record_grounding(...)`; declining grounding scores
  or rising unsupported-claim counts surface as alerts.
- **S2.1 API Layer** — Wraps each request in `ApiMetrics.track_request(...)` and
  in a tracer `span(...)`, propagating `traceparent` inbound/outbound so a
  request trace spans the whole platform.
- **S2.2 Workflow Orchestration** — Uses `WorkflowTracer` so each workflow and
  step is both traced and timed, with failures reflected in workflow health.
- **S2.3 Identity & Multi-Tenancy** — Supplies the `tenant_id` carried as a
  label and in `EventContext`, making every metric, ledger entry and report
  tenant-scoped.
- **S2.4 Feedback** — Reads grounding/quality metrics and emits feedback-derived
  events through the telemetry sinks.
- **S2.5 Administration & Governance** — Consumes `usage_report()`,
  `dashboard()` and health/alert state for operator consoles, quota enforcement
  and governance reporting.
