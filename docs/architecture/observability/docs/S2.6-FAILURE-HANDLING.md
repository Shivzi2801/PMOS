# PMOS — Slice S2.6: Observability & Monitoring — Failure Handling

## 1. The non-fatal guarantee

The central invariant of this slice is simple: **observability must never crash
its host.** Telemetry is a supporting concern. If recording a metric, emitting
an event, evaluating a health check, or firing an alert could raise into a
request handler or workflow step, then the act of *observing* the platform would
become a source of outages. That is unacceptable, so every public method of
`ObservabilityService` — and every fan-out boundary inside the subsystem — is
exception-isolated.

The guarantee is scoped deliberately:

- **Public service methods never raise** for operational reasons (a bad model
  price, a flaky exporter, a malformed alert rule). They degrade observability
  and continue.
- **Low-level primitives still validate strictly.** `MetricsCollector` raises
  `MetricNotRegisteredError`, `MetricTypeError` and `LabelError` for programmer
  mistakes, because those indicate a coding bug that should be caught in tests,
  not silently swallowed in production. The facades and the service wrap these
  primitives and absorb such errors at the boundary.

This split — strict core, forgiving edge — keeps bugs visible during
development while keeping production resilient.

## 2. Where isolation is applied

### Service boundary (`observability_service.py`)

Every `record_*` method wraps its work in `contextlib.suppress(Exception)` (or
an explicit try/except that falls back to a safe default). For example,
`record_generation` returns an estimated cost; if the cost lookup raises for an
unknown model, it catches the error, falls back to `0.0`, still records the
token ledger and metrics it can, and returns. The caller in the generation slice
receives a number and never sees an exception.

### Telemetry fan-out (`telemetry_sink.py`)

`CompositeTelemetrySink` iterates its children and isolates each one: if a child
`emit` raises, the error is wrapped as a `SinkError`, forwarded to the optional
`on_error` callback, and the **next child is still attempted**. One broken
exporter cannot starve the others or propagate upward. `FilteringTelemetrySink`
similarly never lets a predicate error escape the emit path.

### Alert evaluation (`alert_engine.py`)

`AlertEngine.evaluate(...)` runs each rule independently. If a single rule's
condition raises (a bad threshold, a missing metric, a buggy custom condition),
that rule is skipped for the cycle and the remaining rules still evaluate. A
misconfigured alert can never blind the operator to every *other* alert.

### Health probing (`service_health.py`)

Every `HealthCheck.check()` wraps its `_probe()` in a try/except. If the probe
raises — a dependency ping throws, a metric is missing — the check returns a
`HealthCheckResult` with state `UNKNOWN` (or `UNHEALTHY` for a hard dependency
failure) and the captured message, rather than propagating. The aggregator then
folds UNKNOWN into the overall state conservatively.

### Tracing (`distributed_tracer.py`, `trace_span.py`)

Span context managers capture exceptions, set span status to ERROR, emit the
ERROR-severity trace event, and then **re-raise** — tracing observes failures
without hiding them. The one place tracing deliberately raises on its own is
`TraceSpan.end()` when a span is ended twice (`SpanError`), because that is a
programming error in instrumentation code.

## 3. Specific failure modes and their handling

### Unknown model price

`CostTracker`/`PricingTable` raise `CostTrackingError` when asked to price a
model that is neither known nor covered by a default. At the service boundary
this is caught: cost falls back to `0.0`, the exact token ledger is still
updated, and a warning-level signal can be emitted. Billing accuracy degrades
gracefully (a price can be backfilled) instead of breaking generation.

### Failing telemetry exporter

A custom sink that throws on `emit` is contained by `CompositeTelemetrySink`.
The healthy in-memory sink still records the event; the failure is surfaced via
`on_error` for diagnosis. Telemetry delivery is best-effort by design.

### Malformed trace headers

`TraceContext.from_headers` raises `TraceContextError` only for *present but
malformed* headers (bad shape, all-zero ids). Absent headers return `None`,
which the tracer treats as "start a new root trace." Inbound requests with junk
trace headers therefore start a clean trace rather than failing the request; the
API layer can log the malformed header and proceed.

### Broken alert rule

A rule whose condition raises is isolated per-cycle (see §2). It contributes no
transitions and leaves prior state untouched, so it can neither fire spuriously
nor mask other alerts.

### Health-check probe failure

A throwing probe yields UNKNOWN. Critical UNKNOWN/UNHEALTHY checks drive overall
health down (failing readiness, protecting traffic routing); non-critical ones
cap the overall at DEGRADED. The platform fails safe: an unobservable critical
dependency is treated as not-ready rather than assumed healthy.

### Double-ended span

`TraceSpan.end()` raises `SpanError` on the second call. This is intentionally
*not* swallowed, because it signals incorrect instrumentation (e.g. ending a
span both manually and via the context manager) that should be fixed in code.

### Concurrent metric writes

The collector uses lock-striped accumulators keyed by label set, so concurrent
writers to different label combinations do not contend, and writers to the same
combination are serialized correctly. Snapshots take a consistent read without
blocking writers, so a slow reader can never stall the request path.

## 4. Error taxonomy

All exceptions derive from `ObservabilityError`, so a caller that wants to catch
"anything observability" can do so with one except clause:

- `MetricError` → `MetricNotRegisteredError`, `DuplicateMetricError`,
  `MetricTypeError`, `LabelError`
- `TelemetryError` → `SinkError`
- `TracingError` → `SpanError`, `TraceContextError`
- `AlertError` → `InvalidAlertRuleError`
- `HealthError` → `UnknownComponentError`
- `ReportingError`
- `CostTrackingError`

Programmer-error types (metric/label/kind, double-end, invalid rule) are meant
to fail loudly in development. Operational types (sink, cost, malformed context)
are meant to be absorbed at the service edge in production.

## 5. Testing the failure paths

`test_observability.py` exercises isolation explicitly:

- `test_composite_sink_isolates_failures` — a throwing child does not stop the
  healthy sink and surfaces via `on_error`.
- `test_alert_engine_isolates_bad_rule` — a condition that raises does not break
  the evaluation cycle.
- `test_health_check_exception_is_unknown` — a throwing probe yields a non-fatal
  UNKNOWN/UNHEALTHY result.
- `test_cost_tracker_known_and_unknown_model` — unknown models raise
  `CostTrackingError` at the core...
- `test_service_never_crashes_on_bad_input` — ...but the service boundary
  absorbs it and returns a safe `0.0` cost.
- `test_span_lifecycle_and_double_end` — double-ending raises `SpanError`.
