# PMOS — Slice S2.6: Observability & Monitoring — Sequence Diagrams

These diagrams show how the observability subsystem participates in the main
runtime flows. All interactions go through `ObservabilityService`; no other
slice touches internals directly.

## 1. Request trace: API → workflow → retrieval → generation → grounding

A single inbound API request is traced end-to-end. The API layer opens a root
span (or continues one from inbound `traceparent` headers), the workflow
orchestrator opens child spans per step, and each domain slice records its own
metrics through the service. Trace context propagates implicitly via the
tracer's `ContextVar`.

```mermaid
sequenceDiagram
    autonumber
    participant Client
    participant API as S2.1 API Layer
    participant OBS as ObservabilityService
    participant WF as S2.2 Workflow
    participant RET as S1.6 Retrieval
    participant GEN as S1.8 Generation
    participant GND as S1.9 Grounding
    participant SINK as TelemetrySink

    Client->>API: HTTP request (maybe traceparent)
    API->>OBS: span("api:request") + track_request()
    OBS-->>API: active span + RequestTracker
    API->>WF: run workflow (ambient trace context)

    WF->>OBS: workflow_span("answer", tenant)
    OBS-->>WF: workflow span (child of api span)

    WF->>OBS: step_span("retrieve")
    WF->>RET: retrieve(query, tenant)
    RET-->>WF: results
    RET->>OBS: record_retrieval(outcome)
    Note over OBS: latency, results, top_score;<br/>empty_total++ if zero hits

    WF->>OBS: step_span("generate")
    WF->>GEN: generate(context, tenant)
    GEN-->>WF: answer + token counts
    GEN->>OBS: record_generation(model, tokens)
    Note over OBS: metrics + exact token ledger<br/>+ cost ledger; returns cost

    WF->>OBS: step_span("ground")
    WF->>GND: ground(answer, sources)
    GND-->>WF: grounding score
    GND->>OBS: record_grounding(outcome)

    WF-->>API: final answer
    API->>OBS: tracker.set_status(200); span ends
    OBS->>SINK: emit TRACE events (per span)
    API-->>Client: HTTP 200 + answer

    Note over OBS,SINK: Any failure in a span flips status to ERROR<br/>and emits an ERROR-severity trace event,<br/>but never blocks the response.
```

## 2. Metric record → snapshot → alert cycle

The write path (facades record into the collector) is decoupled from the read
path (snapshot consumed by health, alerts and SLA). `run_alert_cycle()` ties
them together and drives the alert state machine.

```mermaid
sequenceDiagram
    autonumber
    participant FAC as Metric Facade
    participant COL as MetricsCollector
    participant SVC as ObservabilityService
    participant HA as HealthAggregator
    participant AE as AlertEngine
    participant SINK as TelemetrySink
    participant LIS as AlertListener

    Note over FAC,COL: continuous write path
    FAC->>COL: increment / observe / set_gauge
    COL->>COL: validate vs registry; lock-striped write

    Note over SVC,LIS: periodic read path (run_alert_cycle)
    SVC->>COL: snapshot()
    COL-->>SVC: immutable MetricsSnapshot
    SVC->>HA: last_report() or evaluate()
    HA-->>SVC: HealthReport
    SVC->>AE: evaluate(snapshot, health)

    loop each registered rule
        AE->>AE: condition.evaluate(snapshot, health)
        alt breach and consecutive >= for_consecutive
            AE->>AE: OK/PENDING -> FIRING
            AE->>SINK: emit "alert.fired"
            AE->>LIS: notify(alert)
        else no breach and was firing
            AE->>AE: FIRING -> RESOLVED -> OK
            AE->>SINK: emit "alert.resolved"
            AE->>LIS: notify(alert)
        else rule raises
            AE->>AE: isolate; skip this rule only
        end
    end
    AE-->>SVC: transitions
```

## 3. Health evaluation

Each check times and isolates itself; the aggregator combines results with
critical/non-critical weighting and emits a health event.

```mermaid
sequenceDiagram
    autonumber
    participant SVC as ObservabilityService
    participant HA as HealthAggregator
    participant C1 as CallableHealthCheck (critical)
    participant C2 as DependencyHealthCheck (non-critical)
    participant SINK as TelemetrySink

    SVC->>HA: evaluate_health()
    HA->>C1: check()
    activate C1
    C1->>C1: time probe; map result -> HealthState
    alt probe raises
        C1-->>HA: HealthCheckResult(UNKNOWN/UNHEALTHY)
    else ok
        C1-->>HA: HealthCheckResult(HEALTHY)
    end
    deactivate C1

    HA->>C2: check()
    activate C2
    C2->>C2: ping(); slow -> DEGRADED, raise -> UNHEALTHY
    C2-->>HA: HealthCheckResult(state, latency_ms)
    deactivate C2

    HA->>HA: aggregate(critical drives overall;<br/>non-critical caps at DEGRADED)
    HA-->>SVC: HealthReport(state, is_ready, is_live)
    SVC->>SINK: emit "health.evaluated"<br/>(ERROR severity if not ready)
```

## 4. Usage report build

The report merges the exact token and cost ledgers — not sampled metrics — so
totals are billing-grade, including cost-only entries such as embeddings.

```mermaid
sequenceDiagram
    autonumber
    participant Caller as Admin/Governance (S2.5)
    participant SVC as ObservabilityService
    participant URB as UsageReportBuilder
    participant TT as TokenTracker
    participant CT as CostTracker

    Caller->>SVC: usage_report(period_label)
    SVC->>URB: build(period_label)
    URB->>TT: entries() (per tenant/model tokens)
    TT-->>URB: token ledger
    URB->>CT: entries() (per tenant/model cost)
    CT-->>URB: cost ledger
    URB->>URB: merge by (tenant, model);<br/>include cost-only entries (embeddings);<br/>sum totals
    URB-->>SVC: UsageReport(tenants, totals)
    SVC-->>Caller: UsageReport
```
