# S1.6 — Retrieval Layer · Architecture

**Wave 1 / Slice 1.6**
**Package:** `backend/retrieval/`
**Depends on:** S1.5 Indexing Layer (chunking, embeddings, vector storage)
**Consumed by:** downstream RAG / answer-generation systems (future slices)

---

## 1. Purpose

The Retrieval Layer is the read-path that sits directly on top of the Indexing
Layer. Given a natural-language query plus a security context (tenant +
principals), it returns an ordered, paginated, access-filtered set of chunks
ready for a RAG pipeline to consume. It owns *no* storage of its own; it is a
pure orchestration + policy layer over S1.5.

It is responsible for the ten concerns enumerated in the slice objective:

| # | Responsibility | Where it lives |
|---|----------------|----------------|
| 1 | Semantic search | `retriever.Retriever` → `VectorIndex.search` |
| 2 | Hybrid retrieval (vector + metadata) | `hybrid_retriever.HybridRetriever`, `filters` |
| 3 | Ranking / reranking | `reranker` |
| 4 | Source ACL filtering | `filters.apply_acl` |
| 5 | Tenant isolation | `filters.enforce_tenant_isolation` |
| 6 | Query expansion hooks | `query_expansion` |
| 7 | Retrieval metrics | `metrics` |
| 8 | Error handling | `errors`, `safe_*` wrappers |
| 9 | Pagination | `retrieval_query.Pagination`, retriever windowing |
| 10 | Downstream contract | `retrieval_result.RetrievalResult` |

## 2. Design principles (carried from S1.1–S1.5)

- **Dataclasses + typing everywhere.** Inputs and outputs are frozen
  value-objects; orchestrators are small mutable dataclasses.
- **Contracts over concretions.** The boundary to S1.5 is the `VectorIndex`
  Protocol, not a concrete class. Any conforming store is retrievable.
- **Security is fail-closed and layered.** Tenant isolation *raises*; ACL
  defaults to private; both run before ranking and pagination.
- **Non-essential stages degrade, they don't fail.** Query expansion and
  reranking are wrapped so a fault falls back gracefully and is recorded as
  `degraded=True` rather than failing the request.
- **Zero external dependencies.** Standard library only; tests use `unittest`.
- **Extension points are Protocols.** `Reranker`, `QueryExpander`,
  `KeywordIndex`, and `MetricsSink` are all swappable — including a future
  LLM reranker — without touching the retriever.

## 4. The S1.5 ↔ S1.6 contract

The retriever requires exactly one thing of the Indexing Layer:

```python
class VectorIndex(Protocol):
    def search(self, *, query_text: str, tenant_id: str,
               top_k: int) -> Sequence[IndexCandidate]: ...
```

`IndexCandidate` is the agreed wire shape (chunk_id, document_id, source, text,
tenant_id, score, metadata). Contractual obligations:

- The index **must** scope its search to `tenant_id`. The retriever
  re-verifies (defence in depth) and raises if a foreign record appears.
- The index returns up to `top_k` candidates ordered by descending similarity.
- Embedding stays on the indexing side (`query_text` is passed through), so the
  retrieval layer adds no embedding dependency — consistent with S1.5 owning
  the embedding model.
- On backend failure the index raises; the retriever wraps it as
  `IndexUnavailableError` and emits a failure metric.

The downstream contract is `RetrievalResult` (hits + `PageInfo` + diagnostics),
which is fully `to_dict()`-serialisable for transport across slice boundaries.

## 5. Retrieval pipeline (single call)

Stage order is **security-first, recall-then-precision**:

```
1. Query expansion      (optional, safe — never fatal)
2. Vector search        (fatal on backend failure)
3. min_score floor      (optional)
4. Tenant isolation     (RAISES on cross-tenant record)
5. ACL filtering        (silently drops inaccessible)
6. Metadata filtering   (silently drops non-matching)
7. Reranking            (optional, safe — never fatal)
8. Pagination           (offset/limit window over survivors)
9. Assemble result + emit metrics
```

Filtering precedes ranking so rerankers (especially a future LLM reranker that
might forward chunk text to an external model) never see records the caller may
not view. Pagination is last so `total_candidates` reflects the true filtered
population and `has_more` is accurate.

## 6. Hybrid retrieval

`HybridRetriever` composes the base `Retriever` and adds two behaviours:

1. **Filter-aware over-fetch.** When metadata filters are present, the
   candidate pool is inflated by `overfetch_factor` (clamped) so a full page
   survives filtering rather than starving.
2. **Score fusion.** An optional `KeywordIndex` contributes a lexical ranking
   that is fused with the vector ranking via Reciprocal Rank Fusion (RRF). The
   fusion is hidden behind an internal `_FusingIndex` that itself satisfies the
   `VectorIndex` contract, so the entire base pipeline (tenant/ACL/filter/
   rerank/paginate/metrics) is reused unchanged. With no keyword index, it
   degrades cleanly to pure-vector retrieval with over-fetch.

## 7. Extension points (future work)

- **LLM reranker:** implement `Reranker.rerank(query, hits)`; inject via
  `Retriever(reranker=...)`. `safe_rerank` already isolates failures.
- **LLM / semantic query expansion:** implement `QueryExpander.expand`; inject
  via `Retriever(expander=...)`. `safe_expand` already isolates failures.
- **Real keyword index (BM25, trigram):** implement `KeywordIndex.search_ranked`.
- **Production metrics backend:** implement `MetricsSink` (StatsD / OTel /
  Prometheus) — no retriever changes required.

---

## 8. Non-goals

- No caching layer (cache keys are available via `RetrievalQuery.fingerprint()`
  for a future slice).
- No cursor-based deep pagination yet; offset is guarded by `MAX_OFFSET`.
- No answer generation — that belongs to the downstream RAG slice that consumes
  `RetrievalResult`.
