# S1.9 — Grounding & Answer Verification: Architecture

> **Audience:** Product Managers, engineers, architects, investors, and enterprise customers.
> **Goal of this document:** explain *what the grounding layer is, why it exists, how every file fits together,* and *how it interacts with the rest of the product* — in plain language, with enough precision that a PM can confidently explain it to a technical or commercial audience.

---

## 1. The one-paragraph version

When the product generates an answer, that answer is only trustworthy if it is actually backed by the customer's real data. The **Grounding & Answer Verification** layer (Slice S1.9) is the component that *checks the homework* of the answer-generation step. It breaks the answer into individual factual statements, looks for evidence behind each one, measures how much of the answer is genuinely supported, flags anything that looks made-up ("hallucinated"), assigns a confidence score from 0 to 1, decides whether the answer should be shown as **Verified**, **Partially Verified**, or **Rejected**, and writes an **audit record** so the whole decision can be replayed later for compliance.

---

## 2. Why this layer exists (the business case)

Large language models are fluent. Fluency is dangerous: a model can produce a confident, well-written sentence that is simply false or unsupported by the documents it was given. For a consumer toy this is annoying. For an **enterprise** product handling refunds, contracts, medical notes, or financial data, an unsupported claim is a liability.

The grounding layer turns "trust me" into "here's the proof." It exists to deliver four things enterprises demand before they will buy:

1. **No silent hallucinations.** Every factual claim is checked against evidence.
2. **Traceability.** Every answer can be traced back to the exact retrieved passages behind it.
3. **A trust signal.** A single confidence number and a clear status the UI can act on.
4. **Auditability.** A permanent, replayable record of *why* an answer was trusted or rejected.

---

## 3. Where S1.9 sits in the system

```
backend/context/      →  retrieves evidence (passages) from the customer's data
backend/generation/   →  writes the answer + attaches citations
backend/grounding/    →  (THIS SLICE) verifies the answer against the evidence
        ↓
      API / UI layer   →  shows the answer with its trust status & confidence
```

- **Upstream — context slice:** supplies the retrieved passages. In our contracts these arrive as `Evidence` objects (`evidence_id`, `text`, `relevance_score`, `source`).
- **Upstream — generation slice:** supplies the written answer text plus `Citation` objects that *claim* which evidence supports which part of the answer.
- **Downstream — API/UI:** receives a `GroundingResult` and uses its `verification_status`, `confidence_score`, and `hallucination_risk` to decide how (or whether) to display the answer.

The grounding layer **never trusts the generation layer's citations at face value.** It re-checks them. This is the central design stance.

---

## 4. The data contracts (the shared vocabulary)

Everything in the slice speaks in terms of a few small, immutable data structures defined in `grounding_result.py`. A PM only needs to understand these five.

| Contract | Plain-English meaning |
|---|---|
| **Evidence** | A passage retrieved from the customer's data. Has an id, the text, and a relevance score. |
| **Citation** | The generator's *claim* that a particular evidence passage backs part of the answer. Treated as a hint, not proof. |
| **Claim** | One atomic, independently checkable statement pulled out of the answer (e.g. "the refund was approved on May 6"). |
| **EvidenceMatch** | A link saying "claim X is supported by evidence Y at strength Z" (FULL / PARTIAL / NO support). |
| **GroundingResult** | The final verdict: verified claims, unsupported claims, confidence, risk, coverage, status, and traceability links. |

Three enums encode the vocabulary of trust:
- **SupportLevel:** `FULL_SUPPORT`, `PARTIAL_SUPPORT`, `NO_SUPPORT`.
- **HallucinationRisk:** `LOW`, `MEDIUM`, `HIGH`.
- **VerificationStatus:** `VERIFIED`, `PARTIALLY_VERIFIED`, `REJECTED`.

## 5. File-by-file walkthrough

Below, every file in `backend/grounding/` is explained the way the slice spec requires: *why it exists, responsibilities, inputs, outputs, dependencies, interaction with other slices, example workflow, failure scenarios, and design decisions.*

### 5.1 `grounding_result.py` — the contracts

- **Why it exists.** Gives every other module one shared, authoritative definition of grounding data, so there are no circular imports and serialisation to JSON (for audit and API) is trivial.
- **Responsibilities.** Define the enums, the input contracts (`Citation`, `Evidence`), the intermediate contracts (`Claim`, `EvidenceMatch`), and the final `GroundingResult`. Provide `to_dict()` on each for clean serialisation.
- **Inputs / outputs.** None — it is pure data. It is *imported* everywhere.
- **Dependencies.** Standard library only.
- **Interaction with other slices.** `Evidence` mirrors the shape of the context slice's output; `Citation` mirrors the generation slice's output. This is the seam between slices.
- **Example workflow.** The pipeline constructs `Claim` objects from the extractor, `EvidenceMatch` objects from the matcher, and finally one `GroundingResult`.
- **Failure scenarios.** Being pure data, it cannot "fail" at runtime; bad data is caught by the modules that consume it.
- **Design decision.** Dataclasses, not raw dicts, so that the shape is enforced by the language and shows up in editors and type checkers.

### 5.2 `errors.py` — the exception family

- **Why it exists.** So failures inside the trust layer are *typed and predictable*, and the pipeline can choose to reject, retry, or degrade based on the failure type.
- **Responsibilities.** Define `GroundingError` (base) and specific subclasses: `MissingCitationsError`, `EmptyEvidenceError`, `MalformedClaimError`, `MalformedAnswerError`, `VerificationTimeoutError`, `ConfidenceComputationError`, `EvidenceMatchingError`, `ClaimExtractionError`, `AuditWriteError`.
- **Inputs / outputs.** Each error carries a machine-readable `code`, a human message, and a structured `details` dict, plus `to_dict()` for logging/audit.
- **Dependencies.** Standard library only.
- **Interaction with other slices.** Codes are stable so upstream/downstream logging and dashboards can aggregate them.
- **Example workflow.** The matcher raises `EmptyEvidenceError`; the pipeline catches it and produces a graceful, REJECTED result instead of crashing.
- **Failure scenarios.** N/A — this *is* the failure-handling vocabulary.
- **Design decision.** A single base class enables `except GroundingError` to catch the whole family, while subclasses allow fine-grained handling.

### 5.3 `claim_extractor.py` — break the answer into claims

- **Why it exists.** Verification has to work at the level of individual facts, not whole paragraphs, so we can say "claim 1 is supported, claim 2 is not."
- **Responsibilities.** Segment the answer into sentences, drop non-factual fragments (greetings, hedges, pure questions), record each claim's character span, and attach citation hints whose span overlaps the claim.
- **Inputs.** `answer_text`, `answer_id`, optional list of `Citation`.
- **Outputs.** A list of `Claim`.
- **Dependencies.** `errors.py`, `grounding_result.py`.
- **Interaction with other slices.** Consumes the generation slice's answer text and citations.
- **Example workflow.** `"The customer requested a refund on May 4. The refund was approved on May 6."` → two `Claim` objects with deterministic ids `ans-1::c0` and `ans-1::c1`.
- **Failure scenarios.** Empty/None answer → `MalformedAnswerError`. Unexpected segmentation error → `ClaimExtractionError`.
- **Design decision.** A deterministic, dependency-free segmenter (with abbreviation handling) rather than a heavyweight NLP model — extraction must be fast, reproducible, and explainable for audits. Claim ids are deterministic so audits can be replayed exactly.

### 5.4 `evidence_matcher.py` — find the support

- **Why it exists.** A claim is only trustworthy if evidence backs it. This module finds, per claim, the best-supporting evidence and how strong that support is.
- **Responsibilities.** Score each claim against each evidence passage, convert the score to a `SupportLevel` using configurable thresholds, give a small bonus when the generator's own citation is the matching evidence, and return both the best match per claim and the full match list for audit.
- **Inputs.** A list of `Claim`, a list of `Evidence`.
- **Outputs.** `(all_matches, best_by_claim)`.
- **Dependencies.** `errors.py`, `grounding_result.py`.
- **Interaction with other slices.** Consumes the context slice's evidence.
- **Example workflow.** Claim "refund approved on May 6" vs evidence "The refund was approved on May 6." → high similarity → `FULL_SUPPORT`.
- **Failure scenarios.** Empty evidence → `EmptyEvidenceError` (so the pipeline decides policy). Unexpected error → `EvidenceMatchingError`.
- **Design decision.** A transparent lexical similarity (blended containment + Jaccard, stopword-filtered) rather than an opaque embedding model. Auditors value *explainability* — "these words overlapped" — over a marginal accuracy gain. The scoring function is isolated so it can be upgraded to embeddings later without changing callers.

### 5.5 `citation_validator.py` — measure coverage

- **Why it exists.** Turns the promise "every factual claim is backed by evidence" into a number: `citation_coverage`.
- **Responsibilities.** Decide per claim whether it is supported (FULL or PARTIAL), compute `citation_coverage = supported / total`, and surface findings: dangling citations (cite an unknown evidence id), cited-but-unsupported claims, supported-but-uncited claims, and partial supports.
- **Inputs.** Claims, the best match per claim, the set of valid evidence ids.
- **Outputs.** A `CitationReport` (coverage, supported/unsupported ids, findings, warnings).
- **Dependencies.** `grounding_result.py`.
- **Interaction with other slices.** Findings feed the audit trail and can drive UI highlighting.
- **Example workflow.** Two claims, both supported → coverage `1.0`, no unsupported ids.
- **Failure scenarios.** Zero claims → coverage defined as `1.0` with a warning (nothing to ground is trivially covered, but flagged).
- **Design decision.** `PARTIAL_SUPPORT` counts toward coverage but is recorded as a finding, so partial support is credited yet still reviewable.

### 5.6 `hallucination_detector.py` — flag the risk

- **Why it exists.** Converts the support picture into a single, explainable risk band that the review gate can act on.
- **Responsibilities.** Identify `NO_SUPPORT` claims and assign `LOW` / `MEDIUM` / `HIGH` using two independent triggers: too many unsupported claims, or coverage below threshold.
- **Inputs.** Claims, best match per claim, citation coverage.
- **Outputs.** A `HallucinationReport` (risk, unsupported claim ids, human-readable reasons).
- **Dependencies.** `grounding_result.py`.
- **Interaction with other slices.** Risk drives the verifier's REJECTED decision and the confidence penalty.
- **Example workflow.** Three unsupported claims and `0.0` coverage → `HIGH`, with reasons listing both triggers.
- **Failure scenarios.** No claims → `LOW` (nothing to hallucinate).
- **Design decision.** Two independent triggers either of which escalates, because the cost of fabrication is asymmetric — we bias toward caution. Thresholds are injected so each enterprise tenant can run a stricter or looser gate.

### 5.7 `confidence_scorer.py` — one number to trust

- **Why it exists.** Stakeholders want a single "how much should I trust this?" scalar.
- **Responsibilities.** Fuse four signals — retrieval relevance, citation coverage, evidence support strength, hallucination risk — into a 0–1 score with a transparent per-signal breakdown.
- **Inputs.** `retrieval_relevance`, `citation_coverage`, `avg_support_weight`, `hallucination_risk`.
- **Outputs.** A `ConfidenceBreakdown` (final score, components, applied penalty).
- **Dependencies.** `errors.py`, `grounding_result.py`.
- **Interaction with other slices.** The score is shown in the UI and gates the VERIFIED status.
- **Example workflow.** Strong everything + LOW risk → score > 0.8. Same signals + HIGH risk → score < 0.4 (because HIGH multiplies the base by 0.3).
- **Failure scenarios.** Weights that don't sum to 1.0 → `ConfidenceComputationError`. Any other failure → same error; the pipeline forces confidence to 0.0 and treats risk as HIGH (fail safe).
- **Design decision.** Additive weighting for the base (weights sum to 1.0) but a *multiplicative* risk penalty, so a single HIGH-risk answer can never present as high-confidence — risk can veto, not merely nudge.

### 5.8 `answer_verifier.py` — make the call

- **Why it exists.** Concentrates the *policy* decision (ship / warn / block) in one small, reviewable place.
- **Responsibilities.** Apply the documented status rules: all claims supported and not HIGH risk → `VERIFIED`; some unsupported but not HIGH risk → `PARTIALLY_VERIFIED`; HIGH risk → `REJECTED`. A confidence floor can downgrade a would-be VERIFIED to PARTIALLY_VERIFIED.
- **Inputs.** `unsupported_count`, `hallucination_risk`, `confidence_score`.
- **Outputs.** A `VerificationDecision` (status + human rationale).
- **Dependencies.** `grounding_result.py`.
- **Interaction with other slices.** The status is the primary gate the API/UI obeys.
- **Example workflow.** Zero unsupported, LOW risk, confidence 0.9 → `VERIFIED`.
- **Failure scenarios.** None of its own — it is pure decision logic over already-computed signals.
- **Design decision.** `REJECTED` is driven by HIGH risk (not confidence alone) so the user-facing reason matches the true cause; confidence is a secondary gate.

### 5.9 `audit_trail.py` — remember everything

- **Why it exists.** Compliance and debugging require that any answer can be replayed and justified after the fact.
- **Responsibilities.** Build an immutable `AuditRecord` (`request_id`, `answer_id`, `retrieval_ids`, `citation_ids`, `grounding_timestamp`, `verification_summary`) and persist it through a pluggable sink.
- **Inputs.** A `GroundingResult` plus request context.
- **Outputs.** A persisted `AuditRecord`.
- **Dependencies.** `errors.py`, `grounding_result.py`.
- **Interaction with other slices.** Provides the compliance backbone enterprises audit against.
- **Example workflow.** After verification, the pipeline builds a record and appends it to the JSON-lines log (or memory in tests).
- **Failure scenarios.** Storage failure → `AuditWriteError`; the pipeline catches it, keeps the answer, and attaches a warning. We never lose the user's answer over an audit hiccup.
- **Design decision.** Append-only records (immutability is what makes it an *audit* trail), UTC ISO-8601 timestamps for cross-region clarity, and a `AuditSink` interface so production can swap in a database/object store without touching the pipeline.

### 5.10 `metrics.py` — see the system

- **Why it exists.** You cannot operate a trust layer you cannot observe.
- **Responsibilities.** Accumulate `grounding_requests`, `verified_answers`, `partially_verified_answers`, `rejected_answers`, `citation_coverage`, `hallucination_rate`, `confidence_distribution`, and `verification_latency`, and expose a JSON snapshot.
- **Inputs.** One `record(...)` call per pipeline run.
- **Outputs.** A `snapshot()` dict ready for a dashboard.
- **Dependencies.** `grounding_result.py`.
- **Interaction with other slices.** A telemetry exporter (Prometheus/OTEL/StatsD) reads the snapshot.
- **Example workflow.** After each `run`, the pipeline records the outcome; ops watches hallucination_rate trend over time.
- **Failure scenarios.** Thread-safe via a lock; metric recording never raises into the request path.
- **Design decision.** Running aggregates (sum/count) rather than storing every sample — memory-stable and dependency-free.

### 5.11 `grounding_pipeline.py` — the conductor

- **Why it exists.** Wires the single-purpose modules into the end-to-end flow and is the one public entry-point for the rest of the product.
- **Responsibilities.** Run the nine stages (extract → match → validate → detect → score → decide → assemble → audit → metrics), enforce a soft timeout, and guarantee that a `GroundingResult` is *always* returned, even on failure.
- **Inputs.** A `GroundingRequest` (`request_id`, `answer_id`, `answer_text`, `evidence`, `citations`, optional `retrieval_relevance`).
- **Outputs.** A `GroundingResult`.
- **Dependencies.** Every other module in the slice.
- **Interaction with other slices.** Called by the generation/API layer; consumes context + generation outputs.
- **Example workflow.** See the sequence-diagram document.
- **Failure scenarios.** Empty evidence, malformed answer, timeout, audit failure, confidence failure — each handled with graceful degradation (detailed in the failure-handling document).
- **Design decision.** Components are *injected* (dependency injection), so thresholds, sinks, and even the similarity strategy can be swapped per tenant or in tests without rewriting the pipeline.

### 5.12 `__init__.py` — the public surface

- **Why it exists.** Gives consumers one clean import surface (`from backend.grounding import GroundingPipeline, GroundingRequest, ...`) and documents the intended entry-point.
- **Responsibilities.** Re-export the contracts, components, orchestrator, and errors; define `__all__`.
- **Design decision.** Explicit `__all__` keeps the public API intentional and stable.

## 6. End-to-end example (the "refund" walkthrough)

**Input answer:** "The customer requested a refund on May 4. The refund was approved on May 6."
**Evidence:** two passages stating exactly those facts, each with relevance 0.95.

1. **Extract** → two claims (`c0`, `c1`).
2. **Match** → both claims hit their evidence at `FULL_SUPPORT`.
3. **Validate** → coverage `1.0`, no unsupported claims.
4. **Detect** → risk `LOW`.
5. **Score** → confidence ~0.9.
6. **Decide** → `VERIFIED`.
7. **Assemble** → `GroundingResult` with both claims verified.
8. **Audit** → one immutable record written.
9. **Metrics** → `verified_answers += 1`.

The UI shows the answer with a green "Verified" badge and a 90% confidence indicator, and a compliance officer can later pull the audit record to see exactly which two passages justified it.

## 7. How to talk about this layer to different audiences

- **To an engineer:** "It's a deterministic verification pipeline; each stage is a pure, injectable component, and the orchestrator guarantees a result and an audit record on every path."
- **To an architect:** "It's the trust boundary between generation and the API. It depends only on data contracts from context and generation, fails safe, and externalises storage and telemetry behind interfaces."
- **To an investor:** "It's our moat for enterprise: it makes every AI answer provable, scored, and auditable — turning a risky black box into a compliant system of record."
- **To an enterprise customer:** "Every answer you see is checked against your own data, scored for confidence, and logged so your auditors can replay exactly why it was trusted."
