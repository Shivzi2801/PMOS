# PMOS Slice S2.4 — Feedback, Evaluation & Continuous Learning
## Architecture Reference

**Version:** 2.4.0
**Status:** Production
**Audience:** Product managers, engineers, solution architects, enterprise customers, investors

---

## 1. Executive Summary

Slice S2.4 is the **feedback, evaluation, and continuous-improvement layer** of the
PMOS (Product Memory Operating System) platform. PMOS is a multi-tenant,
grounded-answer system: it ingests customer knowledge, retrieves relevant
evidence, generates answers, and grounds those answers in cited sources.

S2.4 is the layer that **closes the loop**. It observes how well the system
performed on each answer, captures what humans thought of it, scores quality
along multiple independent axes, routes problematic answers to human review,
and emits durable, auditable *learning signals* and *improvement
recommendations*.

A defining boundary of this slice: **it does not retrain any model**. It is a
*signal factory*, not a training pipeline. It collects and organizes evidence
that downstream teams and automated jobs can later use to improve prompts,
ranking, retrieval, grounding, and model selection. This separation keeps the
online serving path fast and safe while still building the data asset that
makes the platform improve over time.

### Why this matters commercially

- **For enterprise customers:** every answer is measurable, auditable, and
  reviewable. Hallucinations are detected and queued for human attention rather
  than shipped silently. Quality is reportable per tenant and per workspace.
- **For investors:** the slice produces the proprietary feedback data asset —
  the compounding advantage that lets the product improve with usage rather
  than degrade.
- **For product managers:** quality trends, feedback rates, and review backlogs
  are first-class, queryable metrics.
- **For engineers:** a clean, dependency-injected, fully in-memory-testable
  package with explicit failure semantics and a single composition root.

## 2. Design Principles

1. **Signals, not training.** The slice never mutates a model. It produces
   `LearningEvent`s and `ImprovementRecommendation`s consumed by other systems.
2. **Multi-tenant isolation everywhere.** Every record, event, queue read, and
   analytics rollup is scoped by `tenant_id` + `workspace_id`. Cross-tenant
   access is actively rejected, not merely unlikely.
3. **Dependency injection via Protocols.** Storage, metrics, identity
   resolution, and event publishing are all Protocol interfaces with in-memory
   default implementations. Production deployments swap in durable backends
   without touching domain logic.
4. **Value objects are frozen.** Ratings, records, scores, and events are
   immutable dataclasses. This makes them safe to pass across threads and
   trivially auditable.
5. **Graceful degradation.** A failure in one scorer degrades that dimension to
   a zero-weight contribution rather than failing the whole evaluation.
6. **One direction of dependency.** `feedback` depends on `identity`,
   `orchestration`, `grounding`, and `retrieval`. Those slices never depend on
   `feedback`. The loop is closed by *event emission*, not by reverse coupling.

## 4. Per-File Reference

For each module below: **purpose**, **responsibilities**, **inputs**,
**outputs**, and **cross-slice interactions**.

### 4.1 `errors.py`

- **Purpose:** Single exception hierarchy for the entire slice.
- **Responsibilities:** Define `FeedbackError` base (carrying `message`, `code`,
  and structured `context`) and one subclass per failure mode: missing feedback,
  invalid rating, duplicate feedback, missing answer, missing citation,
  evaluation failure, review-workflow failure (with `ReviewItemNotFoundError`
  and `InvalidReviewTransitionError` specializations), analytics aggregation
  failure, and identity-context failure.
- **Inputs:** Error message and optional context dict at construction.
- **Outputs:** Raisable exceptions; each exposes `to_dict()` for structured
  logging and API surfacing.
- **Interactions:** Used by every other module. The `code` field is stable and
  suitable for mapping to HTTP status codes at the `api` slice boundary.

### 4.2 `metrics.py`

- **Purpose:** Observability surface.
- **Responsibilities:** Define the canonical metric names
  (`feedback_received`, `feedback_processed`, `evaluation_runs`,
  `quality_score_average`, `review_queue_size`, `hallucination_reports`,
  `citation_failures`, `improvement_recommendations_generated`), a `MetricsSink`
  Protocol (`increment` / `gauge` / `observe`), an `InMemoryMetricsSink` for
  tests, a `NullMetricsSink`, and the `FeedbackMetrics` facade that maps domain
  events to metric emissions.
- **Inputs:** Domain events from services.
- **Outputs:** Counter increments, gauges, and observations to the sink.
- **Interactions:** In production the sink forwards to the platform's
  telemetry pipeline (the same one used by `orchestration`). Metric labels carry
  tenant/workspace scope from `IdentityContext.scope_labels`.

### 4.3 `identity.py`

- **Purpose:** Bind every feedback artifact to a tenant, workspace, and user.
- **Responsibilities:** `IdentityContext` frozen dataclass (requires
  `tenant_id` and `workspace_id`, optional `user_id` and `actor_role`), a
  `require_user()` guard, a `scope_labels` helper for metrics, and an
  `IdentityResolver` Protocol for turning an auth token into a context.
- **Inputs:** Tenant/workspace/user identifiers, or a token to resolve.
- **Outputs:** A validated, immutable identity context.
- **Interactions with `identity` slice:** This is the integration seam. In
  production, `IdentityResolver` is implemented by `backend/identity`, which
  validates the token, enforces tenant membership, and returns the authoritative
  scope. S2.4 never trusts caller-supplied tenant IDs without resolution.

### 4.4 `rating.py`

- **Purpose:** Typed, validated numeric feedback.
- **Responsibilities:** Frozen value objects `StarRating` (1–5, with a
  `normalized` property mapping to [0,1]), `ConfidenceFeedback` ([0,1]), and
  `UsefulnessFeedback` ([0,1]). All validate at construction and raise
  `InvalidRatingError` on out-of-range input.
- **Inputs:** Raw numbers from the API/UI.
- **Outputs:** Validated immutable rating objects.
- **Interactions:** Consumed by `feedback_record` and the quality scorers.

### 4.5 `reaction.py`

- **Purpose:** Categorical feedback (thumbs / sentiment).
- **Responsibilities:** `Reaction` enum (`THUMBS_UP`, `THUMBS_DOWN`, `POSITIVE`,
  `NEGATIVE`, `NEUTRAL`) with a `polarity` (-1/0/+1), `is_positive` /
  `is_negative` helpers, and a tolerant `from_value()` parser.
- **Inputs:** String reaction values from the UI.
- **Outputs:** Typed reaction with polarity.
- **Interactions:** Drives feedback polarity, which flows into analytics and the
  unified quality score.

### 4.6 `feedback_record.py`

- **Purpose:** The canonical, immutable unit of captured feedback.
- **Responsibilities:** `FeedbackType` enum (thumbs up/down, star rating, free
  text, citation feedback) and the `FeedbackRecord` frozen dataclass carrying
  identity scope, `query_id`, `answer_id`, type, timestamp, and the
  modality-specific payload. Computes a `dedupe_key` (SHA-256 over
  identity + target + type + citation) and a `polarity`, and serializes via
  `to_dict()`.
- **Inputs:** Validated ratings/reactions plus identity and target IDs.
- **Outputs:** An immutable, hashable, auditable record.
- **Interactions:** The `query_id` and `answer_id` tie back to artifacts
  produced by the `retrieval`, `generation`, and `grounding` slices.

### 4.7 `feedback_event.py`

- **Purpose:** Represent feedback as orchestration workflow events.
- **Responsibilities:** `FeedbackEventKind` enum
  (`feedback.submitted`, `feedback.processed`, `evaluation.requested`,
  `review.requested`, `learning.signal`), the `FeedbackEvent` envelope (with
  correlation ID and timestamps), an `EventPublisher` base class, and an
  `InMemoryEventPublisher` for tests.
- **Inputs:** Domain occurrences from the service layer.
- **Outputs:** Serializable events.
- **Interactions with `orchestration` slice:** In production, `EventPublisher`
  is backed by the `orchestration` event bus, so feedback submission and
  evaluation become workflow steps that can trigger downstream automations
  (e.g., auto-opening a review, kicking off a nightly recommendation job).

### 4.8 `quality_score.py`

- **Purpose:** Generic, composable scoring primitive.
- **Responsibilities:** `QualityScore` (a named, clamped-[0,1], weighted value
  with rationale) and `QualityScoreSet` (a labeled collection exposing
  `weighted_average` and serialization).
- **Inputs:** Individual dimension scores.
- **Outputs:** A weighted aggregate and a serializable breakdown.
- **Interactions:** The common currency of all four scorers and the evaluation
  engine.

### 4.9 `answer_quality.py`

- **Purpose:** Score the generated answer itself.
- **Responsibilities:** `AnswerQualityScorer.score()` produces five dimensions —
  relevance (w 1.2), completeness (w 1.0), clarity (w 0.8), usefulness (w 1.1),
  and confidence (w 0.7) — by blending lightweight text heuristics with any
  available user feedback.
- **Inputs:** `AnswerQualityInput` (query text, answer text, requested vs.
  covered aspects, model confidence, feedback records).
- **Outputs:** A `QualityScoreSet` labeled `answer_quality`.
- **Interactions with `generation`:** Consumes the generated answer and the
  model's self-reported confidence.

### 4.10 `citation_quality.py`

- **Purpose:** Measure how well the answer used its sources.
- **Responsibilities:** `CitationQualityScorer.score()` produces usage,
  relevance, coverage, trustworthiness, and acceptance-rate dimensions.
- **Inputs:** `CitationQualityInput` (citation descriptors with relevance/trust,
  total vs. cited claims, citation feedback records).
- **Outputs:** A `QualityScoreSet` labeled `citation_quality`.
- **Interactions with `grounding`:** Citation descriptors and the claim/citation
  counts come directly from the grounding slice's output.

### 4.11 `retrieval_quality.py`

- **Purpose:** Measure the retrieval stage that fed the answer.
- **Responsibilities:** `RetrievalQualityScorer.score()` produces
  retrieved-chunk volume, precision (used / retrieved), a recall *estimate*,
  reranker effectiveness, and source diversity (a Shannon-style entropy over
  source IDs).
- **Inputs:** `RetrievalQualityInput` (retrieved chunks with rank/used/score,
  a missing-evidence flag, expected source count).
- **Outputs:** A `QualityScoreSet` labeled `retrieval_quality`.
- **Interactions with `retrieval`:** Chunk metadata comes from the retrieval
  slice. Recall is explicitly an estimate because true recall requires labeled
  ground truth the online system does not have.

### 4.12 `quality_scorer.py`

- **Purpose:** Roll component scores into one number.
- **Responsibilities:** `UnifiedQualityScorer.score()` takes optional retrieval,
  citation, feedback, and review sub-scores and produces an
  `overall_quality_score` as a weighted average of *whichever sources are
  present* (weights: feedback 1.2, citation 1.2, review 1.0, retrieval 0.8). If
  an unresolved hallucination is flagged, the score is **capped at 0.3**
  regardless of other inputs.
- **Inputs:** `UnifiedQualityInput`.
- **Outputs:** `UnifiedQualityResult` (overall score, contributing sources,
  capped flag, breakdown).
- **Interactions:** The single number surfaced to dashboards and SLAs.

### 4.13 `evaluation_result.py`

- **Purpose:** The structured output of an evaluation run.
- **Responsibilities:** `EvaluationStage` enum (query, retrieval, context
  assembly, generation, grounding, final answer), `StageEvaluation` (a stage +
  its score set), and `EvaluationResult` (all stage evaluations, an overall
  score, and the three failure flags: hallucination suspected, citation
  failure, retrieval failure).
- **Inputs:** Built by the evaluation engine.
- **Outputs:** A serializable, per-stage evaluation record.
- **Interactions:** Consumed by the recommendation engine and the service layer.

### 4.14 `evaluation_engine.py`

- **Purpose:** Orchestrate scoring across all pipeline stages.
- **Responsibilities:** `EvaluationEngine.evaluate()` runs each available scorer,
  assembles `StageEvaluation`s, rolls up an overall score, and derives the
  failure flags using explicit thresholds (retrieval-failure 0.35,
  citation-failure 0.35, grounding-hallucination 0.40). A `_safe()` wrapper
  degrades any scorer exception to a zero-weight contribution so one bad scorer
  cannot abort the run. Raises `EvaluationError` only if *no* stage produced a
  score.
- **Inputs:** `EvaluationContext` (identity, query/answer IDs, the three scorer
  inputs, grounding support ratio, context relevance).
- **Outputs:** An `EvaluationResult`.
- **Interactions:** The hub that consumes outputs of `retrieval`, `generation`,
  and `grounding` and turns them into measurable signals.

### 4.15 `review_item.py`

- **Purpose:** Model a single human-review case.
- **Responsibilities:** `ReviewReason`, `ReviewStatus`, and `ReviewOutcome`
  enums, plus the `ReviewItem` dataclass with an explicit state machine
  (`OPEN → IN_REVIEW → RESOLVED/DISMISSED`). Illegal transitions raise
  `InvalidReviewTransitionError`.
- **Inputs:** Flag reason, identity scope, target IDs, priority.
- **Outputs:** A reviewable, transition-guarded item.
- **Interactions:** Drives the human-in-the-loop quality gate.

### 4.16 `review_queue.py`

- **Purpose:** Tenant-scoped storage and prioritization of review items.
- **Responsibilities:** In-memory queue with `enqueue`, `get` (raises
  `ReviewItemNotFoundError`), `update`, `open_items` (sorted by descending
  priority then age), `next_for_review`, and `backlog_size`. All reads are
  tenant-scoped.
- **Inputs:** Review items.
- **Outputs:** Prioritized work lists.
- **Interactions:** In production, swapped for a durable, queryable store; the
  Protocol shape stays identical.

### 4.17 `human_review.py`

- **Purpose:** The review service API.
- **Responsibilities:** `HumanReviewService` with `flag_answer`,
  `flag_citation`, `flag_hallucination` (high priority + a hallucination-report
  metric), `flag_policy_issue`, `request_manual_review`, `claim`, `resolve`
  (emits a `REVIEW_COMPLETED` learning event), and `dismiss`. Enforces
  same-tenant access and keeps the `review_queue_size` gauge current.
- **Inputs:** Identity + target IDs + outcomes.
- **Outputs:** Updated review items and emitted learning events/metrics.
- **Interactions:** Bridges automated evaluation (auto-flagging) and human
  judgment, and feeds the learning log.

### 4.18 `learning_event.py`

- **Purpose:** Durable, auditable improvement signals.
- **Responsibilities:** `LearningEventType` enum (poor answer, hallucination,
  retrieval failure, citation failure, high-quality answer, review completed),
  the frozen `LearningEvent`, and an append-only `LearningEventLog` with
  tenant/workspace/type filters and a count.
- **Inputs:** Events from the engine, service, and review flows.
- **Outputs:** An immutable audit trail.
- **Interactions:** This is the **primary hand-off to future model-improvement
  work** (see §7). The log is append-only precisely so it can serve as an audit
  record for enterprise compliance.

### 4.19 `improvement_recommendation.py`

- **Purpose:** Turn evaluation results into actionable guidance.
- **Responsibilities:** `RecommendationCategory` enum (improve retrieval,
  chunking, ranking, grounding, prompting, source coverage), `Severity`, the
  `ImprovementRecommendation` dataclass, and a rule-based `RecommendationEngine`
  that maps low stage scores and failure flags to targeted recommendations.
- **Inputs:** An `EvaluationResult`.
- **Outputs:** A list of recommendations with target subsystem, rationale,
  severity, and evidence.
- **Interactions:** Recommendations name the subsystem to change
  (`retrieval`, `generation`, `grounding`, etc.), making them routable to the
  owning team.

### 4.20 `analytics_event.py`

- **Purpose:** High-volume, lightweight counting events.
- **Responsibilities:** `AnalyticsEventType` enum (answer served, feedback
  received, evaluation completed, review opened/closed, citation decision) and
  the frozen `AnalyticsEvent`.
- **Inputs:** Service-layer occurrences.
- **Outputs:** Serializable counting events.
- **Interactions:** Distinguished from learning events: analytics events are
  cheap and aggregate-oriented; learning events are durable and auditable.

### 4.21 `analytics_service.py`

- **Purpose:** Fold events into tenant- and workspace-level rollups.
- **Responsibilities:** `ScopeAnalytics` (answer/feedback counts, positive and
  negative feedback, citation decisions/approvals, reviews opened/closed,
  quality scores) with derived rates, a `quality_trend` classifier
  (improving / declining / stable / insufficient data), and the
  `AnalyticsService` that records events and answers tenant/workspace queries.
- **Inputs:** `AnalyticsEvent`s and quality scores.
- **Outputs:** Queryable analytics objects.
- **Interactions:** Powers customer-facing and internal dashboards.

### 4.22 `feedback_aggregator.py`

- **Purpose:** Summarize raw feedback records.
- **Responsibilities:** `FeedbackAggregator` produces a `FeedbackSummary`
  (totals, polarity counts, star count/sum, citation feedback counts) and offers
  by-answer, by-tenant, and by-workspace groupings.
- **Inputs:** Iterables of `FeedbackRecord`.
- **Outputs:** `FeedbackSummary` objects.
- **Interactions:** Used by the service layer and reporting.

### 4.23 `feedback_collector.py`

- **Purpose:** Validate and persist incoming feedback.
- **Responsibilities:** `FeedbackStore` Protocol + `InMemoryFeedbackStore`, the
  `FeedbackSubmission` DTO (raw, untrusted input), and `FeedbackCollector.collect()`
  which validates the modality, builds a record, verifies that the referenced
  answer and citation exist, and rejects duplicates (unless `allow_update`).
- **Inputs:** `IdentityContext` + `FeedbackSubmission`.
- **Outputs:** A persisted `FeedbackRecord`.
- **Interactions:** The `answer_exists` and `citation_exists` callables are
  satisfied by the `generation`/`grounding` slices in production.

### 4.24 `feedback_service.py`

- **Purpose:** The composition root and public API.
- **Responsibilities:** Wire every collaborator and expose four operations:
  `submit_feedback` (collect + publish events + record analytics + metrics),
  `evaluate_answer` (run the engine, derive learning events, generate
  recommendations, record analytics, auto-flag suspected hallucinations for
  review), `compute_quality` (unified score), and `aggregate_feedback`.
  Produces an `EvaluationReport` bundling result + learning events +
  recommendations.
- **Inputs:** Identity contexts, submissions, and evaluation contexts.
- **Outputs:** Records, evaluation reports, unified quality results, summaries.
- **Interactions:** The single entry point the `api` slice calls.

### 4.25 `__init__.py`

- **Purpose:** Public API surface.
- **Responsibilities:** Re-export the full public API with `__version__` and a
  comprehensive `__all__`; document the slice and its integrations.

## 5. Evaluation Methodology

Evaluation is **multi-stage and additive**. Each pipeline stage is scored
independently so that a failure can be localized:

1. **Query** — interpretability/clarity signals (when available).
2. **Retrieval** — precision, recall estimate, diversity, reranker lift.
3. **Context assembly** — relevance of the assembled context.
4. **Generation** — relevance, completeness, clarity, usefulness, confidence.
5. **Grounding** — support ratio between claims and cited evidence.
6. **Final answer** — the roll-up.

The engine computes an **overall score** as the average of available stage
scores, then derives three **failure flags** from explicit thresholds. Crucially,
flags are derived from *stage scores and ratios*, not from the overall average —
so a single severe failure (e.g., a low grounding ratio) raises the
hallucination flag even when other stages scored well. This prevents a strong
retrieval score from masking a fabricated answer.

**Graceful degradation:** if a scorer raises, that stage contributes a
zero-weight score and the run continues. The run only fails (with
`EvaluationError`) when *no* stage could be scored at all.

## 6. Quality Scoring Methodology

Two layers:

- **Component scorers** (answer, citation, retrieval) each emit a
  `QualityScoreSet` of named, weighted, [0,1]-clamped dimensions with rationale
  strings. Weights encode editorial judgment about what matters most
  (e.g., answer relevance outweighs answer confidence).
- **Unified scorer** combines whatever component sub-scores plus feedback and
  review signals are present into a single `overall_quality_score`, using a
  weighted average over present sources only. The **hallucination cap** is a
  safety override: any unresolved hallucination forces the score to ≤ 0.3,
  ensuring a fabricated-but-fluent answer can never report as high quality.

This design deliberately tolerates missing inputs. Early in a tenant's life
there may be no feedback and no reviews; the score still computes from the
automated signals and refines as human signal accrues.

## 7. Review Lifecycle

```
flag_* ──> OPEN ──claim──> IN_REVIEW ──resolve──> RESOLVED
              │                            │
              └────────dismiss────────────┴──> DISMISSED
```

- Items are created `OPEN` by either automated evaluation (auto-flag on
  suspected hallucination) or explicit human flagging.
- A reviewer `claim`s an item, moving it to `IN_REVIEW`.
- `resolve` records an outcome (confirmed / rejected / corrected / no action),
  closes the item, and **emits a `REVIEW_COMPLETED` learning event** — the human
  judgment becomes a durable signal.
- High-severity reasons (hallucination, policy issue) are enqueued at elevated
  priority so they surface first in `open_items`.
- All transitions are guarded; illegal transitions raise
  `InvalidReviewTransitionError`. All access is tenant-scoped.

## 8. Analytics Lifecycle

1. Service operations emit lightweight `AnalyticsEvent`s.
2. `AnalyticsService` folds them into per-tenant and per-workspace
   `ScopeAnalytics` accumulators.
3. Derived metrics (positive/negative feedback rate, citation approval rate,
   review backlog, average quality, quality trend) are computed on read.
4. Dashboards query tenant- and workspace-level rollups.

Analytics are intentionally separate from learning events: analytics answer
*"how are we doing?"* at aggregate scale; learning events answer *"what specific
thing should we fix, and can we prove why?"* with an audit trail.

## 9. Future Model-Improvement Strategy

S2.4 builds the **data asset**; downstream jobs consume it. The intended
consumption paths, none of which run inside this slice:

- **Prompt improvement.** Cluster `poor_answer_detected` and low clarity/
  completeness events to find prompt failure modes; A/B new prompt templates.
- **Ranking improvement.** Use citation acceptance and `retrieval_precision`
  signals as labels to retrain or tune the reranker.
- **Retrieval improvement.** `retrieval_failure_detected` events with low
  source diversity point to chunking and index-coverage gaps.
- **Grounding improvement.** `hallucination_detected` events with their
  evaluation context become a regression set for grounding changes.
- **Model selection.** Aggregate quality-by-model statistics (carried in event
  metadata) inform per-workload model routing.

Because every signal is tenant/workspace-scoped and auditable, improvement work
can be evaluated per segment and rolled out safely — and enterprise customers
can be shown exactly what evidence drove a change.

## 10. Integration Summary

| Integrates with | How |
|-----------------|-----|
| `identity` | `IdentityResolver` resolves tokens to authoritative tenant/workspace/user scope; every artifact is scoped and cross-tenant access is rejected. |
| `orchestration` | `EventPublisher` emits feedback/evaluation/review events onto the workflow bus, enabling downstream automations. |

| `generation` | Provides the answer text and model confidence; satisfies `answer_exists`. |
| `api` | Calls the four `FeedbackService` operations; maps `FeedbackError.code` to responses. |

## 11. Testing

`test_feedback.py` exercises the full public surface with no external
dependencies (33 tests): ratings and validation, all five feedback-capture
modalities plus duplicate/missing-answer handling, the three component scorers
and the unified scorer (including the hallucination cap), the multi-stage
evaluation engine and its failure flags, the review queue and full review
lifecycle (including cross-tenant rejection and learning-event emission),
append-only learning-log filtering, rule-based recommendation generation, and
tenant/workspace analytics aggregation.
