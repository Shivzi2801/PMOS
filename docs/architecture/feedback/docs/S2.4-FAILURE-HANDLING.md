# PMOS Slice S2.4 — Failure Handling Reference

**Version:** 2.4.0
**Audience:** Engineers, solution architects, enterprise customers, SRE/on-call

---

## 1. Philosophy

S2.4 sits on the *observability* side of PMOS, not the serving path. Its
failures must never silently corrupt the feedback data asset, and must never
let a low-quality or fabricated answer appear high-quality. The design follows
three rules:

1. **Fail loud on integrity violations.** Bad ratings, missing targets, and
   duplicate feedback raise typed exceptions rather than writing dirty data.
2. **Fail soft on partial signal.** A single broken scorer degrades one
   dimension; it does not abort an evaluation or block feedback capture.
3. **Never inflate quality on failure.** When in doubt, the system scores lower
   and routes to human review rather than higher.

Every failure is represented by a subclass of `FeedbackError`, each carrying a
stable `code` and a structured `context` dict, and each serializable via
`to_dict()` for logging and API surfacing.

## 2. Failure Catalogue

### 2.1 Missing feedback (`MissingFeedbackError`)

- **When:** A submission arrives with no usable payload for its declared type
  (e.g., a `STAR_RATING` with no star, a `FREE_TEXT` with empty text).
- **Detection:** `FeedbackCollector._build_record` validates the modality before
  constructing a record.
- **Response:** Raise `MissingFeedbackError`; nothing is persisted.
- **Caller guidance:** The `api` slice maps this to a 400-class response. The UI
  should prevent empty submissions client-side; this is the server-side
  backstop.
- **Why it matters:** Prevents null/garbage records from polluting analytics and
  learning signals.

### 2.2 Invalid rating (`InvalidRatingError`)

- **When:** A star rating is outside 1–5, or a confidence/usefulness value is
  outside [0,1].
- **Detection:** At value-object construction (`StarRating`,
  `ConfidenceFeedback`, `UsefulnessFeedback`) — the earliest possible point.
- **Response:** Raise `InvalidRatingError` before any record exists.
- **Caller guidance:** 400-class response; surface the valid range to the UI.
- **Why it matters:** Ratings feed normalized [0,1] math throughout scoring; an
  out-of-range value would silently skew weighted averages.

### 2.3 Duplicate feedback (`DuplicateFeedbackError`)

- **When:** The same user submits the same feedback type for the same
  answer/citation target twice, and `allow_update` is `False`.
- **Detection:** `FeedbackCollector.collect` checks the record's `dedupe_key`
  (SHA-256 over identity + target + type + citation) against the store.
- **Response:** Raise `DuplicateFeedbackError`. With `allow_update=True`, the
  prior record is replaced instead (the legitimate "user changed their mind"
  path).
- **Caller guidance:** Treat as idempotency protection. If the UI supports
  editing feedback, call with `allow_update=True`.
- **Why it matters:** Without dedup, a double-click or retry would
  double-count sentiment and distort feedback rates.

### 2.4 Missing answer (`MissingAnswerError`)

- **When:** Feedback references an `answer_id` that does not exist for the
  tenant.
- **Detection:** `FeedbackCollector` calls the injected `answer_exists(tenant,
  answer_id)` callable (backed by the `generation` slice in production).
- **Response:** Raise `MissingAnswerError`; nothing is persisted.
- **Caller guidance:** 404/409-class response. Usually indicates a stale client
  or a cross-tenant reference attempt.
- **Why it matters:** Feedback on a non-existent answer is unattributable and
  would create orphaned signals.

### 2.5 Missing citation (`MissingCitationError`)

- **When:** Citation feedback references a `citation_id` that does not exist for
  the answer.
- **Detection:** `FeedbackCollector` calls the injected `citation_exists`
  callable (backed by the `grounding` slice).
- **Response:** Raise `MissingCitationError`; nothing is persisted.
- **Caller guidance:** 404/409-class response.
- **Why it matters:** Citation acceptance rate is a key ranking signal; binding
  it to phantom citations would corrupt that signal.

### 2.6 Evaluation failure (`EvaluationError`)

- **When:** An evaluation run produced **no** scorable stages at all (e.g., an
  `EvaluationContext` with no answer, citation, or retrieval input and no
  grounding ratio).
- **Detection:** `EvaluationEngine.evaluate` checks that at least one stage
  produced a score.
- **Response:** Raise `EvaluationError`.
- **Partial-failure contrast:** If *some* stages score but an individual scorer
  raises, the `_safe()` wrapper degrades that stage to a zero-weight
  contribution and the run **succeeds** with a lower overall score and possibly
  a raised failure flag. Only the total-absence case raises.
- **Caller guidance:** A raised `EvaluationError` is a programming/integration
  error (the caller supplied an empty context), not a data error — fix the call
  site. Per-scorer degradation is expected operational behavior and is visible
  in the per-stage breakdown.
- **Why it matters:** One flaky heuristic must never block the entire feedback
  loop, but a meaningless empty evaluation must never be recorded as if real.

### 2.7 Review-workflow failure (`ReviewWorkflowError` and subclasses)

Two specializations:

- **`ReviewItemNotFoundError`** — `ReviewQueue.get`/`update` for an unknown
  `review_id`, or a read scoped to the wrong tenant. Maps to 404.
- **`InvalidReviewTransitionError`** — an illegal state-machine move
  (e.g., resolving an item that was never claimed, or acting on a closed item).
  Maps to 409.

- **Detection:** In the `ReviewItem` state machine (`_transition`) and the queue
  lookups.
- **Response:** Raise the specific subclass; the item is left unchanged.
- **Cross-tenant guard:** `HumanReviewService` asserts same-tenant access before
  any transition; a mismatch is rejected rather than silently operating on
  another tenant's item.
- **Why it matters:** Review state is an audit surface for enterprise
  compliance; transitions must be total-ordered and tamper-evident.

### 2.8 Analytics aggregation failure (`AnalyticsAggregationError`)

- **When:** An aggregation receives structurally invalid input (e.g., a quality
  trend requested over malformed score data).
- **Detection:** In `AnalyticsService`/`ScopeAnalytics` computation paths.
- **Response:** Raise `AnalyticsAggregationError`. Analytics are a read-side
  concern, so a failure here degrades reporting but **does not** affect feedback
  capture or evaluation, which run on independent paths.
- **Caller guidance:** Dashboards should render a partial/"data unavailable"
  state rather than failing the whole view.
- **Why it matters:** Reporting failures must never cascade into the write path.

### 2.9 Identity-context failure (`IdentityContextError`)

- **When:** An `IdentityContext` is built without a tenant or workspace, or a
  user-required operation is attempted without a `user_id`
  (`require_user()`).
- **Detection:** At context construction and at `require_user()` call sites.
- **Response:** Raise `IdentityContextError`.
- **Caller guidance:** 401/403-class response — the caller must resolve a valid
  identity via the `identity` slice first.
- **Why it matters:** Tenant isolation is a hard security boundary; an
  unscoped operation is never permitted to proceed.

## 3. Failure-Mode Summary Table

| Failure | Exception | Stage | Data written? | Typical HTTP |
|--------|-----------|-------|---------------|--------------|
| Missing feedback payload | `MissingFeedbackError` | Capture | No | 400 |
| Out-of-range rating | `InvalidRatingError` | Capture (value object) | No | 400 |
| Duplicate submission | `DuplicateFeedbackError` | Capture | No (unless `allow_update`) | 409 |
| Unknown answer | `MissingAnswerError` | Capture | No | 404 |
| Unknown citation | `MissingCitationError` | Capture | No | 404 |
| Empty evaluation | `EvaluationError` | Evaluation | No | 500 (integration bug) |
| Single scorer crash | *(none — degraded)* | Evaluation | Yes (zero-weight stage) | 200 |
| Unknown review item | `ReviewItemNotFoundError` | Review | No | 404 |
| Illegal review transition | `InvalidReviewTransitionError` | Review | No | 409 |
| Bad analytics input | `AnalyticsAggregationError` | Analytics (read) | N/A | 500 (report only) |
| Unscoped identity | `IdentityContextError` | All | No | 401/403 |

## 4. Degradation Strategy (the "fail soft" path)

The only intentional soft-failure is **per-scorer degradation** in the
evaluation engine. Rationale and guarantees:

- Each scorer call is wrapped by `_safe()`. An exception is caught, logged, and
  converted to a zero-weight score for that stage.
- A zero-weight contribution does not drag the weighted average toward zero (it
  is excluded from the denominator), but the *absence* of that stage's positive
  signal naturally lowers confidence and can trip a failure flag.
- The per-stage breakdown records which stage degraded, so operators can see
  exactly what happened.
- This guarantees the feedback loop keeps producing signal even while one
  heuristic is being fixed or a dependency is flaky.

Everything else fails hard, because every other failure represents either a
data-integrity violation or a security boundary that must not be crossed.

---

## 5. Observability of Failures

Failures are visible through the metrics surface (`metrics.py`):

- `citation_failures` and `hallucination_reports` increment when the evaluation
  engine raises the corresponding flags or a hallucination is flagged for
  review.
- `review_queue_size` is kept current as items are opened and closed, so a
  backlog spike (often a downstream symptom) is immediately visible.
- Every raised `FeedbackError` carries `code` + `context`, designed to be
  logged structurally and alerted on by code.

The guiding principle: **integrity failures are loud and typed; signal gaps are
soft and observable; quality is never inflated to mask either.**
