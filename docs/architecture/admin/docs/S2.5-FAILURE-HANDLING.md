# PMOS — Wave 2 — Slice S2.5

# Administration & Governance Module — Failure Handling

> Package: `backend/admin/`
> Version: `2.5.0`

This document describes how the Administration & Governance module behaves when things
go wrong: the error model, every failure mode by subsystem, the guarantees that hold
under failure, and the operational signals an operator should watch.

---

## 1. Error Model

All failures are expressed as typed exceptions descending from `AdminError`
(`errors.py`). Nothing in the slice raises a bare `Exception` or a builtin error across
a service boundary. Each `AdminError` carries:

| Field | Purpose |
|---|---|
| `code` | Stable machine-readable identifier (e.g. `tenant.state_invalid`). |
| `http_status` | Status an HTTP edge should map to. |
| `message` | Human-readable explanation. |
| `details` | JSON-serialisable structured context. |

The edge layer catches `AdminError`, calls `to_dict()`, and responds — it never needs to
know the concrete type. Internal callers that need to branch on a specific failure catch
the specific subclass.

### 1.1 Status Mapping

| Exception | `http_status` | Meaning |
|---|---|---|
| `*NotFoundError` | 404 | Target entity does not exist. |
| `ConfigurationValidationError` | 422 | Settings failed schema validation. |
| `ConfigurationVersionConflictError` | 409 | Optimistic-concurrency version mismatch. |
| `TenantStateError`, `WorkspaceStateError` | 409 | Illegal lifecycle transition. |
| `QuotaExceededError` | 429 | Limit reached; back off / retry later. |
| `GovernanceViolationError` | 422 | Subject violates a critical governance rule. |
| `PolicyEvaluationError` | 400 | Malformed policy/context could not be evaluated. |
| base `AdminError` | 400 | Generic administrative failure. |

## 2. Failure Modes by Subsystem

### 2.1 Configuration

- **Invalid settings** → `ConfigurationValidationError`. The validator collects **all**
  violations into `details.violations`, not just the first, so a caller can fix
  everything in one round trip. State is untouched — validation runs *before* any write.
- **Concurrent update** → `ConfigurationVersionConflictError`. `update()` requires the
  caller's `expected_version` to match the stored head. On mismatch the write is rejected
  and the caller must re-read and retry. This prevents lost updates without locking the
  caller out.
- **Missing key** → `ConfigurationNotFoundError` on get/update/rollback of an unknown
  scope key.
- **Bad rollback target** → rollback to a non-existent version raises
  `ConfigurationNotFoundError`; history is never mutated, so a failed rollback leaves the
  chain intact.

*Guarantee:* configuration is append-only and versioned. No partial write can occur —
validation and checksum happen before the new version is committed, and a failure leaves
the previous head as the in-force version.

### 2.2 Feature Flags

- **Unknown flag** → `is_enabled` returns **False** (fail-safe off); it does not raise.
  Lifecycle methods (`enable`, `set_rollout`, overrides) on an unknown flag raise
  `FeatureFlagNotFoundError`.
- **Invalid rollout** → `set_rollout` outside `0–100` raises `FeatureFlagError`.

*Guarantee:* the read path never throws and never fails open. An operator error or a
missing flag results in the feature being *off*, which is the safe direction for a gate.

### 2.3 Policy Engine

- **No matching rule** → the policy's `default_effect` applies. For `AccessPolicy` that
  default is **DENY**, so an unmatched access request is refused, never allowed.
- **Malformed condition / context** → `PolicyEvaluationError`. A missing attribute on the
  context is treated as "condition not satisfied" for positive operators rather than
  crashing the evaluation.
- **Unknown policy id** → `PolicyNotFoundError` from `get`/`unregister`.

*Guarantee:* access decisions fail **closed**. Any ambiguity, absence, or error resolves
to DENY.

### 2.4 Quotas

- **Limit exceeded (enforced)** → `QuotaExceededError` with `quota_key`, `limit`,
  `current`, `requested`; `QUOTA_VIOLATIONS` metric; security audit event. Usage is **not**
  incremented on a rejected consume — the rejected request does not count against the
  window.
- **Limit exceeded (shadow, `enforced=False`)** → no exception; usage is recorded and the
  call is allowed. This is the safe way to observe a limit before turning it on.
- **No quota defined** → treated as unlimited; the call is allowed. Absence of a limit
  never blocks traffic.
- **Unknown quota on management ops** → `QuotaNotFoundError`.

*Guarantee:* `check_and_consume` is atomic under a lock — concurrent callers cannot race
past a limit, and a rejected consume has no side effect on usage.

### 2.5 Tenant & Workspace Administration

- **Illegal transition** → `TenantStateError` / `WorkspaceStateError`, validated against
  the `_TRANSITIONS` map. Re-issuing the current state is an idempotent no-op, not an
  error.
- **Unknown entity** → `TenantNotFoundError` / `WorkspaceNotFoundError`.
- **Workspace under a missing/unusable tenant** → `create_workspace` raises before any
  state is created.
- **Workspace violates critical governance** → `create_workspace` / `update_settings`
  propagate `GovernanceViolationError` from the pre-flight `governance.enforce()`; no
  workspace is created or mutated.

*Guarantee:* lifecycle changes are all-or-nothing. The governance and tenant-existence
checks run *before* any mutation, so a rejected operation leaves no orphaned config or
half-created entity.

### 2.6 Governance

- `validate()` **never raises** — it returns the list of violated rules for reporting.
- `enforce()` raises `GovernanceViolationError` (with the full `violations` list in
  `details`) only when a **CRITICAL** rule is violated; WARNING/INFO violations are
  returned/logged but allowed.
- A malformed rule (missing name or condition) is rejected at `register_rule` time with
  `GovernanceError`, so bad rules never enter the registry.

### 2.7 Audit

- The store is **append-only**; there is no update or delete path by design.
- A failure inside `record()` propagates to the caller so the caller can decide whether
  the originating operation should still be considered successful (operations treat a
  failed audit write as a failure of the whole operation).
- **Tamper detection:** `verify_chain()` recomputes each `event_hash` over canonical JSON
  including the `previous_hash`. Any insertion, deletion, reordering or field edit breaks
  the chain at the first affected event, and `verify_chain()` reports that point.

*Guarantee:* the audit log is tamper-evident. You cannot silently alter history; you can
only detect that it was altered.

### 2.8 Health

- A component/dependency check that **throws** is caught and downgraded to `UNHEALTHY`
  with the error in `detail` — a broken probe never crashes the health endpoint.
- Aggregation is worst-wins: any UNHEALTHY → overall UNHEALTHY; else any DEGRADED →
  DEGRADED; else HEALTHY.
- `readiness()` is false if **any critical dependency** is unhealthy; `liveness()` remains
  true as long as the process can answer.

*Guarantee:* health probes are total functions — they always return a status, even when
the thing they check is on fire.

## 3. Cross-Cutting Guarantees

1. **Validate before mutate.** Every service validates inputs and preconditions before
   touching state, so rejected operations have no side effects.
2. **Fail closed on security.** Access policy defaults to DENY; unknown feature flags read
   as off; enforced quotas block on breach.
3. **Fail open on availability.** Undefined quotas don't block; shadow quotas never block;
   liveness stays up through dependency failure.
4. **Every state change is observable and auditable.** A mutation that cannot be audited
   is treated as a failed mutation.
5. **Idempotent transitions.** Re-issuing a current lifecycle state is a no-op, making
   retries safe.
6. **Optimistic concurrency, not locking the caller out.** Configuration conflicts are
   surfaced as 409s for retry rather than blocking.

## 4. Retry & Recovery Guidance

| Failure | Caller action |
|---|---|
| `ConfigurationVersionConflictError` (409) | Re-read head, re-apply change, retry. |
| `QuotaExceededError` (429) | Back off; retry after the window resets (`details` carries the limit). |
| `*NotFoundError` (404) | Do not retry; the target must be created first. |
| `GovernanceViolationError` (422) | Do not retry as-is; fix the subject per `remediation`. |
| `TenantStateError`/`WorkspaceStateError` (409) | Do not retry; the transition is illegal from the current state. |
| `ConfigurationValidationError` (422) | Fix all `details.violations`, then retry. |

## 5. Operational Signals

Watch these metrics (names in `metrics.py`) to detect failure trends early:

- `QUOTA_VIOLATIONS` rising → tenants hitting limits; investigate before they churn or
  before an enforced rollout bites.
- `GOVERNANCE_VIOLATIONS` rising → drift from policy; check `compliance_report`.
- `POLICY_EVALUATIONS` with a spike in DENYs → possible misconfiguration or attack.
- `HEALTH_STATUS` gauge dropping below HEALTHY → run `check_health` for the failing
  component/dependency detail.
- `CONFIG_CHANGES` spikes → correlate with incidents; use config version history to
  identify and `rollback` a bad change.
- Periodic `AuditService.verify_chain()` returning a break → treat as a security incident.
