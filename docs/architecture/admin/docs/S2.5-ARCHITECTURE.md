# PMOS — Wave 2 — Slice S2.5

# Administration & Governance Module — Architecture

> Package: `backend/admin/`
> Version: `2.5.0`
> Status: Production
> Scope: Administrative controls, governance, platform configuration, operational
> policies, tenant management, quotas, feature flags, and auditability.

---

## 1. Purpose & Position in PMOS

PMOS (Platform Management & Orchestration System) is composed of vertically-sliced
modules delivered across waves. Slice **S2.5 — Administration & Governance** is the
*control plane* of the platform. Where other slices handle the *data plane*
(ingestion, retrieval, generation, serving), S2.5 governs **who** may do **what**,
**how much** of it they may do, **under which policies**, and produces a **tamper-
evident record** of every administrative decision.

The module is deliberately self-contained and dependency-light: it has **no
third-party runtime dependencies** and persists through pluggable in-memory stores
that satisfy narrow `Protocol` interfaces. This lets S2.5 run identically in unit
tests, local development, and (by swapping a store implementation) production,
without changing a line of business logic.

S2.5 exposes its capabilities to the rest of PMOS through a small number of
**service objects**. Upstream slices never reach into models or registries
directly — they call a service, which validates, mutates, emits metrics, and writes
an audit record as a single cohesive operation. This is the single most important
architectural rule of the slice: **state changes flow through services, and every
state change is observable and auditable.**

```
                 ┌─────────────────────────────────────────────┐
                 │                Other PMOS slices              │
                 │  (ingestion, retrieval, generation, serving)  │
                 └───────────────┬───────────────────────────────┘
                                 │ calls service methods only
            ┌────────────────────┼─────────────────────────────────┐
            ▼                    ▼                                  ▼
   ┌────────────────┐  ┌──────────────────┐            ┌───────────────────┐
   │ Configuration  │  │  Policy / Quota   │            │ Tenant / Workspace│
   │   Service      │  │   / Governance    │            │   Administration  │
   └───────┬────────┘  └────────┬──────────┘            └─────────┬─────────┘
           │                    │                                 │
           └──────────┬─────────┴───────────────┬─────────────────┘
                      ▼                          ▼
              ┌───────────────┐          ┌───────────────┐
              │ Audit Service │          │   Metrics     │
              │ (hash chain)  │          │   Sink        │
              └───────────────┘          └───────────────┘
```

## 2. Layered Structure

The package is organised in three layers:

1. **Domain models** (`models/`) — pure data + invariants, no I/O, no orchestration.
2. **Stores & registries** — thread-safe in-memory persistence behind `Protocol`s.
3. **Services** — orchestration: validation → mutation → metrics → audit.

Cross-cutting concerns (`errors.py`, `metrics.py`) sit beside the services and are
imported by every layer that needs them.

```
backend/admin/

├── errors.py                    # exception hierarchy
```

## 3. Cross-Cutting Concerns

### 3.1 `errors.py` — Exception Hierarchy

Every failure mode in the slice is expressed as a typed exception descending from a
single base, `AdminError`. The base carries four fields that make errors safe to
surface across an API boundary:

- `code` — a stable, machine-readable string (e.g. `quota.exceeded`).
- `http_status` — the status an HTTP edge should map the error to.
- `message` — a human-readable description.
- `details` — a JSON-serialisable dict of structured context.

`AdminError.to_dict()` renders all four into a transport-ready payload, so the edge
layer never has to know the concrete exception type — it catches `AdminError`,
serialises, and responds.

The hierarchy mirrors the service boundaries so callers can catch at whatever
granularity they need:

| Family | Base | Notable subclasses |
|---|---|---|
| Configuration | `ConfigurationError` | `ConfigurationNotFoundError`, `ConfigurationValidationError`, `ConfigurationVersionConflictError` |
| Feature flags | `FeatureFlagError` | `FeatureFlagNotFoundError` |
| Policy | `PolicyError` | `PolicyNotFoundError`, `PolicyEvaluationError` |
| Quota | `QuotaError` | `QuotaNotFoundError`, `QuotaExceededError` |
| Tenant | `TenantError` | `TenantNotFoundError`, `TenantStateError` |
| Workspace | `WorkspaceError` | `WorkspaceNotFoundError`, `WorkspaceStateError` |
| Governance | `GovernanceError` | `GovernanceViolationError` |

Two subclasses carry extra structured fields because callers need them
programmatically: `QuotaExceededError` exposes `quota_key`, `limit`, `current`, and
`requested`; `GovernanceViolationError` exposes the full `violations` list. These
become part of `details` automatically.

### 3.2 `metrics.py` — Observability Sink

Observability is a first-class output of the slice, not an afterthought. `metrics.py`
defines a minimal `MetricsSink` `Protocol` with three verbs — `increment` (counters),
`gauge` (point-in-time values), and `observe` (distributions) — each accepting an
optional `tags` mapping.

`InMemoryMetrics` is the default, thread-safe implementation used in tests and local
runs; it records counters, gauges and observations and offers `snapshot()`,
`counter_value()`, `gauge_value()`, `observations()` and `reset()` for assertions.

The module holds a process-global sink reachable via `get_metrics()` and replaceable
via `set_metrics_sink()`. **Critically, services resolve the sink lazily** — they call
`get_metrics().increment(...)` at the moment of the event, never capturing the global
at import time. This late binding is what lets an operator (or a test) swap the sink
after the services have already been constructed. A canonical set of metric-name
constants (`ADMIN_ACTIONS`, `CONFIG_CHANGES`, `POLICY_EVALUATIONS`,
`QUOTA_VIOLATIONS`, `QUOTA_CHECKS`, `GOVERNANCE_VIOLATIONS`, `TENANT_ACTIVITY`,
`WORKSPACE_ACTIVITY`, `FEATURE_FLAG_EVALUATIONS`, `AUDIT_EVENTS`, `HEALTH_STATUS`)
keeps metric names consistent across the whole slice.

## 4. Domain Models (`models/`)

### 4.1 `_base.py` — Foundations

- `utcnow()` — single source of timezone-aware UTC timestamps; every model uses it so
  ordering and hashing are consistent.
- `new_id(prefix)` — generates prefixed, collision-resistant identifiers
  (e.g. `ten_…`, `wsp_…`, `pol_…`), making IDs self-describing in logs.
- `DomainModel` — the base dataclass. Its `to_dict()` recursively serialises nested
  models, enums (to their `value`), datetimes (to ISO-8601) and collections, so any
  model can be rendered to JSON for transport, audit, or checksum without bespoke code.

The enums defined here are the slice's shared vocabulary and the reason illegal states
are largely unrepresentable: `TenantState`, `WorkspaceState`, `PolicyType`,
`PolicyEffect`, `QuotaScope`, `QuotaPeriod`, `AuditCategory`, `HealthStatus`,
`GovernanceSeverity`.

### 4.2 `configuration.py`

`_VersionedConfig` is the internal wrapper binding a settings dict to a `version`
integer, `created_at`/`updated_at` timestamps, an `updated_by` actor, and a `checksum`.
The three public models — `TenantConfiguration`, `WorkspaceConfiguration`,
`SystemConfiguration` — are thin, well-typed faces over that wrapper, differing only in
the scope they bind to (a tenant id, a workspace id, or the platform singleton).

### 4.3 `feature_flag.py`

`FeatureFlag` holds the global `enabled` switch, an optional `rollout_percentage`, and
`tenant_overrides` / `workspace_overrides` maps. Its `evaluate(subject)` method encodes
the resolution order precisely: **workspace override → tenant override → rollout
percentage → global default**. The rollout decision is *deterministic*: the flag key
and the subject key are hashed with SHA-256 into a bucket in `[0, 100)`, so the same
subject always lands on the same side of the threshold and a rollout can be raised
monotonically without re-shuffling who already had the feature.

### 4.4 `policy.py`

`Policy` is the general rule container: `type`, `enabled`, `priority` (lower evaluates
first), `default_effect`, an ordered `rules` list, and optional `tenant_id`/
`workspace_id` scoping. A rule is a small dict — `{match, when:{attr, op, value},
effect}`. Specialisations set safe defaults appropriate to their job:
`AccessPolicy` defaults to **DENY** (deny-by-default security), `RetentionPolicy` adds
`retention_days` + `hard_delete` and defaults to ALLOW, and `GovernanceRule` carries a
nested boolean `condition` tree plus `severity`, `remediation` and `tags`.

### 4.5 `quota.py`

`Quota` defines a limit for a `(scope, tenant, workspace, period)` tuple with a `limit`,
a `unit`, and an `enforced` flag (allowing *shadow* quotas that measure but never
block). Its `key` property produces the canonical string the usage store is indexed by,
and `period_seconds()` maps the `QuotaPeriod` to a rolling-window length (with `TOTAL`
meaning "never resets"). `UsageLimit` is the mutable counterpart: `current` usage plus a
`window_start`, with `window_expired()` driving automatic window resets.

### 4.6 `audit.py`

`AdminAction` records *intent* — the actor, the action, the target, and a context dict.
`AuditEvent` is the *immutable, sealed* record: alongside the action data it carries a
`sequence` number, the `previous_hash`, and its own `event_hash`, computed over a
canonical JSON encoding that includes the previous hash — forming a blockchain-style
chain where altering, deleting, or reordering any event breaks every hash after it.
`PlatformHealth` is the aggregate health snapshot (overall `status`, per-`component` and
per-`dependency` detail, and `checked_at`).

## 5. Configuration Subsystem

Three files cooperate to make configuration **validated, versioned, and reversible**.

### 5.1 `configuration_validator.py`

`ConfigurationValidator` holds a `FieldSpec`-based schema per scope. A `FieldSpec`
declares `type`, `required`, `default`, `choices`, an optional `validator` callable, and
a `description`. `validate(scope, settings)` returns a **normalised** dict (defaults
filled, types checked, choices enforced, custom validators run) or raises
`ConfigurationValidationError` carrying a list of every violation found — not just the
first. When no schema is registered for a scope it falls back to a lenient
JSON-serialisability check, so unknown scopes still cannot store unencodable data.

### 5.2 `configuration_registry.py`

`ConfigurationRegistry[C]` is a generic, thread-safe, in-memory versioned store. Beyond
the obvious `get` / `exists` / `list_keys`, it keeps **full version history** per key.
`update()` supports **optimistic concurrency** via an `expected_version` argument — a
mismatch raises `ConfigurationVersionConflictError` rather than silently clobbering a
concurrent write. `rollback()` does **not** rewrite history; it reads an older version
and *appends* it as a new version, so the audit trail of "we rolled back" is itself
preserved. Each stored version carries a SHA-256 `checksum` of its settings.

### 5.3 `configuration_service.py`

`ConfigurationService` is the orchestrator that upstream code actually calls. It wires
the validator, the registries (system/tenant/workspace), the audit service and metrics
together. It owns `init_system`/`get_system`, full CRUD-plus-history for tenant and
workspace configuration, `rollback`, and — importantly — `effective_config()`, which
layers **system → tenant → workspace** via a deep merge so a caller can ask "what is the
final configuration in force for this workspace?" in one call. Every mutation funnels
through `_record_change()`, which emits the `CONFIG_CHANGES` metric and writes an audit
event, guaranteeing no configuration change is ever silent.

## 6. Feature Flags — `feature_flag_service.py`

`FeatureFlagService` manages the full flag lifecycle: `create_flag`, `get`, `list`,
`enable`/`disable`, `set_rollout` (validated to `0–100`), the four override mutators
(`set`/`clear` × `tenant`/`workspace`), `delete_flag`, and the hot-path `is_enabled`.
`is_enabled` delegates the actual decision to `FeatureFlag.evaluate`, treats an unknown
flag as **off** (fail-safe), and emits `FEATURE_FLAG_EVALUATIONS` on every call so
rollout behaviour is observable in aggregate. Lifecycle mutations are audited; the
read-path evaluation is metered but not audited (it is far too high-volume to audit
per-call, and carries no state change).

---

## 7. Policy Engine — `policy_engine.py`

The engine evaluates three policy families against a runtime **context** dict.

- **Access** (`evaluate_access`): candidate policies are sorted by `priority` ascending;
  the first matching rule's effect wins; if nothing matches, the policy's
  `default_effect` applies — and `AccessPolicy` defaults to DENY, so the *secure
  default* is to refuse.
- **Retention** (`resolve_retention`): the most *specific* applicable policy wins,
  resolved by specificity **workspace > tenant > platform**, answering "how long do we
  keep this, and do we hard-delete?".
- **Compliance** (`compliance_check`): governance-typed policies are evaluated to
  surface matches as findings.

A `PolicyDecision` dataclass packages the outcome (`effect`, the deciding `policy_id`/
`rule`, and a `reason`) so the caller gets an explainable result, not just a boolean.
Condition evaluation supports `eq, ne, in, not_in, gt, gte, lt, lte, contains, exists`
over dotted attribute paths (`_get_attr`). Every evaluation emits `POLICY_EVALUATIONS`.

---

## 8. Quota Subsystem — `quota_service.py`

`QuotaService` covers all five scopes required by the slice — **request, ingestion,
storage, retrieval, generation** — through one uniform mechanism. `define_quota`
registers a limit; `check_and_consume` is the atomic hot-path that, under a lock,
resolves the most specific quota for the `(scope, tenant, workspace, period)`, resets the
window if `window_expired()`, and either records the consumption or — if the limit would
be breached — raises `QuotaExceededError`, emits `QUOTA_VIOLATIONS`, and writes a
*security* audit event. Two design choices matter operationally: an **undefined** quota
is treated as *unlimited* (absence of a limit never blocks legitimate traffic), and a
quota with `enforced=False` runs in **shadow mode** — it counts and meters but never
blocks, which is how new limits are safely dialled in. `usage_report` exposes current
usage and utilisation for dashboards; `QUOTA_CHECKS` is emitted on every check.

---

## 9. Tenant Administration — `tenant_admin_service.py`

`TenantAdminService` owns the tenant lifecycle and nothing else. A `Tenant` moves through
`TenantState` values governed by an explicit `_TRANSITIONS` map, so illegal jumps (e.g.
reactivating a deleted tenant) raise `TenantStateError` instead of corrupting state.
`create_tenant` also **provisions a default configuration** through the configuration
service; `delete_tenant` is a **soft delete** that tears the configuration down while
preserving the audit trail; `suspend`/`reactivate`/`activate` move the state machine; and
`manage_configuration` is the audited path for tenant-scoped config edits. Transitions are
idempotent (re-issuing the current state is a no-op, not an error). Every mutation emits
`TENANT_ACTIVITY` and writes a `TENANT` audit event.

---

## 10. Workspace Administration — `workspace_admin_service.py`

`WorkspaceAdminService` mirrors the tenant service one level down and adds a governance
gate. `create_workspace` first verifies the parent tenant **exists and is usable**, then
runs `governance.enforce()` as a **pre-flight check** so a workspace that would violate a
critical governance rule is never created; on success it provisions workspace
configuration. `update_settings` re-runs governance on the proposed settings;
`archive`/`unarchive`/`delete` drive the `WorkspaceState` machine via its own
`_TRANSITIONS` map. Every mutation emits `WORKSPACE_ACTIVITY` and a `WORKSPACE` audit
event.

---

## 11. Audit Framework — `audit_service.py`

`AuditService` is the tamper-evident system of record. Behind an `AuditStore` `Protocol`
(default `InMemoryAuditStore`, append-only) it `record()`s each event by sealing it into
the hash chain: assigning the next `sequence`, stamping the `previous_hash`, and computing
the `event_hash`. `query()` performs AND-filtering across actor/category/target/time.
`verify_chain()` walks the chain and reports the first break, detecting tampering,
deletion, or reordering. Convenience wrappers `log_admin_action()` and
`log_security_event()` standardise the two most common entry points, and `AUDIT_EVENTS`
is emitted per record.

---

## 12. Platform Health — `health_service.py`

`HealthService` aggregates liveness and readiness. Callers `register_component(name, fn)`
and `register_dependency(name, fn, critical=…)`; each `CheckFn` returns a `ComponentCheck`
(status, detail, latency). `_run` treats a *throwing* check as `UNHEALTHY` rather than
letting it crash the probe. `check_health` aggregates with a worst-wins rule (any
`UNHEALTHY` → UNHEALTHY; else any `DEGRADED` → DEGRADED; else HEALTHY) and emits the
`HEALTH_STATUS` gauge. `liveness()` reports process aliveness; `readiness()` is true only
when every **critical** dependency is healthy — the signal an orchestrator uses to decide
whether to route traffic.

---

## 13. Governance Framework — `governance_service.py`

`GovernanceService` is where platform-wide rules live. `register_rule` admits a
`GovernanceRule` (name + boolean `condition` tree required). The pivotal semantic
convention: **a condition that evaluates true means the subject *violates* the rule** —
conditions describe the bad state. `validate(subject)` returns the list of violated rules
**without raising**, suitable for reporting; `enforce(subject)` raises
`GovernanceViolationError` when any **CRITICAL** rule is violated while allowing warnings
through, suitable for gating (this is what the workspace service calls). The condition
evaluator supports nested `all`/`any`/`not` trees over leaf comparisons.
`compliance_report` aggregates violations across many subjects for a governance
dashboard, and `GOVERNANCE_VIOLATIONS` is emitted whenever violations are found.

## 14. Lifecycles

### 14.1 Configuration Lifecycle

```
register schema ─▶ create (validate→normalise→v1, checksum, audit, metric)
       │
       ├─▶ update (validate→optimistic version check→v_n+1→audit→metric)
       ├─▶ rollback (read v_k → append as v_n+1, history preserved, audit)
       ├─▶ effective_config (system ⊕ tenant ⊕ workspace deep-merge, read-only)
       └─▶ delete (drop key, audit)
```
Configuration is never mutated in place: every change creates a new version with its own
checksum and audit event, and rollback is itself a forward-moving, audited event. This
makes "what changed, when, by whom, and can we undo it?" always answerable.

### 14.2 Tenant Administration Lifecycle

```
        create_tenant
            │  (provisions default config, audit)
            ▼
        PENDING ──activate──▶ ACTIVE ◀──reactivate── SUSPENDED
                                 │  │                    ▲
                                 │  └──────suspend───────┘
                                 │
                              delete (soft; config torn down)
                                 ▼
                              DELETED  (terminal)
```
Transitions are validated against `_TRANSITIONS`; illegal moves raise `TenantStateError`;
identical-state re-issues are idempotent. Every move is metered (`TENANT_ACTIVITY`) and
audited (`TENANT`).

### 14.3 Workspace Lifecycle

```
   create_workspace
       │  tenant must exist & be usable
       │  governance.enforce() pre-flight  ── violates CRITICAL ─▶ rejected
       ▼
     ACTIVE ──archive──▶ ARCHIVED ──unarchive──▶ ACTIVE
       │
       └── delete ─▶ DELETED (terminal)
   update_settings → governance re-check → versioned config update
```

### 14.4 Quota Enforcement Lifecycle

```
 define_quota(scope, period, limit, enforced)
        │
        ▼
 check_and_consume(scope, amount, ctx)  ── (under lock) ──┐
        │                                                 │
        ├─ no matching quota ─────────────▶ ALLOW (unlimited)
        ├─ window expired ─▶ reset window
        ├─ current+amount ≤ limit ─▶ record usage ─▶ ALLOW (QUOTA_CHECKS)
        └─ would exceed:
               ├─ enforced=False (shadow) ─▶ record, ALLOW
               └─ enforced=True ─▶ QuotaExceededError
                                   + QUOTA_VIOLATIONS + security audit
```

### 14.5 Governance Lifecycle

```
 register_rule(condition, severity)
        │
        ├─ validate(subject) ─▶ [violations]            (report, never raises)
        ├─ enforce(subject)  ─▶ raise on CRITICAL        (gate, used by workspaces)
        └─ compliance_report(subjects) ─▶ aggregate findings (GOVERNANCE_VIOLATIONS)
```

## 15. How S2.5 Interacts With the Rest of PMOS

- **Data-plane slices** (ingestion/retrieval/generation) call `QuotaService.check_and_
  consume` before doing expensive work, and read `FeatureFlagService.is_enabled` to gate
  behaviour per tenant/workspace.
- **Request edges** call `PolicyEngine.evaluate_access` to authorise actions and map any
  raised `AdminError` to an HTTP response via `to_dict()`.
- **Provisioning flows** call `TenantAdminService` / `WorkspaceAdminService`, which in
  turn drive `ConfigurationService` and `GovernanceService`.
- **Operators / dashboards** read `HealthService` probes, `QuotaService.usage_report`,
  and `GovernanceService.compliance_report`, and consume the metrics emitted into the
  shared `MetricsSink`.
- **Compliance / security** rely on `AuditService.verify_chain()` for tamper evidence.

Everything funnels through services; every state change is validated, metered, and
audited. That uniformity is the architecture.
