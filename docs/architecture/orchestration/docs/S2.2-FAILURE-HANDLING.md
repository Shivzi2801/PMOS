# S2.2 — Orchestration Engine: Failure Handling

The single most valuable thing an orchestration layer provides is **predictable
behaviour when things go wrong**. Any system can run a happy path. This document
explains exactly how PMOS orchestration handles every category of failure, in
language a Product Manager can carry into a room with engineers, architects,
enterprise security reviewers, or investors.

---

## The mental model: classify, then react

Every failure in the system is first **classified** into one of two buckets:

- **RETRYABLE** — the failure *might* go away if we try again (a timeout, a rate
  limit, a brief network or dependency outage).
- **TERMINAL** — trying again cannot help (bad input, a logic bug, an
  authorization failure, a permanently malformed document).

This single decision drives everything downstream. We never waste time retrying
something that can't succeed, and we never give up on something that probably
would have worked on the second try. Classification lives in `errors.py`
(`classify()`), and each orchestration error self-declares its class.

---

## Failure categories and how each is handled

### 1. Module failures
*A previous slice (retrieval, indexing, generation, …) raises an error.*

- The engine catches it during the failing step and records a `StepTrace` with
  `state="failed"` and the error message, so the result always shows exactly
  which step broke.
- The error keeps its classification. A slice signalling a transient problem
  (e.g. `TransientModuleError`) stays RETRYABLE; anything else is wrapped as a
  `ModuleFailureError` (TERMINAL).
- **Outcome:** retryable → handed up to the retry manager; terminal → workflow
  fails immediately with a clear trace.

### 2. Timeouts
*A workflow takes longer than its allotted budget.*

- Every workflow has a `default_timeout_s` and the context carries a deadline.
- The engine checks the deadline **between steps** and raises
  `StepTimeoutError` (RETRYABLE) if exceeded.
- **Outcome:** the run is retried under the policy; if it keeps blowing the
  budget, it ends in retry exhaustion (see below). This protects the platform
  from a single slow dependency hanging a request indefinitely.

### 3. Partial failures
*Some steps succeed, some non-critical steps fail.*

- A step can be marked `optional=True`. If an optional step fails, the engine
  records the failure under `_partial_failure::<step>` in the context and
  **continues** rather than aborting.
- The final `WorkflowResult` carries `partial=True` so callers know the output
  is usable but degraded.
- **Outcome:** the user still gets an answer; the degradation is visible and
  measurable. (Today the core query/ingestion/reindex steps are all required;
  the mechanism exists for optional enrichment steps added later.)

### 4. Dependency failures
*A downstream service a slice relies on is temporarily unavailable.*

- Slices surface these as `DependencyError` (RETRYABLE).
- **Outcome:** the retry manager backs off and retries, giving the dependency
  time to recover, before eventually failing the workflow if it never does.

### 5. Workflow cancellations
*An operator or user cancels an in-flight job.*

- Cancellation is **cooperative**: `orchestrator.cancel(job_id)` flips a flag on
  the job's context.
- The engine checks that flag between steps and the retry manager checks it
  before each attempt, so a cancelled job stops promptly at the next safe
  boundary and never starts new work.
- **Outcome:** state → CANCELLED, no `WORKFLOW_FAILED` noise (cancellation is
  not a failure), and the job is recorded in history as cancelled.

### 6. Retry exhaustion
*A retryable failure never recovers within the allowed attempts.*

- After `max_attempts`, the retry manager raises `RetryExhaustedError` carrying
  the attempt count and the last underlying error.
- **Outcome:** state → FAILED, `WORKFLOW_FAILED` published, failure counted in
  metrics, full step trace preserved for diagnosis.

## The retry strategy in detail

Configured by `RetryPolicy`:

| Setting | Meaning | Default |
|---|---|---|
| `max_attempts` | total tries including the first | 3 |
| `base_delay_s` | first backoff | 0.5s |
| `max_delay_s` | ceiling on any single backoff | 30s |
| `multiplier` | exponential growth factor | 2.0 |
| `jitter` | ± randomisation to avoid thundering herd | 0.1 |

**Exponential backoff** means delays grow: 0.5s, 1s, 2s, … capped at 30s.
**Jitter** spreads retries so a fleet of clients doesn't all retry in lockstep
and re-overload a recovering dependency. **Failure classification** ensures only
RETRYABLE errors ever enter this loop — a terminal error aborts on the first
occurrence with zero wasted retries.

Each retry: increments the metric `retry_count`, marks the job's retry counter,
moves the state RUNNING → RETRYING (so observers see it), publishes
`JOB_RETRIED` with the attempt number and delay, then sleeps and re-attempts.

## Failure recovery and guarantees

- **No silent corruption of state.** All state changes go through the state
  machine, which rejects illegal transitions. A terminal job can never be
  silently reopened.
- **Reindex ordering invariant.** The reindex workflow always deletes stale
  index entries *before* writing new ones, so retrieval never serves two
  versions of the same document. If the reprocess/reindex step fails after the
  delete, the workflow fails visibly and the document is flagged for re-run —
  preferable to silently serving stale data.
- **Observability never breaks execution.** Event handlers run inside a
  try/except; a broken metrics exporter or webhook increments
  `handler_errors` but cannot fail a customer's workflow.
- **Internal faults are contained.** If the orchestrator itself hits an
  unexpected error (not a known slice error), it increments
  `orchestration_errors`, fails the job cleanly, and still returns a structured
  result rather than crashing the process.
- **Every failure is explainable.** The returned `WorkflowResult` always
  includes the final state, the error message, the error class, the retry count,
  and a per-step trace showing exactly where and why it stopped.

---

## What this means for the business

- **Resilience is built in, not bolted on.** Customers experience fewer
  transient errors because the platform absorbs them automatically.
- **Failures are debuggable in minutes, not hours.** A job id resolves to a
  complete state history and step trace.
- **Safe by default for enterprise data.** Cancellation is clean, reindexing
  cannot corrupt search results, and observers can never take down the core.
- **Tunable per workload.** Latency-sensitive query traffic and long-running
  ingestion jobs can each carry their own retry policy and timeout.
