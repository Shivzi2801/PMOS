# S2.2 — Orchestration Engine: Sequence Diagrams

These diagrams show how the orchestration layer drives each workflow and each
significant failure mode over time. They are written in Mermaid (renders in
GitHub, most IDEs, and docs tooling) with a plain-language walkthrough beneath
each so a Product Manager can narrate them without reading the code.

The recurring participants:

- **API** — the caller (backend/api slice or operator tooling)
- **Orchestrator** — central engine (`orchestrator.py`)
- **Registry / JobManager / StateMachine / RetryManager / Engine / EventBus /
  Metrics** — orchestration internals
- **Slices** — the previously-built capability modules

---

## 1. Query workflow — happy path

```mermaid
sequenceDiagram
    participant API
    participant Orchestrator
    participant JobManager
    participant RetryManager
    participant Engine
    participant Retrieval
    participant Context
    participant Generation
    participant Grounding
    participant EventBus
    participant Metrics

    API->>Orchestrator: run("query", {question})
    Orchestrator->>JobManager: create job (PENDING)
    Orchestrator->>Metrics: record_start("query")
    Orchestrator->>EventBus: WORKFLOW_STARTED
    Orchestrator->>RetryManager: run(attempt)
    RetryManager->>JobManager: state PENDING -> RUNNING
    RetryManager->>Engine: run(workflow, context)
    Engine->>Retrieval: retrieve(question)
    Retrieval-->>Engine: documents
    Engine->>Context: assemble(question, docs)
    Context-->>Engine: assembled context
    Engine->>Generation: generate(question, context)
    Generation-->>Engine: answer
    Engine->>EventBus: ANSWER_GENERATED
    Engine->>Grounding: ground(answer, context)
    Grounding-->>Engine: grounded answer + citations
    Engine-->>RetryManager: step traces
    RetryManager-->>Orchestrator: success
    Orchestrator->>JobManager: state RUNNING -> COMPLETED
    Orchestrator->>Metrics: record_completion + latency
    Orchestrator->>EventBus: WORKFLOW_COMPLETED
    Orchestrator-->>API: WorkflowResult(success, answer, citations)
```

**Walkthrough:** A question comes in. The orchestrator opens a job, announces
the start, and runs the steps in order. Retrieval finds source material, context
assembly packs it for the model, generation writes the answer, grounding checks
it and attaches citations. Each handoff stores its output in the shared context.
On success the job is marked completed and a clean result with the answer and
citations goes back to the caller.

## 2. Ingestion workflow — happy path

```mermaid
sequenceDiagram
    participant API
    participant Orchestrator
    participant Engine
    participant Connectors
    participant Ingestion
    participant Extraction
    participant Resolution
    participant Indexing
    participant EventBus

    API->>Orchestrator: run("ingestion", {source})
    Orchestrator->>Engine: run(workflow, context)
    Engine->>Connectors: fetch(source)
    Connectors-->>Engine: raw documents
    Engine->>Ingestion: ingest(raw docs)
    Ingestion-->>Engine: normalised docs
    Engine->>Extraction: extract(docs)
    Extraction-->>Engine: structured content
    Engine->>Resolution: resolve(content)
    Resolution-->>Engine: resolved entities
    Engine->>Indexing: index(resolved)
    Indexing-->>Engine: index result
    Engine->>EventBus: INDEX_UPDATED(source)
    Engine-->>Orchestrator: step traces
    Orchestrator-->>API: WorkflowResult(success, documents_ingested)
```

**Walkthrough:** New content enters PMOS. Documents are pulled from the source
connector, normalised, parsed into structured content, de-duplicated and
entity-resolved, then written to the index. The `INDEX_UPDATED` event lets
caches and dashboards refresh. The result reports how many documents were
ingested.

## 3. Reindex workflow — happy path

```mermaid
sequenceDiagram
    participant API
    participant Orchestrator
    participant Engine
    participant Indexing
    participant Connectors
    participant Extraction
    participant Resolution
    participant EventBus

    API->>Orchestrator: run("reindex", {document_ids})
    Orchestrator->>Engine: run(workflow, context)
    Engine->>Indexing: delete(document_ids)
    Indexing-->>Engine: deleted (old entries gone)
    alt content supplied
        Note over Engine: use provided updated content
    else content not supplied
        Engine->>Connectors: fetch_by_id(document_ids)
        Connectors-->>Engine: fresh content
    end
    Engine->>Extraction: extract(content)
    Extraction-->>Engine: structured content
    Engine->>Resolution: resolve(content)
    Resolution-->>Engine: resolved entities
    Engine->>Indexing: index(resolved)
    Indexing-->>Engine: index result
    Engine->>EventBus: INDEX_UPDATED(document_ids)
    Engine-->>Orchestrator: step traces
    Orchestrator-->>API: WorkflowResult(success)
```

**Walkthrough:** A known document changed. The stale index entries are deleted
**first**, guaranteeing retrieval never sees two versions at once. The fresh
content is then reprocessed and re-indexed. `INDEX_UPDATED` signals the change.

## 4. Retry then success

```mermaid
sequenceDiagram
    participant Orchestrator
    participant RetryManager
    participant JobManager
    participant Engine
    participant Slice
    participant EventBus
    participant Metrics

    Orchestrator->>RetryManager: run(attempt)
    RetryManager->>JobManager: PENDING -> RUNNING
    RetryManager->>Engine: run (attempt 1)
    Engine->>Slice: call
    Slice-->>Engine: TransientModuleError (RETRYABLE)
    Engine-->>RetryManager: raises retryable error
    RetryManager->>Metrics: record_retry
    RetryManager->>JobManager: RUNNING -> RETRYING
    RetryManager->>EventBus: JOB_RETRIED(attempt 2, delay)
    Note over RetryManager: exponential backoff + jitter
    RetryManager->>JobManager: RETRYING -> RUNNING
    RetryManager->>Engine: run (attempt 2)
    Engine->>Slice: call
    Slice-->>Engine: success
    Engine-->>RetryManager: step traces
    RetryManager-->>Orchestrator: success
    Orchestrator->>JobManager: RUNNING -> COMPLETED
```

**Walkthrough:** A slice hits a transient error. Because it's classified
retryable, the job moves to RETRYING, the retry is counted and announced, the
manager waits (backoff with jitter), then re-runs. The second attempt succeeds
and the job completes. The user never sees the hiccup.

## 5. Retry exhaustion → failure

```mermaid
sequenceDiagram
    participant Orchestrator
    participant RetryManager
    participant Engine
    participant Slice
    participant EventBus
    participant Metrics

    loop max_attempts times
        RetryManager->>Engine: run
        Engine->>Slice: call
        Slice-->>Engine: TransientModuleError (RETRYABLE)
        Engine-->>RetryManager: retryable error
        RetryManager->>Metrics: record_retry
        RetryManager->>EventBus: JOB_RETRIED
    end
    RetryManager-->>Orchestrator: RetryExhaustedError
    Orchestrator->>Metrics: record_failure
    Orchestrator->>EventBus: WORKFLOW_FAILED
    Note over Orchestrator: state -> FAILED, full trace preserved
```

**Walkthrough:** The transient error never clears. After the configured number
of attempts the retry manager gives up with `RetryExhaustedError`. The workflow
is marked FAILED, the failure is counted, `WORKFLOW_FAILED` is published, and the
result carries the last underlying error and a complete step trace for
diagnosis.

## 6. Terminal failure (no retries)

```mermaid
sequenceDiagram
    participant Orchestrator
    participant RetryManager
    participant Engine
    participant Slice

    RetryManager->>Engine: run (attempt 1)
    Engine->>Slice: call
    Slice-->>Engine: ValueError (TERMINAL)
    Engine-->>RetryManager: terminal error (not retried)
    RetryManager-->>Orchestrator: error propagated immediately
    Note over Orchestrator: state -> FAILED, retries = 0
```

**Walkthrough:** Some failures cannot be fixed by retrying — bad input, a logic
error, an auth failure. These are classified terminal, so the retry manager does
not loop. The workflow fails immediately with zero wasted retries.

## 7. Cancellation

```mermaid
sequenceDiagram
    participant Operator
    participant Orchestrator
    participant JobManager
    participant Context
    participant Engine

    Operator->>Orchestrator: cancel(job_id)
    Orchestrator->>JobManager: cancel
    JobManager->>Context: set cancelled flag
    Note over Engine: between steps, checks context.cancelled
    Engine-->>Orchestrator: JobCancelledError
    Orchestrator->>JobManager: state -> CANCELLED
    Note over Orchestrator: no WORKFLOW_FAILED (cancel is not a failure)
```

**Walkthrough:** Cancellation is cooperative. The flag is set on the running
job's context; the engine checks it at the next step boundary and stops cleanly.
The job ends in CANCELLED — distinct from FAILED — so it's not counted as an
error.

## Reading these as a Product Manager

Each diagram answers one stakeholder question:

- **Engineers:** which method is called, in what order, with what data.
- **Architects:** where the boundaries are — the orchestrator never does domain
  work, it only sequences slices.
- **Enterprise customers:** that failures, retries, and cancellations are
  explicit, observable, and safe for their data (note the delete-before-write
  ordering in reindex).
- **Investors:** that reliability is a designed-in platform property — one
  engine makes every current and future workflow resilient and measurable.
