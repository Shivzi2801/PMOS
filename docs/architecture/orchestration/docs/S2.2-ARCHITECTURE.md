# S2.2 — Orchestration Engine: Architecture

## What this slice is

The orchestration layer is the **central execution engine** of PMOS. Every prior
slice (connectors, ingestion, extraction, resolution, indexing, retrieval,
context, generation, grounding) does one job well. None of them knows about the
others. The orchestration layer is the part that knows the *order* things must
happen in, what to do when a step fails, how to retry safely, how to track a
unit of work from start to finish, and how to report what happened.

If you think of the previous slices as specialist departments, orchestration is
the operations manager: it doesn't do the specialist work, it sequences it,
supervises it, and reports on it.

A Product Manager can summarise it in one sentence: **orchestration turns a pile
of independent capabilities into reliable, observable, repeatable business
workflows.**

---

## The three workflows

### 1. Query workflow (the read path)

```
Question → Retrieval → Context Assembly → Generation → Grounding → Response
```

This is what runs every time a user asks PMOS a question. It finds relevant
material (retrieval), packs it into a prompt (context assembly), produces an
answer (generation), checks that answer against the sources and attaches
citations (grounding), and returns a clean response object.

### 2. Ingestion workflow (the write path)

```
Connector → Ingestion → Extraction → Resolution → Indexing
```

This is what runs when new content enters PMOS. It pulls documents from a source
system (connector), normalises them (ingestion), extracts structured content
(extraction), resolves entities and removes duplicates (resolution), and writes
everything into the search index (indexing).

### 3. Reindex workflow (the update path)

```
Document Update → Delete Old Index → Reprocess → Reindex
```

This runs when a document already in PMOS changes. It removes the stale index
entries *first*, reprocesses the fresh content, then writes the new entries.
Deleting before writing is the key safety property: retrieval never sees two
conflicting versions of the same document.

## File-by-file reference

For each file: **why it exists**, **responsibilities**, **inputs**, **outputs**,
and **interactions with previous slices**.

### `__init__.py`
- **Why:** Public front door of the package. One import line gives callers
  everything they need.
- **Responsibilities:** Re-export the orchestrator, dependency bundle, retry
  policy, states, events, and core types.
- **Inputs / Outputs:** None — it is a namespace.
- **Interactions:** None directly; it is what the **API slice** imports.

### `errors.py`
- **Why:** A single, shared error vocabulary. Without it, every module would
  invent its own exceptions and retry logic could never reason about them.
- **Responsibilities:** Define the `OrchestrationError` family and, critically,
  classify every error as **RETRYABLE**, **TERMINAL**, or **UNKNOWN**. This
  classification is what lets the retry manager decide whether trying again
  could possibly help.
- **Inputs:** An exception (for `classify()`).
- **Outputs:** A `FailureClass`.
- **Interactions:** Every slice's errors flow through `classify()`. A transient
  error from, say, the indexing slice becomes retryable; a validation error
  becomes terminal.

### `state_machine.py`
- **Why:** A workflow must move through a known, legal set of states. Allowing
  arbitrary state changes would corrupt job history and hide bugs.
- **Responsibilities:** Enforce the legal transitions between PENDING, RUNNING,
  RETRYING, COMPLETED, FAILED, CANCELLED. Reject illegal moves loudly.
- **Inputs:** A target state.
- **Outputs:** The new state, plus an append-only history of every state the run
  has been in.
- **Interactions:** Used by the job manager for every workflow regardless of
  which slices it touches.

### `event_bus.py`
- **Why:** The orchestrator should not need to know who cares about its events.
  An event bus decouples "something happened" from "who reacts to it."
- **Responsibilities:** Publish lifecycle events; let observers subscribe.
  Guarantee that a broken observer never breaks a workflow.
- **Inputs:** An event type plus payload.
- **Outputs:** Delivered events; an internal `handler_errors` counter.
- **Interactions:** The **API slice** can subscribe to stream progress to users;
  audit/monitoring tooling subscribes for compliance. The ingestion and reindex
  workflows publish `INDEX_UPDATED`; the query workflow publishes
  `ANSWER_GENERATED`.

### `metrics.py`
- **Why:** You cannot run a platform you cannot measure. This is the
  observability backbone.
- **Responsibilities:** Track workflow counts, failures, latency, retries,
  success rate, and internal orchestration errors.
- **Inputs:** Lifecycle signals from the orchestrator.
- **Outputs:** A JSON-serialisable `snapshot()` ready for dashboards or a
  Prometheus exporter.
- **Interactions:** Indirect — it measures the execution of every slice's work
  as seen through the orchestrator.

### `retry_manager.py`
- **Why:** Transient failures are normal in distributed systems (a rate limit, a
  brief network blip). Retrying intelligently turns those into invisible
  hiccups instead of user-facing errors.
- **Responsibilities:** Run a callable with exponential backoff + jitter, a
  maximum attempt count, and failure-class awareness (only retry RETRYABLE
  failures). Stop early if the job is cancelled.
- **Inputs:** A callable, a `RetryPolicy`, a cancellation check.
- **Outputs:** The callable's result, or `RetryExhaustedError`.
- **Interactions:** Wraps the execution of any slice. A flaky retrieval or
  indexing call is retried here.

### `workflow_context.py`
- **Why:** Steps need a shared place to read inputs and publish outputs without
  being tightly coupled to each other.
- **Responsibilities:** Carry the input payload, a scratch data dictionary,
  correlation IDs, a deadline (for timeouts), and a cancellation flag through
  every step of one run.
- **Inputs:** The original request payload.
- **Outputs:** The accumulated `data`, including the final `response`.
- **Interactions:** Each slice-adapter step reads its inputs from and writes its
  outputs to the context.

### `workflow_result.py`
- **Why:** Every run must produce one uniform, serialisable outcome regardless
  of which workflow ran or how it ended.
- **Responsibilities:** Capture success/failure, output, per-step traces,
  timing, retry count, partial-failure flag, and final state.
- **Inputs:** Populated by the orchestrator during execution.
- **Outputs:** `to_dict()` for the API slice to return as JSON.
- **Interactions:** Consumed by the **API slice** and stored in job history.

### `workflow_engine.py`
- **Why:** Separate the *mechanics* of running ordered steps from the *policy* of
  retries and state, so each stays simple.
- **Responsibilities:** Run a workflow's steps in order against a context;
  time each step; check timeout/cancellation between steps; trace each step;
  translate step exceptions into the error taxonomy; allow optional steps to
  degrade rather than abort (partial failure).
- **Inputs:** A `Workflow` and a `WorkflowContext`.
- **Outputs:** A list of `StepTrace`s (and mutations to the context).
- **Interactions:** Calls into every slice indirectly, via the step functions
  defined in the workflow modules.

### `workflow_registry.py`
- **Why:** The orchestrator should look workflows up by name, not hard-code
  imports, so new workflows can be added without touching the core.
- **Responsibilities:** Map workflow names to factory functions that build a
  workflow with its dependencies injected.
- **Inputs:** A workflow name and the dependency bundle.
- **Outputs:** A freshly-built `Workflow`.
- **Interactions:** Holds the factories for the query, ingestion, and reindex
  workflows.

### `query_workflow.py`, `ingestion_workflow.py`, `reindex_workflow.py`
- **Why:** These are the actual business processes — the *only* files that know
  the order of slices and how to call them.
- **Responsibilities:** Define each workflow as an ordered list of named steps;
  each step is a thin adapter that validates inputs, calls a slice, and stores
  the output.
- **Inputs:** The request payload (via context).
- **Outputs:** A `response` object in the context.
- **Interactions — this is where all previous slices connect:**
  - Query: `retrieval → context → generation → grounding`.
  - Ingestion: `connectors → ingestion → extraction → resolution → indexing`.
  - Reindex: `indexing(delete) → extraction → resolution → indexing(write)`,
    optionally re-fetching via `connectors`.
  The adapters are duck-typed (they try several plausible method names) so minor
  API drift in a slice does not break orchestration.

### `job_manager.py`
- **Why:** Someone has to remember what's running, what finished, and what
  happened. That's the job manager.
- **Responsibilities:** Create jobs, track their state machine and live context,
  report status, support cancellation, count retries, and keep workflow history.
- **Inputs:** A workflow name and context (create); a job id (everything else).
- **Outputs:** `Job` records and their serialisable status/history.
- **Interactions:** Slice-agnostic; it tracks every run uniformly.

### `orchestrator.py`
- **Why:** The single public entry point that ties the whole slice together.
- **Responsibilities:** Resolve a workflow, create a job, drive state
  transitions, run the engine wrapped in the retry manager, publish events,
  record metrics, and produce a `WorkflowResult`. It contains **no business
  logic** — only sequencing, retrying, observing, and reporting.
- **Inputs:** A workflow name + payload (or the convenience wrappers
  `run_query` / `run_ingestion` / `run_reindex`).
- **Outputs:** A `WorkflowResult`.
- **Interactions:** Holds the `Dependencies` bundle wiring in every previous
  slice. It is what the **API slice** calls.

### `test_orchestration.py`
- **Why:** Proof the engine behaves under success, retry, cancellation, and
  failure — with every slice faked so tests are fast and deterministic.
- **Responsibilities:** Cover all three workflows, retry logic, state
  transitions, job management, and failure paths (25 tests).

## How a request flows (the full picture)

1. The **API slice** calls `orchestrator.run("query", payload)`.
2. The orchestrator builds the workflow from the **registry** and creates a
   **job** (state: PENDING).
3. It publishes `WORKFLOW_STARTED` and records the start in **metrics**.
4. The **retry manager** runs an attempt: state → RUNNING, the **engine**
   executes each step against the **context**.
5. Each step calls into a previous slice and stores its output.
6. On success: state → COMPLETED, `WORKFLOW_COMPLETED` published, latency
   recorded, `WorkflowResult` returned.
7. On a retryable failure: state → RETRYING, `JOB_RETRIED` published, backoff,
   then a fresh attempt.
8. On a terminal failure or retry exhaustion: state → FAILED,
   `WORKFLOW_FAILED` published.
9. On cancellation: state → CANCELLED.
10. The **job manager** stores the final result in history; the API returns it.

## Why this design holds up for enterprise and investors

- **Single chokepoint for reliability.** Every workflow inherits retries,
  timeouts, cancellation, and observability for free. Reliability is a platform
  property, not something each feature reimplements.
- **Auditability.** Every run has a job id, a full state history, per-step
  traces, and an event stream — the raw material for compliance and debugging.
- **Extensibility.** New workflows are just a new factory in the registry. The
  core engine never changes.
- **Decoupling.** Slices stay independent and individually testable; the
  orchestrator is the only place that knows how they compose. That keeps the
  blast radius of any one change small.
