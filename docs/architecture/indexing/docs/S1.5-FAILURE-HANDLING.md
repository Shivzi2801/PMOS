# S1.5 — Index Fan-Out — Failure Handling

This document specifies how the indexing subsystem behaves under failure. It
covers the four failure classes named by the slice — partial index failures,
duplicate hash collisions, reconciliation failures, retry strategy — plus the
error taxonomy and the cross-cutting invariants that hold under all of them.

---

## 1. Error taxonomy

All errors derive from `IndexingError` (`errors.py`), which carries:

- `category` (`ErrorCategory`) — coarse bucket used as a metrics label.
- `retryable` (`bool`) — whether a retry can plausibly succeed. This is an
  explicit property of each error type, not inferred at the call site (same
  convention as S1.1's transient/permanent distinction).
- ACL-safe `context()` — `tenant_id`, `document_id`, `chunk_id`, category,
  retryable, and (for `PartialIndexError`) the failed target names. **Never
  content, never ACL principals.**

| Error | Category | Retryable | Raised when |
|---|---|---|---|
| `ContractViolation` | contract | no | chunk/document fails required-field or invariant validation |
| `ChunkingError` | chunking | no | chunking policy/config invalid |
| `EmptyDocumentError` | chunking | no | document has no indexable content |
| `HashingError` | hashing | no | hashing precondition violated |
| `HashCollisionError` | hashing | no | same `content_hash` seen for differing content |
| `DedupError` | dedup | no | dedup invariant violated |
| `VectorStoreError` | vector_store | yes (default) | generic store failure |
| `VectorStoreUnavailable` | vector_store | yes | transient connectivity/capacity |
| `VectorStoreRejected` | vector_store | no | payload/schema/partition rejection |
| `PartialIndexError` | fanout | yes | some targets succeeded, others failed |
| `ReconciliationError` | reconciler | yes | reconciliation step failed |

## 2. Partial index failures

**Scenario.** A document's unique chunk set is dispatched to N targets; some
accept, some fail.

**Behavior.**

1. **Isolation.** Each target is upserted independently in `_dispatch`. A
   failure on one target is caught and recorded in a `failures` map; the loop
   continues to the next target. A success on target A is **never** rolled back
   because target B failed.
2. **No double-write.** Upserts are keyed by `chunk_id` (deterministic). A later
   retry that re-sends the same points to a healthy target is an idempotent
   overwrite — it cannot create duplicates. This is what makes "retry only the
   failed targets" safe.
3. **Reporting.** When `failures` is non-empty, `index_document` raises
   `PartialIndexError` carrying `failures = {target_name: IndexingError}`. The
   returned/raised information lets a retry worker re-dispatch to exactly the
   failed targets. `FanOutResult.targets_succeeded` / `targets_failed` record
   the split.
4. **Metrics.** `chunks_indexed` is incremented per *successful* target only, so
   partial success is reflected accurately.

**Test coverage.** `FanOutTests.test_partial_failure_isolated` — a healthy store
plus a `_DeadStore` (always `VectorStoreRejected`); asserts the healthy store
received writes and `PartialIndexError.failures` names the dead target.

## 3. Duplicate hash collisions

There are two distinct cases, and the system treats them differently.

**3a. Ordinary duplicate (expected, benign).** Same `(tenant_id, content_hash)`
*and* same content length ⇒ genuine duplicate. The chunk is classified into
`duplicates`, suppressed from indexing, and counted in `chunks_deduplicated`.
This is the normal dedup path, not a failure.

**3b. Hash collision (anomalous, fatal to the operation).** Same
`(tenant_id, content_hash)` but a **different content length** ⇒
`HashCollisionError` (non-retryable). A real SHA256 collision is computationally
infeasible, so in practice this signals a bug (hash computed over the wrong
field), data corruption, or tampering. It is never retried and must surface to
operators / the reconciliation report. The length fingerprint stored per hash in
`DedupStore` is the cheap guard that makes this detectable without storing
content.

**Cross-tenant note.** Identical content under two different tenants is **not** a
collision and **not** a duplicate — dedup is keyed by `(tenant_id, hash)`, so the
two are indexed separately. Collapsing them would leak content existence across
an ACL boundary.

**Test coverage.** `DedupTests.test_collision_guard` (raises on same hash /
different length), `DedupTests.test_tenant_isolation` (same content, different
tenant ⇒ both unique).

## 4. Reconciliation failures

The reconciler's contract is **resilience over atomicity**: one bad document or
one flaky target must not abort the nightly run.

**Behavior.**

1. **Per-document isolation.** `run()` wraps each document; an `IndexingError`
   from processing one document is caught, increments
   `reconciliation_failures`, and records a `DocumentFinding` with the error —
   then the loop moves on.
2. **Per-target isolation.** Within a document, a `fetch_ids` failure on one
   target is caught, counted, and recorded; other targets are still reconciled.
3. **Repair failures.** If `reindex_chunks` reports per-target failures during
   repair, each is counted into `reconciliation_failures` and recorded as a
   finding; the run continues.
4. **Report.** `ReconciliationReport` aggregates `documents_scanned`,
   `drifted_documents`, `chunks_reindexed`, `orphans_deleted`, and `failures`,
   plus the full findings list. Nothing is thrown out of `run()` for a normal
   data/target failure — failures are data in the report.

**Detection semantics.**

- **Missing** = expected id absent from a target ⇒ repaired by reindex.
- **Stale** = present id whose authoritative `expected_hash` differs from the
  freshly computed expected hash (content changed at that ordinal), or whose id
  the authority no longer recognizes ⇒ repaired by reindex.
- **Orphan** = present id no longer expected ⇒ deleted.

**Repair correctness — the dedup bypass.** Repair calls
`IndexFanOut.reindex_chunks`, which writes the authoritative expected chunks
**directly, skipping the dedup store**. This is deliberate and important: a
missing/stale vector means the *store* lost the write, but the dedup ledger
still lists that content as "seen." The normal `index_document` path would
therefore dedup the chunk away and write nothing, leaving the gap unrepaired.
Bypassing dedup, combined with idempotent upserts, makes repair both effective
and safe (re-writing already-correct points is harmless).

**Test coverage.** `ReconcilerTests` — `test_detect_and_repair_missing`,
`test_detect_orphan_and_delete`, `test_detect_stale`,
`test_reconciliation_failure_counted` (a target whose `fetch_ids` raises;
asserts `failures` and the `reconciliation_failures` metric both increment).

## 5. Retry strategy

**Where.** Retries apply only to vector-store upserts inside
`_upsert_with_retry`. Chunking, hashing, dedup, and contract errors are
deterministic and non-retryable — retrying them is pointless and is not
attempted.

**Policy.** `RetryPolicy(max_attempts, base_delay_s, max_delay_s)` with bounded
exponential backoff: `delay = min(base * 2^(attempt-1), max_delay)`.

**Rules.**

- Only errors with `retryable=True` are retried. `VectorStoreUnavailable`
  (transient) retries; `VectorStoreRejected` (schema/partition/oversize) does
  not — the identical payload would be rejected again.
- Retries are bounded by `max_attempts`. On exhaustion the last error
  propagates to `_dispatch`, which records it as a per-target failure (→ §2).
- Backoff sleep is injected (`sleep` callable) so tests run with zero delay and
  production can supply jittered sleeps.
- Idempotency (chunk_id-keyed upsert) guarantees retries never duplicate data.

**Test coverage.** `FanOutTests.test_retry_then_succeed` — a `_FlakyStore` fails
twice with `VectorStoreUnavailable` then succeeds; asserts exactly three calls
and full indexing.

## 6. Cross-cutting invariants under failure

1. **Fail-closed ACL.** A chunk with an empty `source_acl` is not retrievable;
   `search` hides points whose `acl` does not intersect the caller's principals.
   No failure path can turn a chunk "public."
2. **Tenant isolation.** Every store read/write is tenant-scoped; dedup is
   `(tenant_id, hash)`. No failure or repair path crosses tenants.
3. **No content in telemetry.** Errors, metrics, logs, and reconciliation
   reports use ACL-safe projections only (`safe_descriptor`, `context`). Content
   never leaves the indexing boundary through an error or report.
4. **Idempotent recovery.** Because identity is deterministic and upserts are
   idempotent, every recovery action (retry, partial-target re-dispatch,
   reconciler repair) is safe to run repeatedly.
5. **Progress guarantee.** Chunk planning enforces `overlap < target_size`, so
   chunking always terminates — there is no infinite-loop failure mode.

---

## 7. Failure-handling responsibility map

| Concern | Owner | Mechanism |
|---|---|---|
| Invalid chunk/document | `document_chunk` / `chunker` | `ContractViolation` / `EmptyDocumentError` (no retry) |
| Duplicate content | `dedup` | suppress + count (benign) |
| Hash anomaly | `dedup` + `hashing` | `HashCollisionError` (no retry, surfaced) |
| Transient store outage | `fanout._upsert_with_retry` | bounded exponential backoff |
| Permanent store rejection | `fanout` | `VectorStoreRejected` → per-target failure |
| Some targets down | `fanout._dispatch` | isolate + `PartialIndexError(failures)` |
| Drift over time | `reconciler` | detect missing/stale/orphan + idempotent repair |
| Reconciliation step error | `reconciler.run` | count `reconciliation_failures`, record, continue |
