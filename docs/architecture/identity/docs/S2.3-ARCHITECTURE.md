# S2.3 Рђћ Identity, Access Control & Multi-Tenancy: Architecture

## What this slice is

This is the layer that makes PMOS a **multi-tenant SaaS product** instead of a
single-customer tool. It answers three questions on every request:

1. **Who are you?** (authentication Рђћ sessions and API keys)
2. **Which customer and team do you belong to?** (tenant + workspace resolution)
3. **Are you allowed to do this, on this data?** (RBAC + tenant isolation)

It also records every one of those decisions in an **audit trail** and exposes
**metrics** so the platform is observable and compliant.

A Product Manager can summarise it in one sentence: **this slice lets thousands
of companies safely share one PMOS deployment, each seeing only their own data,
with each user limited to exactly what their role permits Рђћ and a complete record
of who did what.**

## The tenancy hierarchy

```
Tenant (Acme Corp)                     Рєљ the customer / billing & isolation boundary
 РћюРћђРћђ Workspace (Product Team)          Рєљ a team / data domain inside the customer
 Рћѓ    РћюРћђРћђ User (pm@acme.com)           Рєљ a person, holds roles
 Рћѓ    РћћРћђРћђ User (analyst@acme.com)
 РћюРћђРћђ Workspace (Support Team)
 РћћРћђРћђ API Key (ci-pipeline)             Рєљ a machine credential, tenant-scoped
```

- A **Tenant** is the hard isolation boundary. No tenant can ever see another
  tenant's data. This is the single most important security property of the
  platform.
- A **Workspace** is a softer boundary *inside* a tenant for separating teams or
  data domains.
- A **User** belongs to one tenant, optionally one default workspace, and holds
  one or more **roles**.
- An **API Key** is a tenant-scoped machine credential with its own roles, for
  integrations and automation.

## File-by-file reference

For each file: **why it exists**, **responsibilities**, **inputs**, **outputs**,
and **interaction with previous slices**.

### `__init__.py`
- **Why:** One import surface for the whole slice.
- **Responsibilities:** Re-export `IdentityService` and all public types.
- **Inputs/Outputs:** none (namespace).
- **Interaction:** what `backend/api` and `backend/orchestration` import.

### `errors.py`
- **Why:** A precise, catchable exception per failure mode so the API maps each
  to the right HTTP status and the audit log records the right reason.
- **Responsibilities:** Define the `IdentityError` family, each carrying an HTTP
  `code` and a `category` (not_found / unauthenticated / unauthorized /
  isolation / invalid / conflict).
- **Inputs:** a message + optional detail. **Outputs:** `to_dict()` for API
  responses.
- **Interaction:** the API slice catches these to shape responses; orchestration
  treats authorization errors as terminal (non-retryable).

### `tenant.py`
- **Why:** The top-level isolation boundary needs a typed model.
- **Responsibilities:** Hold `tenant_id`, `tenant_name`, `status`, `created_at`,
  `settings`; expose `is_active`.
- **Inputs:** a tenant name + optional settings. **Outputs:** a `Tenant`.
- **Interaction:** the `tenant_id` is the partition key every downstream slice
  (indexing, retrieval, orchestration) will store and filter by.

### `workspace.py`
- **Why:** Customers need to separate teams/domains under one account.
- **Responsibilities:** Model the workspace, always carrying its owning
  `tenant_id`.
- **Inputs:** tenant_id + name. **Outputs:** a `Workspace`.
- **Interaction:** the secondary partition key (`workspace_id`) downstream slices
  can scope data by.

### `user.py`
- **Why:** Represent a human principal and their authorization metadata.
- **Responsibilities:** Hold identity fields + assigned role names. Stores **no
  credentials** (auth is delegated).
- **Inputs:** tenant_id, email, optional workspace/roles. **Outputs:** a `User`.
- **Interaction:** roles drive every permission check the guard performs for this
  user's requests.

### `permission.py`
- **Why:** Permissions are the extensibility surface Рђћ new features add new
  permissions without changing the engine.
- **Responsibilities:** Define the seed permission set and a registry so other
  slices can register their own; support the `*` wildcard.
- **Inputs:** permission names. **Outputs:** the known-permission set.
- **Interaction:** the connectors/ingestion/etc. slices can register feature
  permissions (e.g. `manage_connectors`) here.

### `role.py`
- **Why:** Bundle permissions into assignable roles with inheritance.
- **Responsibilities:** Define the five default roles (ADMIN, MANAGER, ANALYST,
  VIEWER, SYSTEM) and their inheritance chain; allow custom per-tenant roles.
- **Inputs:** none for defaults. **Outputs:** nameРєњ`Role` map.
- **Interaction:** consumed by the RBAC engine.

### `rbac_engine.py`
- **Why:** A single, pure authority for "does this role set grant this
  permission?"
- **Responsibilities:** Flatten role inheritance (cycle-safe), honour wildcards,
  answer `has_permission/has_any/has_all`. Supports a pluggable role provider for
  tenant-custom roles.
- **Inputs:** role names + a permission (+ tenant_id). **Outputs:** boolean.
- **Interaction:** called by the tenant guard on every authorization check.

### `tenant_context.py`
- **Why:** The "who and where" must travel with every operation without being
  passed explicitly everywhere.
- **Responsibilities:** Immutable `TenantContext` (tenant, workspace, user/key,
  roles, auth method) stored in a `contextvars` variable that propagates through
  the call stack and async tasks; `use()` context-manager for leak-free binding.
- **Inputs:** resolved identity. **Outputs:** the bound context, readable via
  `current()`.
- **Interaction:** **this is the object orchestration binds around a workflow run
  so every step is tenant-aware.**

### `tenant_resolver.py`
- **Why:** Turn a raw credential into a verified context Рђћ the front door.
- **Responsibilities:** Resolve from API key, session, or system principal;
  verify tenant is active and workspace belongs to the tenant; record resolution
  latency. **Never trusts a client-supplied tenant_id** Рђћ the tenant is derived
  from the credential.
- **Inputs:** an API key / session token / system principal. **Outputs:** a
  `TenantContext` or a precise error.
- **Interaction:** the API slice calls this at request entry; orchestration calls
  `resolve_system` for background jobs.

### `tenant_guard.py`
- **Why:** One enforcement point for both isolation and authorization.
- **Responsibilities:** `require_permission` (RBAC) and `require_same_tenant /
  require_same_workspace / guard_resource` (isolation); audit every allow/deny.
- **Inputs:** a context + a permission and/or a resource. **Outputs:** nothing on
  allow; raises on deny.
- **Interaction:** every protected operation in every downstream slice should
  pass through here.

### `api_key.py`
- **Why:** Machine credentials need their own model with safe secret handling.
- **Responsibilities:** Generate a token whose secret is shown once; store only a
  salted hash; verify in constant time; track status/expiry/rotation.
- **Inputs:** tenant/workspace/roles. **Outputs:** the record + a one-time raw
  secret.
- **Interaction:** used by orchestration/integrations for non-interactive auth.

### `api_key_manager.py`
- **Why:** Own the full key lifecycle.
- **Responsibilities:** create, validate (hot path, O(1) lookup by embedded id),
  rotate, revoke, list; emit usage metrics + audit.
- **Inputs:** raw secret (validate) or key_id (rotate/revoke). **Outputs:** the
  resolved `APIKey` or an `InvalidAPIKeyError`.
- **Interaction:** invoked by the resolver during API-key resolution.

### `auth_provider.py`
- **Why:** Authentication must be pluggable for enterprise (SSO/SAML/OIDC) without
  building those now.
- **Responsibilities:** Define the `AuthProvider` contract and an
  `AuthenticatedPrincipal`; ship **interface-only** stubs for JWT, OAuth2, SSO,
  SAML, OIDC, plus a registry.
- **Inputs:** an opaque credential. **Outputs:** a principal (when implemented).
- **Interaction:** the identity service routes provider logins into sessions.

### `session_manager.py`
- **Why:** Interactive users need short-lived, revocable sessions.
- **Responsibilities:** create, validate (absolute + idle expiry), revoke (single
  + all-for-user); count active sessions.
- **Inputs:** tenant/user/roles (create) or token (validate). **Outputs:** a
  `Session` or an expiry/not-found error.
- **Interaction:** used by the resolver during session resolution.

### `access_audit.py`
- **Why:** A SaaS platform must record who did what for compliance and forensics.
- **Responsibilities:** Append-only, immutable, tenant-tagged records of user
  actions, permission checks, tenant access, API-key usage, and auth events;
  queryable by tenant/action/outcome.
- **Inputs:** event fields or a `TenantContext`. **Outputs:** stored records.
- **Interaction:** every component writes here; the API can expose it as a
  customer-facing audit log.

### `metrics.py`
- **Why:** Operate and sell the platform on real numbers.
- **Responsibilities:** Track active users/tenants/workspaces, API-key usage,
  permission denials, auth attempts, and tenant-resolution latency.
- **Inputs:** signals from managers/guard/resolver. **Outputs:** `snapshot()`.
- **Interaction:** complements the orchestration metrics from S2.2 for a full
  platform dashboard.

### `identity_service.py`
- **Why:** A single facade so other slices don't wire ten components themselves.
- **Responsibilities:** own the stores; wire all components; expose provisioning,
  login/logout, resolution, authorization, and `run_in_context` for
  orchestration.
- **Inputs:** high-level calls (`create_tenant`, `login`, `resolve`, ...).
  **Outputs:** models, contexts, booleans.
- **Interaction:** **the object `backend/api` and `backend/orchestration` hold a
  reference to.**

### `test_identity.py`
- **Why:** Prove tenancy, RBAC, keys, sessions, and isolation behave Рђћ 31 tests.

## Tenant isolation strategy

Isolation is enforced in depth, not at a single point:

1. **Credential-derived tenancy.** The resolver derives `tenant_id` from the
   credential itself (key or session), never from client input. A caller cannot
   ask to "be" another tenant.
2. **Active-status gating.** Suspended/archived tenants and workspaces fail
   resolution, so a disabled customer is locked out everywhere at once.
3. **Workspace-within-tenant validation.** A workspace only resolves if it
   belongs to the credential's tenant.
4. **The guard.** Any resource carrying a `tenant_id`/`workspace_id` is checked
   against the caller's context before use; a mismatch is a hard
   `CrossTenantAccessError`, audited as a denial.
5. **Context propagation.** Because the `TenantContext` rides along in
   `contextvars`, downstream slices can scope every query without re-deriving
   tenancy Рђћ and the contract is "every data access is filtered by the context's
   tenant_id."

**The rule downstream slices follow:** every stored record carries `tenant_id`
(and `workspace_id` where relevant), and every read/write filters by the bound
context's tenant.

## RBAC strategy

- **Users hold roles, roles hold permissions.** Users never carry raw
  permissions, so access is changed in one place and is fully auditable.
- **Inheritance** keeps roles DRY: VIEWER Ріѓ ANALYST Ріѓ MANAGER; ADMIN and SYSTEM
  hold the wildcard.
- **Wildcard** lets super-roles grant future permissions automatically.
- **Extensible permissions:** new features register new permission names; the
  engine needs no changes.
- **Pure engine, enforced at the guard:** the engine only computes
  grant/deny; the guard combines that with isolation and audit. This separation
  keeps each piece simple and independently testable.

## API key lifecycle

```
create РћђРћђРќХ ACTIVE РћђРћђРћгРћђРћђРќХ rotate РћђРћђРќХ (old: ROTATED, new: ACTIVE)
                    РћюРћђРћђРќХ revoke РћђРћђРќХ REVOKED
                    РћћРћђРћђРќХ (time)  РћђРћђРќХ EXPIRED
```

The raw secret is returned **once** at creation/rotation; only a salted hash is
stored. Validation parses the embedded key id for an O(1) lookup, then verifies
the hash in constant time, and rejects anything not `ACTIVE`-and-unexpired with a
generic error to prevent key enumeration. Every operation is audited and counted.

## Session lifecycle

```
login РћђРћђРќХ create session РћђРћђРќХ ACTIVE
   validate: ok while (now < expires_at) AND (idle < idle_timeout)
   ACTIVE РћђРћђРќХ revoke (logout / logout-everywhere) РћђРћђРќХ gone
   ACTIVE РћђРћђРќХ absolute or idle expiry РћђРћђРќХ purged, SessionExpiredError
```

Sessions are server-side and revocable, supporting "log out everywhere" (e.g.
after a role change or suspected compromise) via `revoke_all_for_user`.

## Audit model

Every security-relevant event becomes an immutable, timestamped, tenant-tagged
record: provisioning, auth success/failure, session create/revoke, permission
allow/deny, tenant/workspace access attempts, and API-key create/use/rotate/
revoke. Records are queryable per tenant, so the audit data is itself
tenant-isolated and can be surfaced as a customer-facing compliance feature.

## Integration with previous slices

- **`backend/api`** calls `IdentityService.resolve(...)` at request entry to get a
  `TenantContext`, then uses `guard.require_permission(...)` before dispatching to
  any slice. Identity errors map cleanly to HTTP statuses via `to_dict()`.
- **`backend/orchestration` (S2.2)** runs each workflow inside
  `IdentityService.run_in_context(ctx, ...)` (or `tenant_scope`), so every step Рђћ
  retrieval, indexing, generation Рђћ executes with a bound tenant context. The
  orchestrator can use a SYSTEM principal (`resolve_system`) for scheduled jobs.
  Permissions like `run_query` / `run_ingestion` / `run_reindex` gate who may
  trigger each workflow.
- **All data slices** (connectors, ingestion, extraction, resolution, indexing,
  retrieval, context, generation, grounding) become tenant-aware by reading the
  bound context's `tenant_id` / `workspace_id` and filtering their data by it.

## SaaS scalability considerations

- **Stateless resolution path.** Sessions and keys are looked up by id; the
  request path holds no per-request global state beyond the `contextvars` context,
  so the layer scales horizontally behind a load balancer.
- **Storage-swappable.** Every store implements a tiny `.get(id)` contract and the
  managers hide their backing dicts, so moving to Postgres/Redis is a drop-in
  behind the same interfaces Рђћ no caller changes.
- **O(1) hot paths.** API-key validation and session validation are constant-time
  lookups; tenant-resolution latency is measured so regressions are caught.
- **Isolation by partition key.** Because everything keys on `tenant_id`, the data
  tier can shard or row-level-secure per tenant for noisy-neighbour isolation and
  data-residency requirements (tenant `settings` can carry a region).
- **Pluggable enterprise auth.** SSO/SAML/OIDC slot in behind `AuthProvider`
  without touching authorization, which is what enterprise sales will require.
- **Least-privilege machine access.** Per-key roles and rotation/revocation
  support automated credential hygiene at fleet scale.
