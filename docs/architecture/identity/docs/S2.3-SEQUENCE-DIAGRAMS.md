# S2.3 — Identity & Access Control: Sequence Diagrams

These Mermaid diagrams show how identity, tenancy, and access control behave over
time across the key flows and failure modes. Each has a plain-language
walkthrough so a Product Manager can narrate it to engineers, security teams,
enterprise customers, investors, or solution architects.

Recurring participants:

- **Client** — a browser/app (sessions) or a service (API keys)
- **API** — the `backend/api` slice
- **IdentityService** — the facade for this slice
- **Resolver / Guard / RBAC / SessionMgr / APIKeyMgr / Audit / Metrics** — internals
- **Orchestrator** — the `backend/orchestration` slice (S2.2)

---

## 1. Interactive login → authenticated request

```mermaid
sequenceDiagram
    participant Client
    participant API
    participant IdentityService
    participant SessionMgr
    participant Resolver
    participant Guard
    participant RBAC
    participant Audit

    Client->>API: login (credential)
    API->>IdentityService: login(user_id)
    IdentityService->>SessionMgr: create(tenant, user, roles)
    SessionMgr-->>IdentityService: Session(token)
    IdentityService->>Audit: auth_success
    IdentityService-->>API: session token
    API-->>Client: set session token

    Note over Client,API: later request with the token
    Client->>API: GET /documents (token)
    API->>IdentityService: resolve(session_token)
    IdentityService->>Resolver: resolve_from_session(token)
    Resolver->>SessionMgr: validate(token)
    SessionMgr-->>Resolver: Session (live)
    Resolver-->>IdentityService: TenantContext
    API->>Guard: require_permission("view_documents", ctx)
    Guard->>RBAC: has_permission(roles, perm, tenant)
    RBAC-->>Guard: true
    Guard->>Audit: permission_check allowed=true
    Guard-->>API: ok
    API-->>Client: documents (tenant-scoped)
```

**Walkthrough:** The user authenticates once and receives a session token. On
each later request the API resolves that token into a tenant context, the guard
confirms the user's roles grant the needed permission, and the audit log records
the check. The documents returned are scoped to the user's tenant.

## 2. Machine request via API key

```mermaid
sequenceDiagram
    participant Service
    participant API
    participant IdentityService
    participant Resolver
    participant APIKeyMgr
    participant Metrics

    Service->>API: POST /ingest (Authorization: pmos_sk_...)
    API->>IdentityService: resolve(api_key=raw)
    IdentityService->>Resolver: resolve_from_api_key(raw)
    Resolver->>APIKeyMgr: validate(raw)
    APIKeyMgr->>APIKeyMgr: parse key_id, lookup, verify hash (constant-time)
    APIKeyMgr->>Metrics: api_key_usage++
    APIKeyMgr-->>Resolver: APIKey (active)
    Resolver-->>IdentityService: TenantContext(auth_method=api_key)
    IdentityService-->>API: context
    API-->>Service: proceed (scoped to key's tenant)
```

**Walkthrough:** A service presents its API key. The manager locates the record
by the id embedded in the token (constant-time hash check), counts the usage, and
returns a tenant context. The key's own roles determine what it may do — a
least-privilege ingestion key can ingest but not manage users.

## 3. RBAC evaluation with inheritance

```mermaid
sequenceDiagram
    participant Guard
    participant RBAC
    participant Roles

    Guard->>RBAC: has_permission(["MANAGER"], "manage_users", tenant)
    RBAC->>Roles: resolve MANAGER
    Roles-->>RBAC: perms{manage_*}, inherits[ANALYST]
    RBAC->>Roles: resolve ANALYST
    Roles-->>RBAC: perms{ingest,...}, inherits[VIEWER]
    RBAC->>Roles: resolve VIEWER
    Roles-->>RBAC: perms{view_documents,...}
    Note over RBAC: flatten union of all inherited permissions
    RBAC-->>Guard: "manage_users" present -> true
```

**Walkthrough:** A permission check walks the role's inheritance chain, unions all
permissions, and answers yes/no. MANAGER inherits ANALYST inherits VIEWER, so a
manager automatically has every lower capability without it being re-listed.

## 4. Tenant isolation block (cross-tenant access denied)

```mermaid
sequenceDiagram
    participant Client
    participant API
    participant Guard
    participant Audit

    Note over Client: principal belongs to Tenant A (even as ADMIN)
    Client->>API: GET /doc/123 (token)
    API->>API: load doc 123 -> owned by Tenant B
    API->>Guard: guard_resource(doc, ctx_A)
    Guard->>Guard: resource.tenant_id (B) != ctx.tenant_id (A)
    Guard->>Audit: tenant_access allowed=false (mismatch)
    Guard-->>API: raise CrossTenantAccessError (403)
    API-->>Client: 403 (no data leaked)
```

**Walkthrough:** A valid, even fully-privileged, user from Tenant A requests a
document owned by Tenant B. The guard compares the resource's tenant to the
caller's and refuses — wildcard/ADMIN does not cross tenants. The denial is
audited and no data leaks.

## 5. API key lifecycle: rotation

```mermaid
sequenceDiagram
    participant Admin
    participant IdentityService
    participant APIKeyMgr
    participant Audit

    Admin->>IdentityService: rotate(key_id)
    IdentityService->>APIKeyMgr: rotate(key_id)
    APIKeyMgr->>APIKeyMgr: create replacement (same tenant/roles)
    APIKeyMgr->>APIKeyMgr: mark old key ROTATED (rotated_to=new)
    APIKeyMgr->>Audit: api_key_rotated
    APIKeyMgr-->>IdentityService: (new record, new raw secret shown once)
    IdentityService-->>Admin: new secret

    Note over APIKeyMgr: subsequent validate(old) -> InvalidAPIKeyError
    Note over APIKeyMgr: validate(new) -> ok
```

**Walkthrough:** Rotation issues a new key inheriting the old one's tenant and
roles, marks the old one as superseded (kept for audit, no longer usable), and
returns the new secret once. Integrations can be migrated without downtime, then
the old key stops working.

## 6. Session expiry

```mermaid
sequenceDiagram
    participant Client
    participant API
    participant SessionMgr

    Client->>API: request (token)
    API->>SessionMgr: validate(token)
    alt within TTL and not idle
        SessionMgr-->>API: Session (refresh last_seen)
        API-->>Client: success
    else expired or idle-timed-out
        SessionMgr->>SessionMgr: purge session
        SessionMgr-->>API: raise SessionExpiredError (401)
        API-->>Client: 401 -> must re-authenticate
    end
```

**Walkthrough:** Each request re-validates the session against both an absolute
lifetime and an idle timeout. An expired or idle session is purged and the client
is told to log in again — stale tokens can't be reused.

## 7. Orchestration workflow inside a tenant context

```mermaid
sequenceDiagram
    participant API
    participant IdentityService
    participant Guard
    participant Orchestrator
    participant Slices
    participant Audit

    API->>IdentityService: resolve(credential) -> ctx
    API->>Guard: require_permission("run_ingestion", ctx)
    Guard-->>API: ok
    API->>IdentityService: run_in_context(ctx, run_workflow)
    IdentityService->>Audit: tenant_scope_enter
    activate IdentityService
    Note over IdentityService: TenantContext bound for the whole run
    IdentityService->>Orchestrator: run("ingestion", payload)
    Orchestrator->>Slices: connector -> ingestion -> ... -> indexing
    Note over Slices: each reads ctx.tenant_id / workspace_id to scope data
    Slices-->>Orchestrator: result
    Orchestrator-->>IdentityService: WorkflowResult
    deactivate IdentityService
    IdentityService-->>API: result (tenant-scoped, audited)
```

**Walkthrough:** Before running a workflow the API checks the caller may run it,
then runs it *inside* the tenant context. Every orchestration step (from S2.2)
executes with that context bound, so each slice scopes its data to the right
tenant and workspace. Entry into the scope is audited, tying the whole job to a
tenant and principal.

## Reading these as a Product Manager

Each diagram answers a stakeholder's core question:

- **Engineers:** the exact call order and where the context is created, bound, and
  checked.
- **Security teams:** that authentication, authorization, and isolation are
  distinct, enforced, fail-closed, and audited — and that wildcard never crosses
  tenants.
- **Enterprise customers:** that their data is isolated by design, their admins
  can rotate keys and force logout, and every access is logged.
- **Investors / architects:** that the platform is genuinely multi-tenant SaaS —
  one deployment safely serving many customers — with pluggable enterprise auth
  ready to slot in.
