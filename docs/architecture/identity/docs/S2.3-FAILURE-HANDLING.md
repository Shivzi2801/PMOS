# S2.3 — Identity & Access Control: Failure Handling

Security layers are judged by how they behave when something is wrong: a bad
credential, an expired session, a request reaching across tenant boundaries. This
document explains exactly how every failure mode is detected, classified, and
surfaced — in language a Product Manager can use with engineers, security teams,
enterprise customers, and auditors.

---

## Principle: fail closed, fail precise, fail audited

Three rules govern every failure path:

1. **Fail closed.** When anything is uncertain — unknown tenant, missing
   workspace, malformed key — access is denied, never granted by default.
2. **Fail precise (internally).** Each failure raises a specific, catchable
   exception with an HTTP-style code and a category, so the API returns the right
   status and operators see the real reason.
3. **Fail audited.** Every denial is written to the access audit, so a refused
   request is as traceable as a successful one.

One deliberate exception to "precise": **authentication failures are
deliberately vague to the client.** An invalid API key or bad session yields a
generic "invalid or inactive" message so attackers cannot distinguish "unknown
key" from "revoked key" from "wrong secret" — that distinction is recorded in the
audit log, not handed to the caller.

---

## Each required failure mode

### 1. Unknown tenant → `UnknownTenantError` (404, not_found)
The resolver looks up the tenant derived from the credential. If it doesn't
exist (or was deleted), resolution stops immediately. **Recovery:** the request
is rejected; nothing downstream runs. Audited as a failed resolution.

### 2. Missing workspace → `WorkspaceNotFoundError` (404, not_found)
A workspace must both exist **and** belong to the credential's tenant. A
workspace from another tenant is treated as "not found" (not "forbidden") so the
existence of other tenants' workspaces is never revealed. **Recovery:** request
rejected; isolation preserved.

### 3. Invalid user → `InvalidUserError` (404, not_found)
Login or user-scoped operations on a non-existent user fail. **Recovery:** no
session is issued; the attempt is counted in `authentication_attempts` and
audited.

### 4. Expired session → `SessionExpiredError` (401, unauthenticated)
Validation checks both absolute TTL and idle timeout. Either being exceeded
purges the session and raises. **Recovery:** the client must re-authenticate; the
stale session is removed so it can't be reused.

### 5. Invalid API key → `InvalidAPIKeyError` (401, unauthenticated)
Covers malformed tokens, unknown ids, revoked keys, rotated (superseded) keys,
expired keys, and wrong secrets — all collapsed into one generic error to prevent
enumeration. Verification is constant-time to avoid timing side-channels.
**Recovery:** rejected and audited; the key id (when parseable) is recorded for
investigation.

### 6. Permission denied → `PermissionDeniedError` (403, unauthorized)
The identity is valid but the roles don't grant the required permission. The
guard records the check (allowed=false) and increments `permission_denials`.
**Recovery:** the specific action is refused; the session/key remains valid for
actions it *is* permitted to perform.

### 7. Tenant mismatch → `TenantMismatchError` / `CrossTenantAccessError` (403, isolation)
A valid principal from tenant A touches a resource owned by tenant B. This is the
most serious category: it's flagged distinctly (category `isolation`), audited as
a denial, and blocked unconditionally — **even for ADMIN/SYSTEM wildcard roles**,
because wildcard grants permissions *within* a tenant, never across tenants.

### 8. Authorization failure → `AuthorizationError` (403, unauthorized)
The base class for the authorization family (permission denied, inactive entity).
Inactive tenants/workspaces/users surface as `InactiveEntityError` so a suspended
customer is locked out cleanly with a clear reason in the audit log.

---

## Failure handling beyond the required list

- **Missing tenant context** (`MissingTenantContextError`): an operation that
  requires a bound context but finds none fails loudly rather than running
  unscoped — preventing accidental cross-tenant data exposure from a coding bug.
- **Duplicate entity** (`DuplicateEntityError`, 409): e.g. a repeated user email
  within a tenant, caught at provisioning.
- **Unimplemented auth provider** (`NotImplementedError`): the SSO/SAML/OIDC/JWT
  stubs raise clearly so an environment misconfiguration (pointing at an
  unbuilt provider) is obvious, not silent.

---

## How failures interact with orchestration (S2.2)

Identity errors are **terminal** from the orchestrator's point of view — retrying
an authorization failure or a cross-tenant access would be pointless and
dangerous. When a workflow runs inside `run_in_context`, a `PermissionDeniedError`
or `CrossTenantAccessError` aborts the workflow immediately (no retries), the
orchestrator marks the job FAILED, and the failure is recorded in both the
orchestration audit (job trace) and the identity audit (the denial). Transient
infrastructure issues are the orchestrator's concern; *security* decisions made
here are always final.

---

## Recovery and safety guarantees

- **No partial authorization.** A request is either fully resolved and authorized
  or it does not proceed. There is no "best effort" middle ground for security.
- **Revocation is immediate.** Revoking a key or session, or suspending a tenant,
  takes effect on the very next request because validation re-checks live state.
- **Isolation cannot be escalated.** No role, including SYSTEM, can cross a tenant
  boundary. Privilege escalation within a tenant is bounded by the role model.
- **Every denial is explainable.** The audit log answers "why was this blocked?"
  with the action, the principal, the tenant, and the reason — within seconds.
- **Generic to attackers, specific to operators.** Clients get safe, vague
  authentication errors; the audit log gets the precise truth.

---

## What this means for the business

- **Enterprise security review-ready.** Fail-closed behaviour, full audit trail,
  constant-time secret checks, and hard tenant isolation are exactly what SOC 2 /
  ISO 27001 / customer security questionnaires ask for.
- **Trust as a feature.** "A user in one company can never, under any role, reach
  another company's data" is a one-line promise backed by enforced, tested,
  audited code.
- **Operational confidence.** Suspending a customer, rotating a leaked key, or
  forcing a global logout are single, immediate, audited actions.
