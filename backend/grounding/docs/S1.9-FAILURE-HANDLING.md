# S1.9 — Grounding & Answer Verification: Failure Handling

> **Audience:** Product Managers, engineers, architects, enterprise customers.
> **Purpose:** explain every way the grounding layer can fail and exactly how it degrades — so that a PM can promise customers, truthfully, that *"the system never crashes on the user and never silently ships an unverified answer."*

---

## 1. The guiding principle: **fail safe, never fail open**

The grounding layer is a safety component. When something goes wrong, it must err toward *distrust*, not toward shipping an unverified answer. Concretely:

- If the layer cannot prove an answer is grounded, it returns **REJECTED** with `HIGH` risk and `0.0` confidence — never a falsely reassuring "Verified."
- The layer **always returns a `GroundingResult`.** It never raises an exception into the calling generation/API layer. Every failure becomes a structured result plus a human-readable `warning`.
- The user's answer is **never lost** because of an internal bookkeeping failure (e.g. the audit log being unavailable).

This combination — *always return something, but make that something safe* — is what lets the rest of the product treat grounding as a dependable gate.

---

## 2. Two tiers of degradation

| Tier | What happens | User-visible effect |
|---|---|---|
| **Hard degrade** | The answer cannot be verified at all (no evidence, malformed answer, timeout). | A `REJECTED` result; the UI withholds or heavily caveats the answer. |
| **Soft degrade** | Verification succeeds but a *secondary* function failed (audit write, a single sub-signal). | The result is returned normally; a `warning` is attached and logged for ops. |

---

## 3. Failure scenarios, one by one

Each scenario lists: **trigger → detection → handling → user impact → why this choice.**

### 3.1 Missing citations
- **Trigger:** the generation layer produced an answer with no citation field at all.
- **Detection:** `MissingCitationsError` is available for callers that pre-validate; within the pipeline, absent citations simply mean claims carry no citation hints.
- **Handling:** the pipeline proceeds. Claims with no generator citation can still be *supported* by evidence (the matcher checks regardless), but each is recorded as a `SUPPORTED_BUT_UNCITED` finding for citation-hygiene reporting.
- **User impact:** none directly; coverage and risk reflect the true support.
- **Why:** the generator forgetting to cite should not, by itself, reject a genuinely grounded answer — but it should be visible in audit.

### 3.2 Empty evidence
- **Trigger:** the context slice returned no passages.
- **Detection:** `EvidenceMatcher.match` raises `EmptyEvidenceError`.
- **Handling:** the pipeline catches it and returns a **hard-degraded** result: `REJECTED`, `HIGH` risk, `0.0` confidence, with a warning `"No evidence available..."`. The degraded result is still **audited**.
- **User impact:** the answer is blocked or shown with a strong "could not be verified" caveat.
- **Why:** with nothing to ground against, no claim can be trusted. Returning REJECTED is the only safe choice.

### 3.3 Malformed claims
- **Trigger:** a claim is structurally invalid (e.g. corrupt span data).
- **Detection:** `MalformedClaimError` from the extractor path.
- **Handling:** treated as a grounding error; the pipeline hard-degrades to `REJECTED` with a warning rather than processing a corrupt claim.
- **User impact:** answer blocked/caveated.
- **Why:** a corrupt claim cannot be reliably matched; trusting it would be unsafe.

### 3.4 Malformed / empty answer
- **Trigger:** `answer_text` is `None`, blank, or whitespace.
- **Detection:** `ClaimExtractor.extract` raises `MalformedAnswerError`.
- **Handling:** pipeline catches it → hard-degraded `REJECTED` with a `"Malformed answer..."` warning. Still audited.
- **User impact:** nothing is shown as verified; the upstream answer is treated as unusable.
- **Why:** you cannot verify content that isn't there.

### 3.5 Verification timeout
- **Trigger:** the full verification exceeds the configured wall-clock budget (`timeout_seconds`, default 5s).
- **Detection:** the pipeline checks the deadline *between stages*; on overrun it raises `VerificationTimeoutError`.
- **Handling:** caught → hard-degraded `REJECTED` with a `"Verification timed out..."` warning. Still audited.
- **User impact:** rather than hanging the user, the system blocks/caveats the answer quickly.
- **Why:** a slow verification must never become a hung request. A predictable, bounded latency is itself an enterprise SLA requirement. (Note: the deadline is checked at stage boundaries, so a single pathological stage is still bounded by the surrounding checks.)

### 3.6 Confidence computation failure
- **Trigger:** the scorer fails — e.g. mis-configured weights that don't sum to 1.0, or an unexpected math error.
- **Detection:** `ConfidenceComputationError`.
- **Handling:** **fail safe.** The pipeline catches it *inside* the flow, sets `confidence_score = 0.0`, escalates `hallucination_risk` to `HIGH`, and adds a warning. Because risk is now HIGH, the verifier returns `REJECTED`.
- **User impact:** the answer is rejected rather than shown with a missing/garbage confidence.
- **Why:** a confidence number we cannot compute must default to *no confidence*, not to a guess.

### 3.7 Audit write failure
- **Trigger:** the audit sink cannot persist (disk full, store unavailable).
- **Detection:** `AuditTrail.persist` raises `AuditWriteError`.
- **Handling:** **soft degrade.** The pipeline catches it, **keeps the verification result intact**, and attaches an `"Audit write failed..."` warning. Ops alerting fires on the error code.
- **User impact:** the user still gets their (correctly verified) answer.
- **Why:** losing a user's answer because a log was momentarily unavailable would be a worse failure than a temporarily missing audit line. The error code makes the gap loud and recoverable.
- **Operational note:** persistent audit failures are a compliance incident and should page on-call; the metric/warning stream is the trigger.

### 3.8 Evidence-matching failure (unexpected)
- **Trigger:** an unforeseen error during scoring.
- **Detection:** `EvidenceMatchingError`.
- **Handling:** caught as a generic `GroundingError` → hard-degraded `REJECTED`.
- **Why:** if we cannot reliably establish support, we do not pretend the answer is grounded.

### 3.9 Claim-extraction failure (unexpected)
- **Trigger:** an unforeseen error during segmentation.
- **Detection:** `ClaimExtractionError`.
- **Handling:** caught as a generic `GroundingError` → hard-degraded `REJECTED`.
- **Why:** no reliable claims means no reliable verification.

## 4. The catch-all guarantee

The pipeline's `run()` wraps the core flow in a layered `try/except`:

```
try:
    result = _run_inner(...)          # the happy path
except MalformedAnswerError:          # → REJECTED + warning
except EmptyEvidenceError:            # → REJECTED + warning
except VerificationTimeoutError:      # → REJECTED + warning
except GroundingError:                # any other typed failure → REJECTED + warning
# metrics are recorded on EVERY path, success or degraded
```

Two invariants fall out of this structure and are covered by tests:

1. **A `GroundingResult` is always produced** — no exception escapes `run()`.
2. **Metrics are always recorded** — even degraded/rejected runs increment `grounding_requests` and the appropriate status counter, so dashboards reflect reality.

---

## 5. What each failure looks like to the consumer

The API/UI layer only needs to read three fields to handle every failure uniformly:

- `verification_status` — `REJECTED` means "do not present as fact."
- `hallucination_risk` — `HIGH` means "strong caution / hide."
- `warnings` — a list of human-readable strings explaining what happened, suitable for ops logs and (filtered) for support tooling.

This means the consumer needs **no special-case code per failure type** — the grounding layer normalises every internal failure into the same safe shape.

---

## 6. Testing the failure paths

The suite in `test_grounding.py` exercises each degradation explicitly:

- `test_pipeline_empty_evidence_degrades_gracefully` — empty evidence → REJECTED + warning, still audited.
- `test_pipeline_malformed_answer_degrades_gracefully` — blank answer → REJECTED + warning.
- `test_pipeline_timeout_degrades_gracefully` — zero timeout → REJECTED + warning.
- `test_pipeline_audit_failure_does_not_lose_answer` — failing sink → answer preserved, warning attached.
- `test_confidence_weights_must_sum_to_one` — mis-configured scorer raises the right error.
- `test_pipeline_hallucinated_answer_rejected` — fabrication → HIGH risk → REJECTED.

Together these prove the two invariants above and the fail-safe stance.
