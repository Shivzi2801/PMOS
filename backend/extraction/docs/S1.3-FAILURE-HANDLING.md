# S1.3 ŌĆö F-10 Extraction Engine ŌĆö Failure Handling

This document enumerates every failure mode the extraction engine can
encounter, how it is detected, how it is handled, and how it surfaces in
metrics. The governing principle: **a fault in one optional stage must never
discard a document wholesale**, while **structurally invalid input must fail
fast and loud**.

---

## 1. Failure taxonomy

| # | Failure | Classification | Pipeline behavior | Metric |
|---|---|---|---|---|
| 1 | `document is None` | Terminal / non-retryable | Raise `MalformedDocumentError` | `extraction_failures_total{stage=validate}` |
| 2 | Missing required field (`text`, `tenantId`, ŌĆ”) | Terminal / non-retryable | Raise `MalformedDocumentError` | `extraction_failures_total{stage=validate}` |
| 3 | Wrong field type (e.g. `text` not a string) | Terminal / non-retryable | Raise `MalformedDocumentError` | `extraction_failures_total{stage=validate}` |
| 4 | Empty / whitespace-only `text` | **Not** a failure | Return empty `ExtractionResult` | ŌĆö (documents_total incremented) |
| 5 | Stage extractor raises unexpectedly | Recoverable / isolated | Skip stage (treat as empty), continue cascade | `extraction_failures_total{stage=<name>}` |
| 6 | Extractor emits out-of-band confidence | Self-healing | Clamp into method band | ŌĆö |
| 7 | Invalid `raw_signal` passed to scorer | Programming error | Raise `ConfidenceScoringError` | ŌĆö |
| 8 | Invalid `FactAtom` (empty triple / bad confidence) | Programming error | Raise `ValueError` at construction | ŌĆö |

---

## 2. Detailed handling

### 2.1 Malformed document (cases 1ŌĆō3)
Detected in `validate_canonical_document` before any extraction work. The
pipeline increments `extraction_failures_total{stage=validate}`, logs a
warning, and **re-raises** `MalformedDocumentError`. This is terminal: the
document is not retryable as-is. Callers route it to a dead-letter path.

Rationale: malformed input indicates an upstream contract violation (Slice
1.2). Silently swallowing it would hide ingestion bugs.

### 2.2 Empty document (case 4)
An empty or whitespace-only body is a **valid** document that simply yields no
atoms. The pipeline still:
- increments `extraction_documents_total`
- records `extraction_latency_ms`
- increments `extraction_atoms_total` by `0`
- returns an `ExtractionResult` with `atoms == []`

This distinction (valid-but-empty vs malformed) is explicitly tested.

### 2.3 Isolated extractor fault (case 5)
Each stage runs inside `_run_stage`, which catches **all** exceptions,
increments `extraction_failures_total{stage=<extractor.name>}`, logs the
traceback, and returns an empty atom list for that stage. Because the rule
stage returning empty triggers the heuristic fallback, a crash in Stage 1 is
transparently covered by Stage 2 when possible. The document still produces a
valid result.

Rationale: extractors are independent, best-effort contributors. One faulty
template or regex must not deny the document all extraction.

### 2.4 Confidence safety (cases 6ŌĆō7)
- `ConfidenceScorer.apply` clamps any out-of-band confidence into the method's
  band, so a buggy extractor cannot emit an invalid score downstream.
- `ConfidenceScorer.score` raises `ConfidenceScoringError` for a `raw_signal`
  outside `[0, 1]` ŌĆö this is a programming error in caller code, surfaced
  immediately rather than masked.

### 2.5 Contract validation (case 8)
`FactAtom.__post_init__` rejects empty subject/predicate/object and
out-of-range confidence at construction time. Extractors therefore cannot
produce structurally invalid atoms; any attempt fails loudly in development and
tests.

---

## 3. Idempotency & retries

The pipeline is pure with respect to input (no persistence, no external I/O in
S1.3), so re-running `run(document)` on the same input is **idempotent** and
yields an equivalent `ExtractionResult` (modulo `extractedAt` / `latencyMs`).
Terminal failures (malformed input) are not retryable without fixing the input;
isolated stage faults are already self-mitigated within a single run.

---

## 4. Observability contract

Operators alert on:
- **Rising `extraction_failures_total{stage=validate}`** ŌåÆ upstream Slice 1.2
  contract regression.
- **Rising `extraction_failures_total{stage=rule_based|heuristic}`** ŌåÆ buggy
  template/regex deploy; safe to roll back without data loss.
- **`extraction_latency_ms` p99 growth** ŌåÆ pathological input size or regex
  backtracking; investigate templates.
- **`extraction_atoms_total` collapse to ~0** while documents flow ŌåÆ extractor
  regression or corpus shift.

## 5. Future-slice considerations (not implemented here)

When Stage 3 LLM routing is wired:
- Network/timeout/quota errors will be classified **recoverable** and isolated
  exactly like case 5, falling back to whatever stages 1ŌĆō2 produced.
- Gateway-level guardrail rejections (PII, injection re-screen) will be a new
  terminal class with its own `stage` label.
- Persistence failures (once added) will be retryable with backoff, separate
  from extraction failures.
