# S1.4 — Failure Handling

This document enumerates the failure modes of the Slice 1.4 resolution
pipeline and the defined behaviour for each.

## 1. Merge Failures

**Condition:** An atom cannot be resolved — empty name, or a name that
normalizes to the empty string (e.g. punctuation-only).

**Behaviour:** `Resolver.resolve` raises `ResolutionError(atom_id=...)`.
The error carries the offending `atom_id` for audit. The pipeline does not
silently swallow this — a name that cannot be normalized indicates a defect
in upstream extraction and should surface loudly.

**Recovery:** Caller logs the `atom_id`, drops the atom, and continues the
batch. No partial entity is created, so canonical state stays consistent.

## 2. ACL Violations

**Condition:** A principal attempts to contribute evidence from a source it
may not see — explicit DENY rule, cross-tenant access, or an unregistered
source (fail-closed default).

**Behaviour:** Inside `process_one`, `_validate_acl` raises
`ACLViolationError`, which the pipeline catches, increments
`acl_rejections`, records the `atom_id` in `rejected`, and returns `None`.
The resolver is never invoked for a denied atom, so denied evidence cannot
influence any canonical entity or its provenance.

**Invariant:** ACL enforcement precedes resolution and merge. There is no
code path where a merge occurs before its source ACL is evaluated.

## 3. Orphan Provenance

**Condition:** A `ProvenanceRecord` has no `source_id` (or no `tenant_id`),
or its tenant does not match the resolving principal's tenant.

**Behaviour:**
- Missing `source_id` / `tenant_id` is rejected at construction
  (`ValueError`), so an orphan record can never be instantiated.
- A well-formed record whose `tenant_id` differs from the principal raises
  `ProvenanceError(extraction_id=...)` in `_validate_provenance`.

**Rationale:** Provenance is the audit backbone. An entity with detached or
mismatched lineage is unauditable and is therefore treated as a hard error
rather than a recoverable skip.

## 4. Entity Conflicts

**Condition:** Two surface forms collide on the same normalized key but
represent different real-world entities (e.g. an org and a product both
named "Apex"), or the same surface form arrives with different identity
types.

**Behaviour:** Canonical ids are namespaced by `identity_type`
(`f"{identity_type}:{normalized}"`), so an ORGANIZATION "Apex" and a PRODUCT
"Apex" resolve to **distinct** canonical ids and never merge. Within a
single identity type, deterministic normalized matching is intentional —
same-type same-normalized forms are treated as the same entity by design.

**Confidence:** When forms merge via lower-confidence strategies (alias,
normalized), the entity's confidence is set to the conservative minimum,
flagging the entity for downstream review without blocking the merge.

## 5. Unknown / Mis-wired Source

**Condition:** An atom references a `source_id` with no registered ACL.

**Behaviour:** `ACLRegistry.check` returns `DENY` (fail closed). The atom is
rejected exactly as an explicit DENY would be. This prevents new or
mis-configured connectors from leaking evidence into shared canonical state.

## Error Taxonomy

| Error | Raised When | Pipeline Disposition |
|-------|-------------|----------------------|
| `ResolutionError` | name empty / unnormalizable | propagate (batch continues) |
| `ACLViolationError` | denied / cross-tenant / unknown source | caught, counted, atom rejected |
| `ProvenanceError` | orphan or tenant-mismatched provenance | propagate (hard error) |

All three derive from `PMOSResolutionBaseError` for family-level handling.
