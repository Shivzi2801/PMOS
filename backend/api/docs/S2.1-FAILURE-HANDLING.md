# S2.1 — Failure Handling

This document explains every way a request can fail, what the client sees, and
why the design is safe. All failures are funneled through **one** mechanism —
the handlers registered in `error_handlers.py` — so that no matter where a
failure originates, the client always receives the same standardized envelope.

## 1. The universal error shape

Every failure returns the standard envelope with `status: "error"`:

```json
{
  "request_id": "f3a9c1e0...",
  "timestamp": "2026-06-17T12:00:00+00:00",
  "status": "error",
  "data": null,
  "errors": [
    { "code": "validation_error", "message": "...", "field": "question" }
  ]
}
```

- `code` is **machine-readable and stable** — clients branch on it.
- `message` is human-readable and safe (never leaks internals).
- `field` is present for failures tied to a specific input.
- `request_id` is the correlation id, so a failing call can be traced in logs.

## 2. The seven failure classes (per spec)

| # | Failure | Trigger | HTTP | `code` | Source |
|---|---------|---------|------|--------|--------|
| 1 | **Validation failure** | Payload violates a request model (missing field, blank string, out-of-range value) | `422` | `validation_error` | FastAPI `RequestValidationError` |
| 2 | **Missing resource** | Connector / document / job not found | `404` | `resource_not_found` | `ResourceNotFoundError` |
| 3 | **Pipeline failure** | A backend module raised while processing | `502` | `pipeline_error` | `PipelineError` |
| 4 | **Timeout** | A backend stage exceeded its time budget | `504` | `pipeline_timeout` | `PipelineTimeoutError` |
| 5 | **Rate-limit violation** | Caller exceeded the policy for the endpoint scope | `429` | `rate_limit_exceeded` | `RateLimitExceeded` / middleware |
| 6 | **Authentication failure** | Auth provider rejects the request (future) | `401` | `authentication_failed` | `AuthError` |
| 7 | **Unexpected exception** | Anything else | `500` | `internal_error` | catch-all handler |

## 3. How each failure is handled

### 3.1 Validation failures (422)
FastAPI validates the body against the request model **before** the handler
runs. The validation handler walks every field error, strips the leading
`body` from the location path, and emits one `ErrorDetail` per problem so the
client learns *all* invalid fields at once. No business logic executes.

### 3.2 Missing resources (404)
When the service adapter (or a backend module) cannot find a referenced
connector, document, or job, it raises `ResourceNotFoundError(resource, id)`.
The handler reports the resource type in `field` and the human-readable id in
`message`.

### 3.3 Pipeline failures (502)
Each adapter method wraps the backend call in a guard that converts any raised
exception into `PipelineError(stage, message)`. The handler logs the full
detail **server-side only** and returns a `502` naming the failed `stage`
(e.g. `retrieval`) in `field`. The client learns *which* stage failed without
seeing internal detail.

### 3.4 Timeouts (504)
When a stage exceeds its time budget it raises `PipelineTimeoutError(stage,
timeout_s)`, mapped to `504 Gateway Timeout`. Timeouts are deliberately
distinct from generic pipeline failures so clients (and dashboards) can treat
"slow/unavailable dependency" differently from "dependency errored."

### 3.5 Rate-limit violations (429)
Enforced in `RateLimitMiddleware` *before* the handler. On breach the
middleware:
1. records a `rate_limit_events` metric,
2. returns a `429` envelope with `code: rate_limit_exceeded`,
3. sets a `Retry-After` header computed from the token-bucket deficit.

Operational endpoints (`/health`, `/version`, `/metrics`) are exempt so
monitors are never throttled.

### 3.6 Authentication failures (401)
Today every request resolves to an anonymous principal, so 401 is not emitted.
The path is wired so that when a real provider is installed (`auth_stub.py`),
it raises `AuthError` and the handler returns a `401` — with no change to route
handlers.

### 3.7 Unexpected exceptions (500)
The catch-all handler is the safety net. It logs the full traceback
server-side via `logger.exception` and returns a generic `internal_error`
message. **It never leaks stack traces, exception text, or internal structure
to the caller.** This is both a security and a UX guarantee.

## 4. Failure flow diagram

```
request ──▶ middleware ──▶ validation ──▶ handler ──▶ service ──▶ backend
   │            │              │             │            │           │
   │            │ 429          │ 422         │            │ raises    │
   │            │ (rate limit) │ (bad input) │            ▼           │
   │            ▼              ▼             │     PipelineError/Timeout/NotFound
   │      [error_handlers.py] ◀──────────────┴───────────┘
   │            │
   │            │  builds standard error envelope (status, code, field, request_id)
   ▼            ▼
client ◀── consistent JSON, correct HTTP status, X-Request-ID echoed
```

## 5. Why this design is safe

- **Single funnel.** Every failure, from any layer, exits through
  `error_handlers.py`. There is exactly one definition of the error contract.
- **No leakage.** Internal messages and tracebacks are logged, never returned.
- **Traceability.** Every error carries the `request_id`, so support and
  engineering can pinpoint the exact request in logs.
- **Stable codes.** Clients integrate against `code`, not prose, so error
  handling on the client side never breaks when wording changes.
- **Distinguishable severity.** Bad input (422), throttling (429), missing data
  (404), dependency error (502), dependency slow (504), and internal bug (500)
  are all separable — essential for dashboards, alerting, and SLAs.

## 6. What a PM can tell stakeholders

"Every PMOS API call either succeeds with data or fails with a clearly-coded
error in the exact same JSON shape. We distinguish bad requests from throttling
from downstream problems from internal bugs, we never leak internals to
callers, and every error is traceable to a single request id." That sentence is
the reliability story enterprise buyers and integration partners ask for.
