# S2.1 — API Layer Architecture

## 1. Purpose of this slice

Slice S2.1 builds the **API layer**: the single, production-grade HTTP entry
point into PMOS. Everything PMOS can already do lives in nine backend modules
(`connectors`, `ingestion`, `extraction`, `resolution`, `indexing`,
`retrieval`, `context`, `generation`, `grounding`). Those modules are powerful
but they have no front door. The API layer is that front door.

It does five jobs and nothing else:

1. **Receive** HTTP requests.
2. **Validate** their payloads.
3. **Route** them to the right backend service.
4. **Return** a standardized response.
5. **Observe** what happened (health, metrics, logs, correlation ids).

By keeping the API thin and pushing all behavior into the backend modules, we
get a clean separation: the backend owns *what PMOS knows and does*, the API
owns *how the outside world talks to it*. This is what makes the same backend
reusable by a future UI, by enterprise customers, and by partner integrations
without change.

## 2. The big picture

```
                         ┌─────────────────────────────────────────┐
   UI / partner /        │              PMOS API (S2.1)             │
   external client  ───▶ │                                          │
   (HTTP + JSON)         │  middleware → router → service adapter   │
                         │                                          │
                         └───────────────────┬──────────────────────┘
                                             │  in-process calls
                                             ▼
        connectors · ingestion · extraction · resolution · indexing ·
            retrieval · context · generation · grounding   (S1.x)
```

The API never reimplements any backend logic. It *adapts* and *delegates*.

| `response_models.py` | **Outbound contract.** The universal envelope plus per-endpoint data payloads. | handler data | standardized JSON |
| `error_handlers.py` | **Centralized failure translation.** Maps each exception type to the right HTTP status + error envelope. | exceptions | error envelopes |
| `auth_stub.py` | **Auth abstraction.** Future-ready strategy interfaces for JWT, OAuth, SSO, API keys. Currently allows anonymous. | request headers | a `Principal` |

| `services.py` | **Service adapter.** The one place that knows how to call the nine backend modules; isolates the API from their internals. | handler args | normalized dicts |

## 4. The API request lifecycle

Every request flows through the same pipeline, assembled in `app.py`:

```
client
  │  HTTP POST /api/v1/answer  { "question": "..." }
  ▼
CorrelationIdMiddleware   → assigns request_id, sets X-Request-ID
  ▼
RateLimitMiddleware       → token-bucket check for this client + endpoint scope
  ▼                          (429 + Retry-After if exceeded)
TimingMetricsMiddleware   → starts the latency timer, logs, records metrics
  ▼
FastAPI validation        → payload checked against AnswerRequest (422 if bad)
  ▼
router.answer handler     → resolves Principal (auth_stub), calls services
  ▼
PMOSServices.answer       → retrieval → context → generation → grounding
  ▼
ResponseEnvelope.ok       → { request_id, timestamp, status, data, errors }
  ▼
(unwinds back through middleware: metrics recorded, timing header, request id echoed)
  ▼
client receives standardized JSON
```

If anything raises along the way, the registered handlers in
`error_handlers.py` intercept it and emit the *same* envelope shape with an
error status — the client never sees a raw stack trace or an inconsistent body.

## 5. Validation flow

1. The route declares a typed request model (e.g. `AnswerRequest`).
2. FastAPI parses the JSON body and validates it against that model.
3. Field constraints (lengths, ranges, required fields, non-blank strings)
   encode the business rules. A `top_k` of 9999 or a blank `question` fails
   here.
4. On failure, `RequestValidationError` is raised *before* any handler code
   runs, and `error_handlers.py` converts it into a `422` envelope whose
   `errors[]` name the offending fields.

This means **no handler ever receives an invalid payload**, and validation
behavior is identical across all endpoints.

## 6. How this slice interacts with all previous slices

The API does not duplicate any prior slice — it exposes them. The mapping:

| Endpoint | Backend module(s) invoked (via `services.py`) |
|----------|-----------------------------------------------|
| `POST /connectors` | `connectors` |
| `POST /documents/ingest` | `ingestion` |
| `POST /documents/process` | `extraction` → `resolution` → `indexing` |
| `POST /search` | `retrieval` |
| `POST /answer` | `retrieval` → `context` → `generation` → `grounding` |
| `POST /grounding/verify` | `grounding` |
| `GET /health` | reports connectivity of **all nine** modules |

`services.py` is the seam. It imports the real modules when present and calls
their public functions. If a module is unavailable in a given environment it
uses a clearly-labelled deterministic fallback (every fallback marks itself in
`metadata`/`mode` so it can never be mistaken for real output). This keeps the
API independently testable while remaining fully wired for production.

## 7. Authentication strategy

Real authentication is intentionally **not** implemented in S2.1, but the slice
is built so that enabling it later is a strategy swap, not a rewrite. See
`auth_stub.py`:

- `AuthProvider` is an abstract strategy interface (`authenticate(headers) →
  Principal`).
- `JWTAuthProvider`, `OAuthAuthProvider`, `SSOAuthProvider`, and
  `APIKeyAuthProvider` declare the exact shape each future mechanism will take
  and currently resolve to an anonymous principal.
- `get_auth_provider` is a FastAPI dependency; switching the active provider
  (or making it deny-by-default) is one line.

Because every handler already depends on a `Principal`, turning on real auth
requires zero changes to the route handlers.

## 8. Future scalability strategy

- **Versioning.** Major versions are mounted side by side under `/api/v{n}`.
  Shipping `/api/v2` is additive; `/api/v1` clients never break. See
  `versioning.py`.
- **Auth.** Pluggable providers (above) for JWT/OAuth/SSO/API keys.
- **Rate limiting.** The `RateLimiter` interface lets us swap the in-memory
  token bucket for a Redis-backed limiter to share state across many API
  instances, with no change to middleware or routes.
- **Observability.** `api_metrics` is backend-agnostic; the same record calls
  can later feed Prometheus/OpenTelemetry exporters.
- **Statelessness.** The API holds no per-request state beyond the in-process
  metrics counter, so it scales horizontally behind a load balancer.
- **Adapter isolation.** Because all backend coupling lives in `services.py`,
  the backend modules can evolve independently of the public contract.

## 9. Why a Product Manager should care

This slice is what turns PMOS from "a set of capabilities" into "a product
others can build on." It is the contract you can show an enterprise customer
("here is a stable, versioned, authenticated, rate-limited, observable API"),
the foundation the UI team builds against, and the integration point partners
plug into. The standardized envelope and centralized error handling mean every
consumer — internal or external — gets the same predictable experience, which
is exactly what reduces integration cost and support load as PMOS grows.
