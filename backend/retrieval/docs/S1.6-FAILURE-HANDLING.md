# S1.6 — Retrieval Layer · Failure Handling

This document is the operational contract for how the Retrieval Layer behaves
under every failure mode. The governing principle, consistent with prior
slices:

> **Security and correctness failures are fail-closed (raise). Quality-enhancing
> stages are fail-open (degrade and record).**

---

## 1. Error taxonomy

All errors derive from `RetrievalError` and carry a stable `.code` and a
structured `.details` mapping (never secrets). Downstream code branches on
`.code`, not message text.

| Exception | `.code` | Class | Raised when |
|-----------|---------|-------|-------------|
| `InvalidQueryError` | `invalid_query` | input | Empty text, bad filter operand |
| `PaginationError` | `invalid_pagination` | input | Negative offset, oversized/zero limit, offset > `MAX_OFFSET` |
| `TenantIsolationError` | `tenant_isolation_violation` | **fatal** | Missing tenant, or a foreign-tenant record in the candidate set |
| `AclDeniedError` | `acl_denied` | fatal | Explicit by-id access check fails (not ordinary filtering) |
| `IndexUnavailableError` | `index_unavailable` | **fatal** | S1.5 `search` raised |
| `RerankError` | `rerank_failed` | recoverable | Reranker misconfig; caught by `safe_rerank` |
| `QueryExpansionError` | `query_expansion_failed` | recoverable | Expander failure; caught by `safe_expand` |

---

## 2. Fail-closed paths (request raises)

### 2.1 Tenant isolation
- **Trigger:** query has no `tenant_id`, or any candidate's `tenant_id` differs
  from the query's.
- **Behaviour:** `enforce_tenant_isolation` raises `TenantIsolationError`
  *before* ACL, filtering, ranking, or pagination. A failure metric and event
  (`success=false`) are emitted.
- **Why fatal:** a cross-tenant record means an upstream scoping defect. Silent
  dropping would mask a data-leak bug. This is defence-in-depth on top of S1.5's
  own tenant scoping.
- **Operator action:** alert on any `tenant_isolation_violation`. It is never
  expected in normal operation.

### 2.2 Index unavailable
- **Trigger:** the S1.5 `VectorIndex.search` raises for any reason.
- **Behaviour:** wrapped as `IndexUnavailableError` with the original cause
  chained (`raise ... from exc`); failure metric + event emitted.
- **Caller guidance:** treat as a transient dependency failure — retry with
  backoff; surface a 503-class response upstream. Do **not** fall back to stale
  or unfiltered data.

### 2.3 Invalid input
- **Trigger:** malformed query/filters/pagination.
- **Behaviour:** raised at construction time (`__post_init__`), so an invalid
  query can never reach the index. No metrics are emitted (the request never
  started). Caller should map to a 400-class response.

---

## 3. Fail-open paths (request succeeds, `degraded=true`)

These stages improve quality but are not required for a correct, safe result.
Each is wrapped so a fault cannot break retrieval.

### 3.1 Query expansion — `safe_expand`
- On any exception, the **original** query is used unchanged.
- `ExpansionResult.degraded = True`, note `query_expansion_failed[:Type]`.
- Surfaced as `diagnostics.degraded=true` and the `retrieval.degraded` metric.

### 3.2 Reranking — `safe_rerank`
- On `RerankError` or any exception, the **vector-similarity order is
  preserved** and `degraded=True` is returned.
- `diagnostics.reranked=false`, `diagnostics.degraded=true`,
  `retrieval.degraded` incremented.
- Rationale: an unranked-but-correct result is strictly better than no result.

> A future LLM reranker inherits this safety automatically — a model timeout or
> bad response degrades to vector order rather than failing the query.

---

## 4. Ordinary (non-error) empty outcomes

Some outcomes return a valid, empty-or-small result and are **not** errors:

| Situation | Outcome |
|-----------|---------|
| Vector store returns no candidates | empty `RetrievalResult`, `retrieval.empty_result` incremented |
| All candidates fail the ACL check | hits dropped silently, `retrieval.acl_dropped` counted |
| All candidates fail metadata filters | hits dropped silently, `retrieval.filter_dropped` counted |
| `min_score` excludes everything | empty result |
| Page offset beyond filtered population | empty page, `has_more=false` |

ACL denials are deliberately **silent drops**, not errors: "you may not see
this document" is an ordinary access outcome and must not leak the document's
existence via an exception. The only ACL *error* (`AclDeniedError`) is reserved
for an explicit "fetch this specific id" check.

---

## 5. Metrics emitted per outcome

| Outcome | Metrics |
|---------|---------|
| Success | `queries`, `candidates`, `returned`, `latency_ms`, `success`, (+`empty_result`/`degraded`/`reranked`/`expanded` as applicable), `record_event(success=true)` |
| Fatal failure | `queries`, `failure`, `latency_ms`, `record_event(success=false, error_code=...)` |
| ACL / filter drops | `acl_dropped`, `filter_dropped` (counts, even on success) |

Every call produces exactly one `RetrievalEvent`, making success rate, latency
percentiles, degradation rate, and drop ratios computable from the event stream
alone.

---

## 6. Invariants (must always hold)

1. No `RetrievalResult` ever contains a hit whose `tenant_id` ≠ the query's.
2. No hit is returned that the requesting principals cannot read per its ACL.
3. A document with no/empty ACL is private to the tenant (never world-readable
   by default).
4. Expansion and reranking failures never change the success/failure status of
   a request — only its `degraded` flag.
5. `page.total_candidates` reflects the post-filter population; `has_more` is
   exact.
6. Every terminal path (success or raise) emits exactly one metrics event.

These invariants are covered by `test_retrieval.py` (34 tests, all passing).

---

## 7. Operator runbook (quick reference)

| Symptom | Likely cause | Action |
|---------|--------------|--------|
| Spike in `tenant_isolation_violation` | upstream scoping bug in S1.5 | page on-call; treat as potential data-leak incident |
| Spike in `index_unavailable` | S1.5 store down / network | check S1.5 health; retries with backoff |
| Rising `retrieval.degraded` | reranker/expander faults | inspect reranker/expander logs; results still safe |
| High `acl_dropped` ratio | principals mismatch / ACL ingestion issue | verify principal resolution + S1.4/S1.5 ACL metadata |
| Empty results climbing | embedding/index drift or over-restrictive filters | inspect query patterns and filter usage |
