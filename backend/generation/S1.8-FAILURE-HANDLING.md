# S1.8 — Failure Handling

This document enumerates every failure mode the Generation layer recognises,
how it is detected, and the resulting behaviour. The guiding principle is
**fail closed**: when grounding cannot be guaranteed, S1.8 returns a safe,
explicit status rather than an unsupported answer.

## 1. Error taxonomy

All errors derive from `GenerationError`.

| Exception | Retryable | Meaning |
|-----------|-----------|---------|
| `TransientProviderError` | yes | Temporary provider failure (timeout, 5xx, connection) |
| `ProviderTimeoutError` | yes | Call exceeded deadline |
| `RateLimitedError` | yes | Provider returned 429 / overloaded |
| `PermanentProviderError` | no | Auth, bad request, 4xx — not recoverable by retry |
| `UnknownProviderError` | no | Requested provider not registered (config error) |
| `ConfigurationError` | no | Missing/invalid configuration |
| `ContractViolationError` | no | PromptPackage/response invariant violated |
| `GroundingError` / `UnsupportedAnswerError` | no | Answer failed grounding |

Each provider adapter is responsible for mapping its native SDK exceptions
onto this taxonomy (`_normalise_*_error`). Unknown provider exceptions are
treated as **transient** so a bounded retry can recover from undocumented
hiccups, while clearly-permanent classes (auth, bad request) are not retried.

## 2. Terminal answer statuses

The pipeline always returns a `GeneratedAnswer`; failures are encoded as
`AnswerStatus`, never propagated as exceptions to the caller.

| Status | Trigger | Answer text | Confidence |
|--------|---------|-------------|-----------|
| `OK` | Grounded, confidence ≥ threshold | model output | computed |
| `REFUSED` | Model emitted the configured refusal phrase | model output | 0.0 |
| `EMPTY_CONTEXT` | No context chunks supplied (pre-flight) | refusal text | 0.0 |
| `LOW_CONFIDENCE` | Mean context score below floor, **or** confidence < threshold | refusal text | 0.0 / computed |
| `UNSUPPORTED` | Cites unknown markers, or asserts content with no citation | refusal text | computed |
| `PROVIDER_ERROR` | Provider failed after retries (or permanent) | empty | 0.0 |

## 3. Retry policy

`RetryPolicy(max_attempts=3, base_delay_s=0.2, max_delay_s=5.0, jitter=True)`.

- Only `ProviderError` with `retryable=True` is retried.
- Delay is bounded exponential back-off with full jitter:
  `delay = uniform(0, min(base * 2^(n-1), max))`.
- `max_attempts` caps total provider calls. On exhaustion the last exception is
  re-raised inside the pipeline and converted to `PROVIDER_ERROR`.
- The `sleep` function is injectable; tests pass a no-op to avoid real delays.
- Each retry increments `generation.retries{attempt=n}`.

**Permanent errors are never retried** — they fail fast to `PROVIDER_ERROR`.

## 4. Pre-flight gates (fail before spending an LLM call)

`safety.preflight` runs before the provider is invoked:

1. **Empty context.** No chunks ⇒ `EMPTY_CONTEXT`. The provider is *not* called.
2. **Score floor.** If `params.min_context_score > 0` and the mean chunk score
   is below it ⇒ `LOW_CONFIDENCE`. The provider is *not* called.

These gates protect against burning tokens on requests that cannot be grounded.

## 5. Grounding failures

Detected post-generation by `grounding.enforce`:

- **Hallucinated citation.** The answer cites a marker not present in the
  supplied chunk set ⇒ `UNSUPPORTED`. The offending markers are recorded in
  `diagnostics["unknown_markers"]`.
- **Unsupported assertion.** Substantive answer text with zero valid citations
  ⇒ `UNSUPPORTED`.
- **Refusal.** The model emitted the configured refusal phrase ⇒ `REFUSED`.

In all three cases S1.8 returns the safe refusal text (for the gated statuses)
and never surfaces an answer it cannot attribute to context.

## 6. Confidence threshold

After a positive grounding verdict, the computed confidence is compared with
`params.confidence_threshold`. Below threshold ⇒ `LOW_CONFIDENCE`, downgrading
an otherwise-grounded answer. This lets callers tune strictness per request.

## 7. Partial / degraded responses

- **Empty completion** (provider returns no text): treated as non-assertive ⇒
  `UNSUPPORTED` via the grounding layer.
- **Missing usage data** from a provider: token metrics default to 0; the
  request still succeeds. Metrics consumers must tolerate zero counts.
- **Chunk without a CitationRecord:** the chunk id is still tracked as
  supporting context, but no `BoundCitation` is produced. The answer remains
  `OK` if otherwise grounded.

## 8. Idempotency & correlation

`PromptPackage.request_id` is propagated unchanged into every `LLMRequest`,
metric tag context, and the returned `GeneratedAnswer`, enabling end-to-end
tracing across slices. Generation has no persistent side-effects, so retries
at any layer are safe to repeat.

## 9. Observability of failures

| Metric | When |
|--------|------|
| `generation.failures{kind=...}` | Any non-OK terminal status (one per request) |
| `generation.retries{attempt=n}` | Each retry of a transient provider error |
| `generation.latency_ms{status=...}` | Every request, success or failure |

The double-counting guard ensures exactly one `failures` increment per failed
request: the terminal handler is the sole emitter.

## 10. Failure-handling test coverage

- `test_empty_context_short_circuits` — pre-flight, provider not called
- `test_low_context_score_gate` — score floor, provider not called
- `test_unsupported_answer_when_marker_unknown` — hallucinated citation
- `test_refusal_status` — refusal phrase detection
- `test_low_confidence_threshold` — threshold downgrade
- `test_retry_then_success` — transient recovery + retry metrics
- `test_retry_exhausted_returns_provider_error` — bounded retries
- `test_permanent_error_not_retried` — fast-fail on permanent error
